1. 開場：一句話版本

0:00 — 2 min

大家好，今天要報告我碩論做了什麼。

先講結論：我沒有發明新的零知識證明系統。我做的是在現有的 Halo2 框架裡面，設計一套電路層級的方法論，讓它能精確計算 exp、ln、sin 這些超越函數。

用一個大家熟悉的類比：Halo2/PLONK 是 CPU 的指令集，我沒有改指令集。我是在這個指令集上面寫了一個高效的「組合語言程式」。

— 停頓，確認大家理解這個定位 —

為什麼要特別強調這點？因為等一下你們會看到我們有做安全性論證，有證 completeness、soundness、ZK。但那是電路層的論證，不是協議層——PLONK 和 KZG 的安全性是他們的論文證的，不是我們。這個區分很重要，等一下安全性那邊會詳細講。

2. 問題：ZK 電路算不了超越函數

2:00 — 3 min

核心問題很簡單：ZK 算術電路只能做 Fp 上的加法跟乘法。exp(x)、sin(x) 這些超越函數不是多項式，所以不能用有限次加法和乘法精確算出來。

但是很多應用需要這些函數。ZKML 裡面神經網路的激活函數——sigmoid、GELU——都是超越函數。可驗證金融計算裡面，Black-Scholes 用到 erf、exp、ln。你只要想在 ZK 裡面跑這些東西，就得解決這個問題。

現有兩條路：

第一條是查表法，ezkl 他們的做法。預算一張大表，每個輸入查對應的輸出。精度完美，ULP 等於零。但問題是表很大——一個函數在 pp=16 的時候需要 65,536 筆，pp=28 就需要 2.68 億筆，根本不可行。而且每個函數都需要自己的表。

第二條是多項式近似。用低次多項式去逼近函數，在電路裡面用 Horner 法求值。zkTaylor 和 ZKFixedPointChip 走的是這條路，我們也是。

但文獻裡面有四個缺口：他們用 Taylor 不是 L∞ 最優的；沒有人做過 ULP 誤差分析；沒有人探索過 (pp, d) 的設計空間；也沒有人用過延遲截斷。

我們就是把這四個缺口補起來的。

3. 我們用了什麼系統、改了什麼

5:00 — 2 min

我們用的是 Halo2，PSE fork v0.3.0，底下是 PLONK + KZG，曲線是 BN254。

重點：我們一個東西都沒改。PLONK 的算術化、KZG 的承諾方案、Fiat-Shamir 變換、lookup argument——全部原封不動。

我們做的事情全部在 Halo2 提供的 custom gate 機制裡面。我們設計了 6 個 gate：horner_step、truncation、r_byte_decomp、w_byte_decomp、rem_byte_decomp、r_range_check。加上一張共用的 256 筆 ByteTable。就這些。

如果有人問「為什麼不用 STARK / Groth16？」—— 回答：Halo2 的 custom gate 讓我們能精準控制每個約束的結構，而且 KZG 的 proof 很小。但方法論本身不依賴 Halo2，只要底層系統支援 custom gates 和 lookup arguments 就能移植。

4. 三個貢獻

7:00 — 5 min

貢獻一：Minimax 取代 Taylor。

這是數值分析層面的改進。之前 zkTaylor 用 Taylor 級數——Taylor 把精度集中在展開點，端點的誤差爆炸。我們改用 Minimax 多項式，用 Remez 算法計算，它把誤差預算均勻分配到整個區間，是 L∞ 最優的。

結果：在 13 個函數上，Minimax 比 Taylor 改善 2 倍到 4,330 倍。52 個配置中有 29 個 Minimax 做得到但 Taylor 完全失敗。

— 停頓 —

貢獻二：延遲截斷。

這個是電路設計技巧。標準做法是 Horner 求值每一步乘法之後都截斷，也就是除以 S，把尺度恢復回來。這樣每步引入一次捨入誤差，d 步下來累積 O(d) 個 ULP。

我們的觀察是：BN254 的域有 254 位，但定點數運算只用到幾十位。中間值不會溢位。所以我們讓 Horner 鏈的中間值在域中自然增長，到最後才做一次截斷。

(d + 1) × pp < 254 — Headroom Constraint

pp=16、d=5 的時候只用 96 位，還有 158 位餘量。結果：13/13 函數都必須用延遲截斷才能達到 ULP=1。逐步截斷全部失敗。

— 停頓 —

貢獻三：(pp, d) 設計空間分析。

這是實驗科學。我們把 13 個函數乘以 7 個精度乘以 7 個次數，總共 546 個有效配置全部跑過，窮舉驗證 ULP。最有意思的發現是量化飽和：增加多項式次數 d 到一個臨界值之後，精度反而變差。因為高次係數在定點量化後的相對誤差太大，乘以 xd 之後被放大。這個現象在浮點數世界不存在。

5. 安全性：兩層論證

12:00 — 5 min

好，這裡是今天最重要的部分，也是你們作為密碼學的同學最在意的部分。

放慢語速，這段要講清楚

一個 zkSNARK 的安全性需要在兩個層次分別論證。

第一層是協議層。PLONK 說：「如果 witness 滿足所有約束，prover 可以生出有效的 proof」——這是 completeness。KZG 說：「給我一個有效的 proof，我可以提取出一個滿足約束的 witness」——這是 knowledge soundness。PLONK 的 simulator 構造保證 proof 不洩露 witness——這是 zero-knowledge。

這三個性質全部由 PLONK/KZG 的論文證明。我們原封不動繼承，沒有重新證。

— 停頓，確認大家跟上 —

但是，協議層只管「約束有沒有被滿足」。它不知道你的約束到底代表什麼意思。如果你的電路約束設計有 bug——比如忘了檢查輸入範圍——PLONK 還是會忠實地執行，但系統就不安全了。

所以需要第二層：電路層的論證。這是我們的工作。

我們論證了三件事：

第一，Perfect Completeness：對每一個合法輸入 R，誠實的 prover 都能構造一個滿足所有 6 個 gate 約束的 witness。我們逐約束檢查——ExplicitRC 的 gap 分解、Horner gates 的中間值不溢位（用 headroom constraint 保證）、截斷 gate 的整數除法恆等式、byte 分解的位寬範圍。

第二，Knowledge Soundness：假設 PLONK 的 extractor 給了我們一個滿足所有約束的 witness，我們論證這個 witness 必然滿足三個性質——R 在 [0, Rmax]、w 在 [0, S)、w 是正確的 Horner 求值結果。

第三，Computational Zero-Knowledge：我們檢查我們的電路設計沒有意外洩漏秘密。所有 prover 秘密值——R、中間值、輸出 w——全部放在 advice columns。Halo2 會對 advice column 加 blinding factor 再做 KZG commitment，所以是 computationally hiding。我們的電路不用 instance columns，所以沒有 witness 值被暴露為 public input。而且我們只用 custom gates 和 lookup arguments，都是 Halo2 的標準原語，沒有非標準操作。

— 停頓 —

直覺是這樣的：協議層保證「約束被滿足 ⇒ proof 有效」。電路層保證「我們的約束 ⇔ 我們宣稱的性質」。兩層合在一起才是完整的安全論證。

這裡可以預期會有人提問。如果有人問「那你們的安全性到底是 conditional 的？」—— 對，是 conditional on PLONK/KZG being secure。但所有在 Halo2 上面跑的電路都一樣。ezkl 也是、circom 也是。區別是我們有顯式寫出電路層的論證，很多系統根本沒做這一步。

5b. ExplicitRC：我們自己設計的約束

補充，視時間決定

如果有時間，多講一個我們自己設計的東西：ExplicitRC。

問題是這樣的：多項式近似只在特定區間 [0, Rmax] 上有效。如果惡意 prover 偷偷餵一個超出範圍的 R，多項式外推的結果是錯的，但電路不會報錯。所以我們需要在電路裡強制 R ≤ Rmax。

做法是 gap 分解：

gap_lo + gap_hi × 256 + R = Rmax

其中 gap_lo 和 gap_hi 都必須是 [0, 255] 的 byte，由 ByteTable lookup 強制。如果 R > Rmax，那 Rmax − R 在 Fp 中會是 p 減掉一個小正整數，是一個超過 2253 的巨大數字。兩個 byte 的線性組合最大只到 65,535，所以 lookup 必然失敗。

這個保證是確定性的——不依賴機率論證或計算假設，只依賴有限域的代數性質。

我們也分析了另一個方案 ORV——利用多項式單調性隱式驗證——結果發現 13 個函數中有 7 個不安全。所以最後採用了 ExplicitRC。

7. Q&A 預備（預期問題 + 建議回答）

備用
Q1：你說安全性是 conditional on PLONK，那如果 PLONK 被攻破怎麼辦？
跟所有用 PLONK 的系統一樣會壞掉。ezkl、circom 電路也是。但 PLONK 從 2019 年到現在已經被廣泛審計，而且我們用的 KZG 是基於 q-SDH 假設，是標準假設。我們的電路層論證是在 PLONK secure 的前提下做的，這和所有 ZK 應用論文的做法一致。
Q2：你們的 ZK 論證只是「檢查 advice columns」，這算證明嗎？
Zero-knowledge 在電路層的責任是確保我們的設計不引入額外洩漏。Halo2 已經保證 advice columns 是 blinded 的，我們需要驗證的是：(1) 所有秘密值確實在 advice columns 裡面、(2) 沒有用 instance columns 暴露值、(3) 沒有用非標準操作繞過 blinding。這是一個 checklist 式的驗證，但它和 completeness / soundness 的建構式論證不同，因為 ZK 的保證是從 PLONK 的 simulator 來的，我們只需要確認我們沒有破壞它的前提。
Q3：這跟 ezkl 比起來有什麼優勢？
ezkl 用查表法，ULP=0，精度比我們好。但我們的優勢在高精度場景（pp≥28 查表不可行）和多函數場景（13 個函數共用 256 筆 lookup vs 每個函數 65K 筆）。另外 ezkl 沒有做電路層的安全性論證——他們沒有寫出「電路的 completeness/soundness/ZK 為什麼成立」。
Q4：Minimax 和 Taylor 的差距為什麼那麼大？4,330 倍？
Taylor 的誤差集中在區間端點，而 ULP 看的是 worst-case。在端點，Taylor 的 truncation error 可能比 Minimax 的 L∞ 誤差大幾千倍。加上定點量化放大效應，差距就更大。4,330× 是 ln(1+x) 在 d=5, pp=16 的情況。
Q5：延遲截斷可以用在其他電路嗎？不只超越函數？
可以。只要你在 BN254 上做定點數乘法鏈、滿足 headroom constraint (d+1)·pp < 254，就可以用延遲截斷。這個技巧不限於 Horner 求值。任何需要連續乘法的定點數計算都能用。
Q6：為什麼不用 instance columns 做 public input？
我們的場景是「計算 f(x) 的近似值」，輸入 x 和輸出 w 都是 prover 的秘密——prover 想證明「我知道一個 x，使得 f(x) ≈ w」但不想讓 verifier 知道 x 和 w 是什麼。如果用 instance columns，x 和 w 就變成 public input，失去 ZK 的意義。在實際應用中（例如 ZKML），這些值通常是需要保密的。
Q7：你們的數值分析貢獻到底在哪？Minimax 和 Remez 是 1934 年的東西。
誠實講，數值分析部分我們沒有理論創新。我們的貢獻是把 Minimax 放進 ZK 的定點數環境、做了窮舉 ULP 驗證、量化了和 Taylor 的差距、發現了量化飽和現象。這些是「應用」和「實驗科學」，不是「理論」。和所有密碼學工程的論文一樣——SHA-256 電路也不是新密碼學。
