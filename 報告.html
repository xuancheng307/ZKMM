Motivation03 / 86

金融合規與持倉隱私的張力

對沖基金必須向監管證明風險合規，但持倉本身是商業機密 — 公開 = 策略外洩、front-running。

傳統做法

基金交出完整持倉 → 監管自己算 → 策略曝光，競爭對手可複製

ZK 合規

基金提交 proof → 監管毫秒驗證 → 持倉永遠保密，proof 存在 = 合規

ZK 是唯一能同時做到「即時可驗證 + 永久隱私」的技術；MPC 需多方協作延遲高，FHE 在乘法深度下對超越函數過於昂貴。

Running Example04 / 86

一筆具體的 BS Delta 交易

整個報告都會拉回這筆交易 — 代入看數字怎麼跑。

S=100  K=100  σ=20%  T=30 天  r=5%  q=0  → Δ ≈ 0.523

基金想向監管與 LP 證明「投組加權 Delta 絕對值 ≤ τ」，而不想揭露任何持倉明細（標的、數量、方向）。

d1 = [ln(S/K) + (r - q + σ²/2) · T] / (σ · √T)
   = [ln(1) + (0.05 + 0.02) · 30/365] / (0.20 · √(30/365))
   ≈ 0.0577

Δ_call = e^(-qT) · N(d1) = 1 · (0.5 + 0.5·erf(0.0577/√2))
       ≈ 0.523
Status Quo05 / 86

現有方案為什麼不夠

  • 全揭露：交完整持倉給監管 — 商業機密外洩，front-running 成本放大
  • 信任第三方：外部審計 — 單點信任、延遲數週、單筆成本數萬美元
  • 事後抽查 + 罰款：監管週期以季為單位 — 造假空間大
  • MPC / FHE：需多方協作或乘法深度過高，超越函數實務不可行

ZK 的獨特賣點：單次本地計算 + 毫秒驗證 + 永久隱私。

ZK Value06 / 86

ZK 能提供什麼

「我能證明某件事是真的，但不透露為什麼是真的。」

  • Completeness — 誠實 prover 總能通過
  • Knowledge Soundness — 惡意 prover 不能偽造通過（且真的「知道」witness）
  • Zero-knowledge — 驗證者除了「命題為真」外什麼都學不到

關鍵：ZK 電路裡只有有限域上的加法和乘法 — 沒有 if、沒有迴圈、沒有浮點。這是本論文要解決的核心工程問題。

※ 這裡先講高階性質；Part III 會進 Halo2 內部看 soundness 怎麼被電路約束機制落地。

PF1 Statement15 / 86

PF1 形式化陳述 — 5 條件

∃ witness w s.t. 對所有 i ∈ [1, N]：

  • (a) MerkleVerify(p_i, path_i, state_root) = true
  • (b) MerkleVerify(mkt_i, mkt_path_i, market_data_root) = true
  • (c) |BS(S,K_i,T_i,r,q,σ_i) − settle_price_i| ≤ ε_σ（σ 正向驗證）
  • (d) T_i × 365 = expiry_date_i − today（T 電路內計算）
  • (e) |Σ w_i · Δ_BS(S,K_i,T_i,r,q,σ_i)| ≤ τ（合規）

Part V 會把這 5 條件對應到 31 個 custom gate / 66 條多項式約束。

Two Trees17 / 86

兩棵 Merkle Tree

Position State Tree

記錄當前持倉快照

  • 5 欄位：asset_id, w, K, expiry_date, type
  • Depth 7（生產 N_max=128；測試 N=8）
  • 不足 N_max 填 dummy 葉子（隱藏真實持倉數量）

Market Data Tree

記錄當日市場資料

  • 8 欄位：underlying_id, strike, expiry, asset_type, S (spot), r, q, settle_price
  • Depth ~20（涵蓋所有標的 × 合約）
  • 每日由資料提供者重算並發布 root

為什麼用 Merkle 而非 Halo2 lookup？ Market data 每日變 — 若用 fixed column lookup 每日得重做 setup + VK。Merkle 讓電路結構一次 setup 永久有效，只有 root 值每日更新。

Recap: Groth16 / R1CS22 / 86

先回到大家熟悉的 — Groth16 / R1CS / QAP

  • R1CS：constraint 形式固定為 (A·z) ⊙ (B·z) = C·z，每條 constraint 一個 row
  • QAP：把 R1CS 轉成多項式 identity，用 pairing 一次驗證
  • Trusted setup：每個 circuit 各做一次 ceremony（toxic waste），circuit 改了重做
  • Proof size：~3 G1 元素（~200 B，非常小）
  • 沒有 lookup：range check 得一位一位拆

這些是 Halo2 不一樣的地方的對照基準。

Plonkish23 / 86

Halo2 = Plonkish arithmetization

整個電路是一張「表格」：列 = 一行計算、欄 = 不同類型的變數；gate 是欄之間的多項式關係。

R1CS (Groth16)

固定 a·b=c 形式

一條 constraint 一個 row

一個 row 只有一條 constraint

→

Plonkish (Halo2)

自訂高次多項式 gate

一個 gate 可跨多列多欄

一個 row 可同時啟用多條 constraint

Columns24 / 86

四類欄：advice / fixed / instance / selector

R1CS

  • witness vector z
  • 矩陣 A/B/C entry（係數）
  • public 部份：z 前綴
  • (沒明確 gate 開關)
↔

Halo2

  • advice：prover 填，私密（被 blinding 保護）
  • fixed：電路定義時寫死，例如多項式係數
  • instance：public input，verifier 也看得到
  • selector：打開 / 關閉 gate（fixed 的特例）
Custom Gates25 / 86

Custom gate — 自訂高次多項式

R1CS 的 gate 固定是 bilinear；Halo2 可以寫任意 degree 的 constraint。

meta.create_gate("horner", |meta| {
  let q    = meta.query_selector(q_horner);
  let acc  = meta.query_advice(acc, Rotation::cur());
  let next = meta.query_advice(acc, Rotation::next());
  let c    = meta.query_fixed(coeff, Rotation::cur());
  let x    = meta.query_advice(x, Rotation::cur());
  vec![q * (next - (acc * x + c))]   // degree-3 約束
});

白話：當 q=1，下一列的 acc = 這列 acc × x + coeff — Horner 一步到位。R1CS 要拆成兩條 constraint + 一個中間變數。

Why high degree26 / 86

為什麼 Halo2 能 degree 高？

PLONK 最早只允許 degree-2。Halo2 靠更高次商多項式 + rotation + lookup，gate 實務 degree 可達 8–10。

  • 好處：一條 constraint 幹以前 5–10 條 R1CS 的事
  • 代價：prover FFT 範圍放大到 ≥ 2·degree·n，但仍 quasi-linear
  • Poseidon x⁵ S-box 就是高 degree 的直接受益者
Copy Constraints27 / 86

Copy constraint — permutation argument

R1CS

同一變數在多條 constraint 用 — 天然「同一 z[i]」，不需要額外機制

↔

Halo2

不同 cell (row, col) 之間用 permutation argument 綁定 = 同值；零額外 gate 成本

Part V 的 Sigma-to-MDT 8 條約束、pipeline 中間值的銜接，本質都是 copy constraint。

Lookup28 / 86

Lookup argument — R1CS 沒有的東西

「我這個 advice 值一定落在這張預定義表裡。」

  • 用途：range check (0 ≤ x < 2⁸)、S-box、非線性查表
  • R1CS 要做 8-bit range check 得拆 8 個位元 = 8 條 constraint
  • Halo2 一次 lookup 就好，表可由多 circuit 共用
  • 本 PF1 的 ByteTable（256 entries）是整個電路 range check 的萬能工具
Universal SRS29 / 86

Universal SRS — 不是 per-circuit ceremony

Groth16

每個 circuit 各做一次 ceremony，VK 不同；circuit 改了重做

↔

Halo2 + KZG

一次 univariate SRS 所有 circuit 共用；circuit 換了只換 VK，不需新 ceremony

⚠ 仍需 structured reference（toxic waste 問題存在）— 只是「一次搞定，多 circuit 共用」。對金融應用（每日 MDT root 變動）這是 critical feature。

Prove/Verify30 / 86

Prover / Verifier 流程（Plonk-style）

  1. Prover：填 advice → FFT 得多項式 → KZG commit 各 column polynomial
  2. Verifier：Fiat-Shamir 生 random challenge ζ
  3. Prover：在 ζ 點 evaluate 所有多項式 + 建 quotient poly commitment
  4. Prover：送 opening proof（KZG proof of evaluation）
  5. Verifier：一對 pairing 檢查 gate identity、permutation、lookup 全部成立

驗證者 work 是常數級（幾個 pairing）— 這是 KZG 的甜蜜點。Proof 大小 ~KB 級。

Rows & k31 / 86

Row / Column / k

  • k = 2 的幾次方的指數 — 總列數 = 2ᵏ
  • PF1 N=128 production：k ≈ 19 → ~524K rows 上限，實際使用 ~289K rows
  • PF1 N=8 test：k 更小（加速開發）
  • 欄數固定、k 決定 prover FFT size 與 proving time

每增 k 1 — rows 翻倍，proving time 約翻倍。選 k 是「夠用就好」的最小冪次。

MockProver32 / 86

MockProver vs release proving

  • MockProver：不做密碼學，只檢查約束滿足 — 開發階段用、秒級
  • Release (KZG)：實際 commitment + FFT + Fiat-Shamir — benchmark 用
  • 兩者語義相同（約束通過 ⟺ proof 可產出）、但效能差幾個量級

本研究的 B1-B14 copy constraint 審計、negative tests 都先用 MockProver 快速迭代。

Implications33 / 86

這些特性如何影響 PF1 電路設計

  • Custom gate 高 degree → Horner、A2 segment-selector、BS price 一條搞定
  • Lookup → ByteTable 256 entries 服務所有 pp48 range check（fp_mul 6-byte rem、fp_div 8-byte gap、Horner 6-byte 截斷、wide_range 7-byte）
  • Copy constraint → 8 條 Sigma-to-MDT + pipeline 中介值銜接，零 gate 成本
  • Universal SRS → Market data 每日變 root 不用重做 ceremony

PART IV35 / 86

Part IV — Poseidon

Part IV 結束後你會知道：

  • 為什麼電路裡不用 SHA / Keccak — 兩個數量級成本差
  • Poseidon sponge + S-box x⁵ 的形狀
  • 8 full + 57 partial rounds = 65 rows/permutation 的代價拆解
  • PF1 的 Merkle 驗證（85% 的電路成本）為什麼被 Poseidon 定義
Why not SHA36 / 86

為何不用 SHA / Keccak

  • SHA-256 在 ZK 電路中 ~25,000 約束 / hash（位元邏輯拆成布林）
  • Poseidon ~64 約束 / row × 65 rows/permutation — 代數層運算，直接在有限域上
  • Zcash / Filecoin / Polygon 生產驗證、業界標準
  • 兩者差約 兩個數量級 — 這個選擇決定整個系統是否可行

PF1 的 Merkle 驗證（Part A + A'）佔總 rows 85%。如果用 SHA，整個電路直接爆表。

Sponge37 / 86

Sponge 結構

Poseidon 是固定長度的 permutation（t=3）。任意長度輸入透過 sponge 封裝：

  • State vector：t=3 個 field element，分為 rate=2 + capacity=1
  • Absorb：把輸入 chunk 加到 rate 部分 → 跑一次 permutation
  • Squeeze：從 rate 讀 output chunk → 跑一次 permutation

PF1 使用：Position leaf（5 欄位 → 3 次 permutation）、Market Data leaf（8 欄位 → 4 次 permutation）。Merkle 每層 hash(left, right) = 1 次 permutation。

S-box38 / 86

S-box：x ↦ x⁵

Halo2 允許高 degree gate → S-box 直接用 x⁵（而非 x⁻¹ 反逆或 bit-split）。

  • degree 5 足以抵抗差分 / 線性分析（Poseidon 論文證明）
  • 一條 custom gate 直接寫：sq = a·a; p = sq·sq·a（兩條約束）
  • 相容 BLS12-381 base field（gcd(5, p−1) = 1 → x⁵ 是 permutation）
Rounds39 / 86

Full vs Partial rounds

  • Full rounds：所有 t=3 state element 都過 S-box + MDS — 安全邊界
  • Partial rounds：只有第一個 element 過 S-box，其餘線性 — 省約束
  • PF1 實作 (BLS12-381, t=3)：8 full + 57 partial = 65 rows / permutation

BS Delta43 / 86

BS Delta → 三個超越函數

d1 = [ln(S/K) + (r - q + σ²/2) · T] / (σ · √T)
Δ_call = e^(-qT) · N(d1)
N(x)   = 0.5 + 0.5 · erf(x / √2)
  • ln(S/K) — piecewise Remez 4 segments × width 0.25
  • erf(d1/√2) → N(d1) — piecewise Remez 12 segments × width 0.5 + saturation
  • exp(−qT) — Method B byte-decompose + lookup (11 rows)
  • σ√T、σ²/2 — fp_mul + sqrt_verify (Newton)

⚠ Δ_put 由 put-call parity：Δ_put = Δ_call − e^(-qT)，只需一個減法閘。

Fixed-point44 / 86

Fixed-point pp=48

所有中間值 = integer × 2⁻⁴⁸。S = 2⁴⁸ ≈ 281 兆（14.4 位十進位精度）。

  • 加法 ↔ 整數加法（field element）
  • 乘法 → a·b = w·S + rem，rem 右移 48 位（6-byte decomposition + ByteTable lookup）
  • 除法 → w·b + rem = a·S，rem 做 8-byte 分解（除數可能 > 2⁴⁸）
  • sqrt → Newton 迭代，電路內 sqrt_verify: d·(d−1) = 0 where d = a − b（1 ULP）

端到端絕對誤差 ~2×10⁻¹⁴ — 遠超任何金融應用的精度需求。

Horner46 / 86

Per-step Horner kernel

P(x) = c_n xⁿ + ... + c_0，巢狀寫法：P(x) = c_0 + x(c_1 + x(c_2 + ...))

// 每一步約束：acc_next · S + rem = acc · x + coeff
// S=2^48 為截斷（右移 48 位），rem 由 ByteTable 驗證 ∈ [0, S)
meta.create_gate("horner_per_step", |meta| {
  let q    = meta.query_selector(q_horner);
  let acc  = meta.query_advice(acc_col, cur);
  let next = meta.query_advice(acc_col, next);
  let c    = meta.query_fixed(coeff_col, cur);
  let x    = meta.query_advice(x_col, cur);
  let rem  = meta.query_advice(rem_col, cur);
  vec![q * (next * S + rem - acc * x - c)]
});

d=12 多項式 → 12 rows；kernel 精度 ~1.2 ULP / segment。

Why polynomial47 / 86

為何用多項式逼近？

  • ln / erf / exp 在有限域上沒有 closed form — 電路只有 + 和 ×
  • Taylor：展開點附近好，離開急速發散 — 需要極多項
  • Minimax (Remez)：等紋波誤差、最壞誤差最小 — 由 Chebyshev 等振盪定理保證最優
  • pp=48 d=14 相同次數下：erf Minimax 比 Taylor 準 2.6×10⁸ 倍、ln 準 6.8×10¹⁰ 倍

Piecewise49 / 86

Piecewise — ln 4 段、erf 12 段

單一多項式在寬區間會遇到「量化飽和」— 高次係數的定點量化誤差被放大。切窄段後每段 d=12 足夠。

  • ln：[1, 2) 等分 4 段 × width 0.25 + leading-bit decomposition
  • erf：[0, 4] × 等分 12 段 × width 0.5 + saturation tail（|x| > 4 夾到 S）
  • d=12 是經過 degree sweep 找到的最優點（逼近誤差 + 截斷累積的 U 曲線最低點）
Lead-bit50 / 86

ln 的 leading-bit decomposition

把 x > 0 寫成 x = 2ᵉ · m, m ∈ [1, 2)；ln(x) = e·ln(2) + ln(m)。

// lead_bit gate (3 constraints):
//   v = (S + r) · 2^e + low       // v = (S+r)·2^e + low
//   low 做 3-byte decomposition   // 確保 low < 2^e
//   gap 做 3-byte decomposition   // 確保 r+S 在 [S, 2S)

exp 用類似結構的 int_div gate：qT ÷ ln(2) → (k, r)，r ∈ [0, ln2)，之後 exp 由 2ᵏ × exp(r) 組合。

Saturation51 / 86

Saturation 處理

erf(x) → 1 當 |x| 很大；x > 4 直接夾到 S。電路要能「雙向」保證：

  • sat=1 ⇒ abs_in ≥ (N−1)·SEG_WIDTH（否則惡意 prover 可提前飽和）
  • sat=0 ⇒ seg_idx < N−1（否則可能繞過 saturation 直接套錯段）

惡意 prover 故事： 想讓 d₁ 很負時偽造 erf → −1（過飽和），或想繞過 segment 選擇套錯係數 — 這些都被 A2 gate 封閉。

A2 Gate52 / 86

Segment-selector gate (A2)

A2 gate 把 (seg_idx, coeff_slot, abs_in, sat_flag) 四件事綁在一起：

  • seg_idx ↔ coeff：fixed column 係數 slot 由 seg_idx 選，錯配會使 horner_per_step 約束失敗
  • sat flag 雙向綁定：sat_excess = abs_in − (N−1)·W，sat=1 ⇒ sat_excess ≥ 0；sat=0 ⇒ seg_idx < N−1
  • 透過 byte decomposition 強制 sat_excess 與 seg_idx 的 non-negativity — 任何方向的偽造都失敗

這是 PF1 的 soundness 封閉的關鍵 — 把「選段」這個看似無害的控制流鎖死在一條 gate 裡。

Method B exp53 / 86

Method B — exp by chunk lookup

利用 exp(a+b) = exp(a) · exp(b)。把 48-bit 輸入拆成 6 個 byte，每個位置預算一張 256-entry 查表：

  • 6 張查表 × 256 entries = 1,536 總 entries
  • 6 次 lookup（一個 byte 一次）+ 5 次 fp_mul 連乘 = 11 rows
  • 比 d=12 Horner 的 12 rows 更緊湊，精度 ~8 ULP（可接受）

⚠ Method B 內部 lookup 鏈在當前實作為 witness + bs_price 兜底（Section 12.4 Layer 2 已記載為計畫強化）。
