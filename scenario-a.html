Scenario A — Production-Grade Protocol

Black-Scholes Delta Hedging
Privacy Compliance

對沖基金向監管機構證明「投資組合 Delta 風險在合規範圍內」，
但永遠不透露任何持倉細節 — 包括操作哪支股票、持有多少部位。

Problem

為什麼需要隱私合規？

監管機構要求對沖基金定期報告風險暴露。傳統做法是交出完整持倉清單 — 但這等於把交易策略全部公開。競爭對手一旦知道你的部位結構，就能搶先交易（front-running）， 讓建倉成本大幅上升。

傳統做法

基金交出完整持倉 → 監管自己算 → 策略曝光

ZK 合規

基金提交 ZK proof → 監管毫秒級驗證 → 持倉永遠保密

ZK 的獨特價值

零知識證明是唯一能同時做到即時可驗證 + 永久隱私的技術。 證明的存在本身就代表合規 — 無法生成 proof 就是不合規。

Trust Architecture

一次性簽署

為什麼需要四層？

拿掉任何一層都會導致系統崩塌 —

拿掉什麼後果
拿掉 ZK (L3)監管必須看到全部持倉才能驗證 → 隱私消失
拿掉交易所簽名 (L2)無法保證輸入真實 → ZK 證明毫無意義
拿掉 Genesis (L1)信任鏈沒有起點 → 所有後續狀態不可信
拿掉抽查 (L4)交易所串通無代價 → L2 假設站不住
Data Structure

兩棵 Merkle Tree

整個系統的數據結構建立在兩棵互相綁定的 Merkle tree 上。

Trade Action Tree

交易動作樹 — 記錄所有已簽名的交易

  • 每片葉子 = 一筆交易 + 交易所簽名
  • Exchange A 簽了自己的交易
  • Exchange B 簽了自己的交易
  • 沒有任何一家交易所看到全貌

葉子結構：asset, qty, price, direction, timestamp, exchange_sig

Position State Tree

持倉狀態樹 — 記錄當前持倉快照

  • 每片葉子 = 一筆持倉的完整參數
  • State(T) = State(T-1) + apply(Trades(T))
  • 本期持倉 = 上期持倉 + 本期所有交易的淨效果
  • 不足 Nmax 的部分填 dummy 葉子（隱藏真實持倉數量）

葉子結構：asset_id, w, K, expiry_date, type（5 欄位，全部靜態）

市場參數（S, r, q, σ）由 Market Data Table 提供；T 由電路從 expiry_date 和 today 計算

Zero-Knowledge Proofs

兩個 ZK 證明

Proof #1: Delta Compliance

本研究核心貢獻

證明：給定 Position State Tree 中的持倉，按 Black-Scholes 公式計算的投資組合 Delta 滿足 |Δtotal| ≤ ε。

包含三個超越函數：ln、erf、exp — 全部用 Minimax 多項式在 ZK 電路內計算。

Proof #2: State Transition

架構設計（標準技術）

證明：State(T) = State(T-1) + apply(Trades(T))，且每筆交易都有合法交易所的有效簽名。

使用 EdDSA/Babyjubjub 驗簽（~6K constraints/次），不涉及超越函數。

為什麼超越函數是瓶頸？

ZK 電路只能做有限域上的加法和乘法。Black-Scholes 公式中的 ln、exp、erf 都不是加減乘除能算的。 傳統方法用巨大的查找表（50M+ 行），在金融級精度下完全不可行。 我們的 Minimax 多項式方法只需 256 行共用的 ByteTable。

Core Contribution

三個超越函數

Black-Scholes Delta 公式 Δcall = e-qT ⋅ Φ(d1) 需要三種超越函數：

函數用途公式中的位置意義
ln(x) 計算 moneyness ln(S/K) 衡量選擇權是價內還是價外
erf(x) 常態分布 CDF Φ(d1) = ½[1 + erf(d1/√2)] 選擇權被行使的機率
exp(x) 股息折現 e-qT 考慮股息對價值的影響

ln 和 erf 共用同一個 256-entry ByteTable，只換係數向量 — 「函數無關電路」的核心價值。 exp 使用 Method B（Chunk Lookup）— 利用 exp(a+b)=exp(a)·exp(b) 的代數性質拆成小表，每次只需 ~3 rows。

Formal Statement

ZK 陳述與公開輸入

完整的零知識陳述

“我知道一組持倉 (asset_idi, wi, Ki, expiry_datei, typei) 和對應的 State / Market Data Merkle paths，使得：
 (1) 每筆持倉都在 State(T) root 中，且覆蓋所有葉子；
 (2) 市場參數 (Sj, r, qj, settle_pricei) 來自 market_data_root 的公開數據表；
 (3) σi 滿足 BS(S, K, T, r, q, σ) ≈ settle_price（正向驗證）；
 (4) 選擇權 Delta 按 BS 計算，equity Delta = 1，|Δtotal| ≤ ε。”

Proof 存在 = 合規

電路中有 |Δtotal| ≤ τ 的硬約束。如果不滿足，proof 根本無法生成。 所以驗證者不需要看到 Δ 的值 — proof 的存在本身就代表合規。收不到合法 proof 就是不合規。

公開輸入（4 項，驗證者可見）

State(T) root — 持倉的密碼學承諾
market_data_root — 每日結算數據表的承諾
ε — 合規門檻（監管規定）
today — 結算日期（計算 Ti 用）

Δ 的確切數值永遠不公開。

私密見證（驗證者不可見）

asset_idi（標的）、wi（持倉量）、Ki（履約價）、expiry_datei（到期日）、typei（call/put/equity）
Sj（股價）、r（利率）、qj（股息率）、σi（波動率 witness）、settle_pricei

市場參數來自公開表，但查詢行為是私密的（Merkle 路徑不公開）。

Security Analysis

攻擊防禦矩陣

Layer 4 Detail

監管隨機抽查機制

密碼學無法防禦「簽名者本身就是壞人」。Layer 4 用制度設計彌補這個缺口。

01

監管隨機選取 k 個葉子索引

例如 k=5，從 Position State Tree 中指定

02

基金揭露被選中的葉子

持倉明細 + Merkle proof + 產生該持倉的交易歷史鏈

03

監管反向查核交易所

「基金說 2/15 在你們這裡買了 50 手 AAPL call K=180，有嗎？」

04

交叉驗證

Merkle proof 正確 + 交易歷史一致 + 交易所確認 → 通過

統計保證

每期抽查 k 筆，若有 p% 倉位造假，單期發現機率 = 1−(1−p)k。 連續 n 期不被發現的機率 = [(1−p)k]n，隨期數指數衰減 — 即使單期發現率只有 65%，12 個月後未被發現的機率已低於 0.0003%。

核心結論

沒有人看到完整組合 — 但四層信任模型讓每個參與者都能信任系統的正確性。 隱私是永久的，不存在「之後全部公開」的退路。

Assumptions

信任假設

假設層次合理性
監管機構誠實簽署 State(0) L1 監管是系統的存在理由 — 若不信任監管，合規本身無意義
交易所誠實簽署交易 L2 交易所是受監管實體，偽造簽名 = 刑事犯罪
交易所不與基金串通 L4 靠隨機抽查 + 反向查核威懾，非盲目信任
ZK 密碼學安全 L2-L3 基於離散對數困難假設，無已知攻擊

相對傳統做法的改善

信任單一清算所 → 分散到多家交易所 + 監管抽查
完全公開持倉 → 永遠不需要公開完整組合
信任中心化計算方 → ZK 證明消除對計算方的信任需求
