1 What This Work Is (And Isn't)

Framing the contribution precisely

NOT the contribution

"We found better polynomial coefficients using Remez."

Remez is a tool — the same way halo2 is a tool. The Remez exchange algorithm has been known since 1934.

IS the contribution

A complete methodology for high-precision transcendental function evaluation inside zero-knowledge circuits — spanning circuit design, numerical analysis, and end-to-end financial model proofs.

1
Universal Horner Kernel A parameterizable halo2 circuit gadget (3 custom gates, universal ByteTable[0..255]) that evaluates any polynomial. Switching functions = swapping d coefficients — no circuit redesign required.
2
Truncation Strategy Analysis First identification that deferred truncation overflows BN254 at pp≥32, making per-step truncation the only viable path for high-precision ZK arithmetic.
3
Design Space Map 637 (pp, d) configurations across 13 functions — including the "quantization saturation" boundary where increasing polynomial degree hurts precision rather than helping.
4
Coefficient Conditioning Discovery First finding that function structure (coefficient pattern) creates a 10× precision gap at identical kernel parameters. No precedent in ZK fixed-point literature.
5
End-to-End Financial Model Proofs From abstract methodology to working KZG proofs for Black-Scholes option pricing and Merton structural model default probability.

3 Scenario C — High Precision (Merton CDS Portfolio)

Prove portfolio default probability below regulatory threshold without revealing positions

SCENARIO C

Application

A financial institution proves that its portfolio-level probability of default (PD) is below a regulatory threshold using the Merton structural model, without disclosing the asset composition or individual position sizes. Portfolio composition is proprietary intellectual property.

Why High Precision?

Precision Requirement

Investment-grade default probabilities are in the range 0.001–0.01. At these magnitudes, even small arithmetic errors can flip a pass/fail regulatory decision. The methodology must deliver enough effective bits to keep the error well below the decision boundary.

Functions Required

ln(1+x) + erf(x) via Merton structural model: PD = Φ(−d2), where d2 depends on ln(V/D) and the cumulative normal is computed via erf.

4 Ablation Studies — Why Each Component Matters

Isolating the effect of each design decision

A1

Taylor vs Minimax — The Critical Comparison

This is not a marginal optimization. The data below shows that minimax approximation is a prerequisite for high-precision ZK fixed-point transcendentals — not merely an improvement over Taylor series.

Config Taylor ULP Minimax ULP Ratio Taylor Bits Minimax Bits
ln  pp=32 d=10 204,063,608 10.4 19.5M× 4.4 28.6
ln  pp=48 d=14 9,694,560,262,569 142.5 68.0B× 4.9 40.8
erf pp=32 d=10 3,211,380 8.0 402K× 10.4 29.0
erf pp=48 d=14 3,780,048,854 15.2 248M× 16.2 44.1
Critical Insight

The improvement ratio grows with precision. At pp=16, both approaches might look similar. At pp=48, Taylor yields 4.9 effective bits while Minimax delivers 40.8 bits. Minimax is not an optimization — it is a prerequisite.

A2

Deferred Truncation Overflow

Deferred truncation (accumulating the full-precision intermediate and truncating only once at the end) saves constraints but hits a hard wall: BN254's 253-bit field element capacity.

Config Overflow Step Max Bits Required BN254 Capacity Margin Status
pp=24 d=8 Never 212 253 +41 bits OK
pp=32 d=10 Step 8 347 253 −94 bits OVERFLOW
pp=48 d=14 Step 5 713 253 −460 bits OVERFLOW
Resolution

Per-step truncation keeps intermediates bounded: 48 bits (pp=24), 64 bits (pp=32), 96 bits (pp=48) — always well within BN254's 253-bit capacity. This is the only viable strategy for pp≥32.

A3

Precision Scaling

Effective bits scale roughly linearly with precision parameter pp. However, the gap between functions widens at high precision — this is the conditioning effect studied in A4.

Measure pp=16 d=6 pp=24 d=8 pp=32 d=10 pp=48 d=14
ln — Max ULP 4.3 6.2 10.4 142.5
ln — Effective Bits 13.9 21.4 28.6 40.8
erf — Max ULP 4.4 6.8 8.0 15.2
erf — Effective Bits 13.9 21.2 29.0 44.1
Observation

At pp=16, ln and erf have nearly identical precision (13.9 bits each). At pp=48, erf achieves 44.1 bits while ln reaches only 40.8 bits — a growing gap that reveals function-specific conditioning effects.

A4

Coefficient Conditioning — Novel Finding

At identical kernel parameters (pp=48, d=14), the same circuit produces dramatically different precision for different functions. The cause is the structure of the minimax coefficients themselves.

ln(1+x) — 142 ULP

All 14 coefficients are significant. Alternating signs cause cancellation during fixed-point Horner evaluation, amplifying rounding errors at each step.

Same kernel. Same parameters. Same circuit. The only difference is the coefficient vector loaded into the gadget. This has no precedent in ZK fixed-point literature and has direct implications for circuit designers choosing which functions to support at which precision targets.

A5

Lookup Table Comparison

Comparing our polynomial+ByteTable approach against the two main alternatives: direct lookup and segmented lookup tables.

Dimension Direct Lookup Segmented Lookup Ours (Polynomial + ByteTable)
Max pp supported 16 ~28 48+
Table entries (pp=32) IMPOSSIBLE ~1,280 / function 256 (universal)
Proof size (pp=32) N/A ~60 KB ~7 KB
Adding new function New huge table New segment table New coefficients only
Advantage

The universal ByteTable[0..255] is shared across all functions. Direct lookup cannot reach pp=32 (it would require 232 = 4 billion entries). Segmented lookup scales better but still needs per-function tables and produces 8× larger proofs.

6 Known Limitations

Honest assessment of current scope boundaries

1

erf domain restriction. The erf approximation covers [0, 1], which restricts the Merton model to default probabilities above approximately 0.08 (speculative-grade). Investment-grade PDs (below 0.01) would require extending the domain via range reduction or piecewise approximation.

2

pp=48 full circuit not yet built. The pp=48 d=14 results are kernel-level benchmarks (isolated Horner evaluation). The complete Merton circuit at pp=48 has not been integrated end-to-end. Constraint counts and proving times may differ from kernel extrapolations.

3

Debug-build proving times. The ~15 second proving times for Scenario C were measured in Rust debug mode. Release-build benchmarks with compiler optimizations are expected to be significantly faster but have not been formally measured.

4

Portfolio batching not implemented. The current implementation proves a single (V, D, σ) tuple per proof. Batching multiple positions into a single proof — required for real portfolio-level attestation — is architecturally supported by the kernel design but not yet implemented.

All experiments implemented in Rust (halo2 + KZG). Tests: 34/34 pass. Coefficients generated via Python mpmath Remez exchange at 80-digit precision.
