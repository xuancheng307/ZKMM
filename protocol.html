01 — Protocol Overview

協議總覽

每個結算日，對沖基金透過以下流程向監管機構證明合規：

交易所

簽署交易 + 發布批次承諾

每家交易所簽署自己執行的交易，並發布 batch_root（本期所有交易的 Merkle 承諾）

數據商

發布 Market Data Table

每日收盤後，數據提供者發布結算數據（股價 S、利率 r、股息率 q、結算價 settle_price），承諾為 Merkle tree

基金

建構 Trade Action Tree

將本期所有已簽名交易組織成 Merkle tree

基金

計算狀態轉移 State(T) = State(T-1) + apply(Trades)

將交易效果套用到上期持倉，得到本期持倉狀態樹

基金

生成 Proof #2（State Transition）

證明每筆交易都有交易所簽名，且 State(T) = State(T-1) + Trades(T)

基金

計算 σ 並生成 Proof #1（Delta Compliance）

用 Newton-Raphson 反推 σ，在電路內驗證 BS(σ) ≈ settle_price，計算 Delta，驗證 |Δ| ≤ ε

基金

提交給監管機構

提交 State(T) root + market_data_root + τ + Proof #1 + Proof #2

監管

驗證兩個 proof（毫秒級）

Proof #1 通過 → Delta 合規。Proof #2 通過 → 持倉真實。兩者都通過 → 合規確認。

核心設計原則

Proof 的存在本身就代表合規 — 無法生成 proof 就是不合規。 監管機構不需要看到 Δ 的數值，不需要知道基金操作哪些標的。 兩個 proof 獨立驗證：Proof #2 產出的 verified root 作為 Proof #1 的公開輸入。

02 — Merkle Trees

三棵 Merkle Tree

整個協議建立在三棵 Merkle tree 上。每棵樹承諾不同類型的數據，由不同角色維護。

Position State Tree

持倉狀態樹 — 當前持倉的完整快照

用途：記錄基金在每個結算時點的所有持倉。 每片葉子代表一筆選擇權部位，不足 Nmax 的部分以 dummy 葉子（w=0）填充， 確保驗證者無法從 trace 長度推斷真實持倉數量。

更新時機：僅在交易發生時更新（透過 Proof #2 的 State Transition）。

葉子結構：

asset_id w K T opt_type
欄位意義來源
asset_id標的資產識別碼交易簽名
w持倉數量（正=多，負=空，0=dummy）交易簽名
K履約價格（fixed-point pp=24）交易簽名
T到期時間（fixed-point pp=24）交易簽名
opt_type選擇權類型（0=call, 1=put）交易簽名
深度：7
最大葉子：128 (Nmax)
Hash：Poseidon
Root 公開：state_root

Trade Action Tree

交易動作樹 — 本期所有已簽名的交易

用途：記錄結算期內所有交易動作。每片葉子代表一筆已簽名的交易。 交易所各自只簽署自己執行的交易 — 沒有任何一家交易所看到基金的完整組合。

更新時機：每個結算期重新建構。

葉子結構：

asset_id qty price direction K T opt_type timestamp exchange_id signature

交易內容   簽章後設資料

深度：動態
葉子數：T_MAX/期
Hash：Poseidon
Root 公開：batch_root

Market Data Table

市場數據表 — 每日收盤結算數據

用途：提供非交易參數（股價 S、利率 r、股息率 q、結算價格）的公開可驗證來源。 消除了「基金可以填假的市場參數」的信任缺口。 表本身是公開的，但基金查詢了哪些條目（即操作哪些標的）是私密的。

更新時機：每日收盤後由數據提供者重新發布。電路結構和 VK 不受影響 — 僅 root 值更新。

葉子結構：

asset_id S r q settle_price
欄位意義備註
asset_id標的資產識別碼與 Position State Tree 中的 asset_id 對應
S股價（收盤價，fixed-point pp=24）每日更新
r無風險利率（全域參數）所有標的共用
q股息率（per-asset）每日更新
settle_price選擇權結算價格用於 σ 驗證：BS(S,K,T,r,q,σ) ≈ settle_price
深度：≈ 20
最大葉子：≈ 1M 條目
Hash：Poseidon
Root 公開：market_data_root

三棵樹的關係

Trade Action Tree 驅動 Position State Tree 的更新 （State(T) = State(T-1) + apply(Trades)，由 Proof #2 保證）。
Position State Tree 的每筆部位透過 asset_id 連結到 Market Data Table （市場參數查詢，由 Proof #1 內的 Merkle 驗證保證）。
三棵樹的 root 全部作為公開輸入 — 驗證者只看到三個 hash 值，看不到任何內容。

03 — Zero-Knowledge Proofs

兩個零知識證明

兩個 proof 各自獨立生成、獨立驗證。Proof #2 的輸出（verified state_root）成為 Proof #1 的公開輸入， 不需要遞歸證明組合（recursive composition）。

#1

Delta Compliance

Delta 合規證明 — 本研究核心

證明基金的投資組合 Delta 風險在合規範圍內，且所有計算參數來自可驗證的公開數據。

公開輸入（3 個）
state_root — Position State Tree root
market_data_root — Market Data Table root
τ — 合規門檻（監管規定）
私密見證
每筆部位的 (asset_id, w, K, T, type)
每筆部位的 State Merkle path
每筆部位的 Market Data Merkle path
每筆部位的 σ（prover 計算，電路驗證）

電路內操作：State Merkle 驗證 → Market Data 查詢 → σ 驗證 → Black-Scholes Delta 計算（ln, erf, exp）→ 累加 → |Δ| ≤ τ

#2

State Transition

狀態轉移證明 — 標準技術

證明持倉狀態的每次變化都可追溯到已簽名的合法交易，且沒有遺漏。

公開輸入（3 個）
state_root_prev — State(T-1) root
state_root — State(T) root
batch_root — 交易所批次承諾 root
私密見證
每筆交易的完整參數 + 交易所簽名
每筆交易的 Merkle path（舊根 → 新根）
增量式 Merkle 更新鏈

電路內操作：驗證交易所簽名（EdDSA）→ 驗證交易承諾 → 增量式 Merkle 更新 → 鏈接 root 轉移

完整的零知識陳述

Proof #1：「我知道一組持倉和對應的 Merkle paths，使得：
 (1) 每筆持倉都在 state_root 中，且覆蓋所有葉子；
 (2) 每筆持倉的市場參數來自 market_data_root 承諾的公開數據表；
 (3) 每筆 σi 滿足 BS(S,K,T,r,q,σ) ≈ settle_price；
 (4) |Δtotal| ≤ τ。」

Proof #2：「我知道一批交易 Trades(T)，使得：
 (1) 每筆交易都有合法交易所的簽名；
 (2) 所有交易的承諾 = 交易所公布的 batch_root；
 (3) 依序套用交易到 State(T-1) 得到 State(T)。」

06 — In-Circuit Tables

電路內查表

Halo2 PLONKish 電路支援 lookup argument — 用固定的表格約束值域。 以下是 Proof #1 電路中使用的查表結構。

ByteTable（256 行）

用途：Minimax Horner kernel 的多項式係數查表。 ln、erf、exp 三種超越函數共用同一張表，只換係數向量。

結構：每行 = (byte_index, coeff_ln, coeff_erf, coeff_exp)。 byte_index ∈ [0, 255] 對應輸入的高 8 位元。

性質：Fixed column，在 setup 時綁定進 VK。 永遠不變（數學常數），不受每日數據影響。

256 行 × 27 lookup arguments / 呼叫

Poseidon 常數

用途：Poseidon hash 函數的 round constants 和 MDS matrix。 所有 Merkle 驗證（State Tree + Market Data Table + Trade Tree）共用。

結構：固定的域元素向量，約 ~200 個常數。

性質：Fixed column / 硬編碼常數。 由 Poseidon 規格決定，永遠不變。

PERM_ROWS ≈ 66 行 / 排列

為什麼 Market Data Table 不是電路內查表？

Halo2 的 fixed column 在 setup 時綁定進 verifying key（VK）。Market Data 每日變動 — 如果用 fixed column，就必須每天重跑 trusted setup + 更換 VK。對驗證者完全不可行。
因此 Market Data Table 使用 Merkle 承諾（root 作為公開輸入）， 電路結構和 VK 永遠不變，只有輸入值每天更新。

為什麼選 EdDSA on Babyjubjub？

Babyjubjub 是專為 ZK 電路設計的橢圓曲線，與 BN254 原生相容。 EdDSA/Babyjubjub 每次驗簽約 ~6,000 constraints， 而 ECDSA（以太坊標準）需要 ~20,000-30,000 constraints。 在 Proof #2 中可能需要驗證數十次簽名 — 3-5 倍的效率差距很關鍵。

部署流程

一次性 Setup：生成電路的 proving key (PK) 和 verifying key (VK)。 電路結構固定 — VK 永遠不需要更新（Market Data 用 Merkle 而非 fixed column）。
每日 Proving：基金在收盤後約 10 分鐘內完成兩個 proof 的生成。 唯一變動的是公開輸入值（root 值和 τ）。
每日 Verification：監管收到 proof 後毫秒級驗證。 如果兩個 proof 都通過 → 合規確認。

瓶頸分析

Market Data Merkle 路徑驗證（depth ≈ 20）是最大的行數成本（55%）， 但這是保證非交易參數（S, r, q, settle_price）真實性的必要代價。 不使用 Halo2 lookup argument 的原因是 Market Data 每日變動， 而 fixed column 在 setup 時綁定 VK — 不可能每天重跑 setup。

我們的多項式方法（Part B: Delta 管線）僅佔 3% — 這證明了 Minimax Horner kernel 的高效率： 三個超越函數的電路成本已經不是瓶頸。
