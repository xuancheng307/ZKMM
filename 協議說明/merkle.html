Merkle Tree 實作細節

電路內 Poseidon Merkle 樹 — 從葉節點雜湊到根驗證

持倉狀態樹、市場數據樹、交易批次樹的統一架構

1為什麼需要 Merkle Tree？

在 ZKMM 協議中，監管機構需要驗證基金的持倉狀態和市場數據是否正確，但不能看到具體持倉細節。Merkle 樹提供了完美的解決方案：

核心思想：將大量數據壓縮為一個根雜湊（root hash）作為公開輸入。電路內部驗證每筆數據確實屬於該根雜湊對應的樹 — 而驗證者只看到根雜湊，看不到任何具體數據。

協議中使用三棵 Merkle 樹：

樹名稱 葉節點內容 欄位數 用途 所屬證明
Position State Tree 基金持倉部位 5 承諾所有持倉 Proof #1 & #2
Market Data Tree 市場行情數據 8 承諾價格/利率/結算價 Proof #1
Trade Action Tree 簽章交易記錄 可變 承諾每筆交易 Proof #2

圖：depth=3 的完整二元 Merkle 樹（示意，實際 depth=7 共 128 葉）。每個 H() 是一次 Poseidon 置換。

每個內部節點的計算方式為 H(left, right)：將左右子節點的雜湊值作為輸入，經過一次 Poseidon 海綿吸收得到父節點雜湊值。

2.3 Merkle 路徑（Membership Proof）

要證明某個葉節點 $L_i$ 屬於根 $R$ 對應的樹，需要提供一條認證路徑（authentication path）：

Merkle 路徑 = 從葉到根途中，每層的兄弟節點雜湊值（sibling hash）加上位置位元（position bit，標記自己是左子還是右子）。

以 depth=3、驗證 $L_5$（index=5，二進位 101）為例：

層級 position bit 含義 兄弟節點 計算
0（葉層） 1 $L_5$ 是右子 $L_4$ $h_0 = H(L_4, L_5)$
1 0 $h_0$ 是左子 $H_3 = H(L_6, L_7)$ $h_1 = H(h_0, H_3)$
2 1 $h_1$ 是右子 $H_{01}$ $h_2 = H(H_{01}, h_1)$

驗證：$h_2 \stackrel{?}{=} \text{Root}$。如果相等，則 $L_5$ 確實屬於該樹。

複雜度：驗證一條路徑只需 $d$ 次雜湊（$d$ = 深度），與葉節點總數 $N$ 無關。$N=128$ 時只需 7 次 Poseidon 置換。

3葉節點雜湊：Poseidon 海綿

葉節點不是單一域元素，而是由多個欄位組成的結構體。需要先將所有欄位雜湊為一個域元素，再放入 Merkle 樹。

3.1 Position 葉節點（5 欄位）

# 欄位 型別 說明
0 w fp24 (signed) 持有數量（2's complement 表示負數）
1 k fp24 履約價（equity 為 0）
2 expiry_date 整數 到期日（days since epoch；equity 為 0）
3 asset_type 列舉 0 = 歐式 Call, 1 = 歐式 Put, 2 = Equity
4 underlying_id 整數 標的資產編號

海綿雜湊流程（3 輪吸收）

使用 Poseidon 海綿模式（$t=3$, rate $r=2$, capacity $c=1$），將 5 個欄位分組吸收：

[0, 0, 0] → +[w, k] → π → +[expiry, type] → π → +[id, 0] → π → output = s[1]

每次吸收將一對欄位加到狀態的速率部分：

$$\text{state}[1] \mathrel{+}= \text{field}[2i], \quad \text{state}[2] \mathrel{+}= \text{field}[2i+1]$$

然後施加完整的 Poseidon 置換 $\pi$（65 輪：4 full + 57 partial + 4 full）。5 個欄位需要 $\lceil 5/2 \rceil = 3$ 次吸收，最後一次的第二個位置填 0。

對應實作：hash_position_leaf() — 迴圈每次取 2 個欄位加入 state[1] 和 state[2]，然後調用 permute()。最終輸出 state[1]。

3.2 Market Data 葉節點（8 欄位）

# 欄位 型別 說明
0underlying_id整數標的資產編號
1strikefp24履約價（合約鍵值）
2expiry_date整數到期日（合約鍵值）
3asset_type列舉資產類型（合約鍵值）
4spotfp24現貨價格 $S$
5rfp24無風險利率
6qfp24股利率
7settle_pricefp24官方結算價格

8 個欄位需要 $\lceil 8/2 \rceil = 4$ 次海綿吸收，流程與 Position 葉節點相同但多一輪：

[0, 0, 0] → +[id, strike] → π → +[expiry, type] → π → +[spot, r] → π → +[q, settle] → π → output = s[1]

3.3 Dummy 葉節點

當基金的實際持倉數量少於 $N_{\max} = 128$ 時，剩餘的葉節點以空白部位（dummy leaf）填充。Dummy 的設計需要避免影響計算正確性：

欄位 Dummy 值 設計原因
w 0 持有量為零 → 對 Delta 無貢獻
k $1 \times 2^{24}$（= $S$, ATM） 避免 $K/S$ 產生除以零
expiry_date 0 不影響時間計算
asset_type 0（EU Call） 任意合法值
underlying_id 0 指向預設標的
為什麼 k 不能設為 0？Proof #1 電路中，Black-Scholes 計算涉及 $d_1 = \frac{\ln(S/K) + \ldots}{\sigma\sqrt{T}}$。若 $K = 0$，則 $\ln(S/K) \to \infty$，導致電路中的定點運算溢出。設置 $K = S$（ATM）使得 $\ln(S/K) = 0$，安全地避開此問題。

4內部節點：2-to-1 Poseidon 雜湊

Merkle 樹的每個內部節點將兩個子雜湊壓縮為一個父雜湊：

$$\text{parent} = H_{\text{2to1}}(\text{left}, \text{right}) = \text{Poseidon}([0, \text{left}, \text{right}]).\text{squeeze}()$$

具體步驟：

  1. 初始狀態 $[0, 0, 0]$
  2. 吸收：$\text{state}[1] \mathrel{+}= \text{left}$, $\text{state}[2] \mathrel{+}= \text{right}$
  3. 執行一次完整 Poseidon 置換 $\pi$（66 列）
  4. 輸出 $\text{state}[1]$
效率：每個內部節點只需 1 次 Poseidon 置換 = 66 列。整條 Merkle 路徑（7 層）= $7 \times 66 = 462$ 列。加上葉節點雜湊（3 次置換 = 198 列），一次完整驗證共 $\approx 660$ 列。

5電路內 Merkle 路徑驗證

在零知識電路中驗證 Merkle 路徑需要三個核心元件：葉節點雜湊、條件交換閘、Poseidon 置換。

5.1 電路佈局

元件 Advice 列 Selector 功能
Poseidon state s[0], s[1], s[2] — Poseidon 狀態暫存器
absorb absorb[0], absorb[1] q_absorb 海綿吸收值（欄位 / 兄弟節點 / 當前雜湊）
bit bit q_swap 位置位元（0=左, 1=右）

5.2 條件交換閘（Merkle Swap Gate）

在每層 Merkle 路徑中，需要根據 position bit $b$ 決定誰是左子、誰是右子：

若 $b = 0$（當前節點是左子）：$\text{left} = \text{cur}$, $\text{right} = \text{sib}$
若 $b = 1$（當前節點是右子）：$\text{left} = \text{sib}$, $\text{right} = \text{cur}$

電路中以三個約束實現此邏輯（無需條件分支）：

$$b \cdot (1 - b) = 0 \quad \text{（布林約束：確保 } b \in \{0, 1\}\text{）}$$ $$\text{left} = \text{cur} + b \cdot (\text{sib} - \text{cur})$$ $$\text{right} = \text{sib} + b \cdot (\text{cur} - \text{sib})$$

當 $b = 0$：$\text{left} = \text{cur}$, $\text{right} = \text{sib}$。
當 $b = 1$：$\text{left} = \text{cur} + (\text{sib} - \text{cur}) = \text{sib}$, $\text{right} = \text{sib} + (\text{cur} - \text{sib}) = \text{cur}$。

為什麼不用 if-else？算術電路沒有分支指令。所有路徑都必須執行。Swap gate 以純代數方式實現條件選擇，每層只需 3 個約束。

5.3 完整驗證流程

一次 Merkle 驗證的完整電路流程：

階段一：葉節點雜湊（Leaf Hashing）

初始化
[0, 0, 0] → 吸收 pair 1
+ [f₀, f₁] → Poseidon π
66 列 → 吸收 pair 2
+ [f₂, f₃] → Poseidon π
66 列 → ... → leaf_hash
= s[1]

階段二：路徑驗證（Path Verification）

對每一層 $\ell = 0, 1, \ldots, d-1$：

讀入
cur, sib, bit → Swap Gate
3 約束 → 吸收
[left, right] → Poseidon π
66 列 → next_hash
= s[1]

最後一層的 next_hash 即為計算出的根雜湊，與公開輸入的 state_root 或 market_data_root 進行約束等式檢查。

5.4 電路返回值

Merkle 驗證函數返回三個關鍵的已賦值單元（AssignedCell），供後續電路使用：

返回值 用途
leaf_cells[] 個別葉節點欄位的 cell — 傳遞給 Delta 計算、時間驗證等後續電路
leaf_hash_cell 葉節點雜湊值 — 用於跨電路一致性檢查
root_cell 根雜湊值 — 約束到 instance 列（公開輸入）
一魚兩吃：Merkle 驗證不僅確認葉節點「屬於」承諾的樹，同時將每個欄位暴露為已賦值 cell，使後續電路可以直接引用這些欄位值 — 無需重複輸入或額外的等式約束。

6約束計數分析

6.1 每次驗證的列數

電路列數計算公式：

$$\text{merkle\_rows}(n_f, d) = \left(\left\lceil \frac{n_f}{2} \right\rceil + d\right) \times 66$$

其中 $n_f$ = 葉節點欄位數，$d$ = 樹深度，66 = Poseidon 每次置換的列數。

樹類型 欄位數 $n_f$ 海綿輪數 路徑層數 $d$ 總置換次數 總列數
Position 5 3 7 10 660
Market Data 8 4 7 11 726

6.2 在完整電路中的佔比

在 Proof #1 中，每個持倉部位需要兩次 Merkle 驗證（一次 Position + 一次 Market Data）：

元件 列數/部位 佔比
Position Merkle 驗證 660 ~33%
Market Data Merkle 驗證 726 ~36%
Delta 計算 + 合規檢查 ~631 ~31%
總計 ~2,017 100%
觀察：Merkle 驗證佔了每個部位約 69% 的電路列數。這是因為 Poseidon 置換本身需要 66 列（65 輪 + 1 列初始化），而每次驗證需要 10-11 次置換。Poseidon 的「電路友好」是相對於 SHA-256（~25,000 列/次）而言 — 在絕對數量上，Poseidon 仍是 Merkle 驗證的主要成本。

7設計決策：為什麼不用 Lookup Table？

Halo2 提供強大的 Lookup Argument，可以用查找表代替部分約束。為什麼 Market Data 不直接用 lookup table 查詢，而是用 Merkle 樹？

問題：Lookup table 是 setup 階段固定的（寫入 fixed column）。Market Data（現貨價格、利率等）每天都在變。如果用 lookup，每天都需要重新跑 setup — 產生新的 Proving Key 和 Verifying Key。
Merkle 解決方案：Market Data 放在 witness（private input）中，由 Merkle 證明綁定到公開的 market_data_root。Setup / VK / PK 不隨數據改變 — 只有 root 作為 instance 更新。這實現了一次 setup，永久使用。
面向 Lookup Table 方式 Merkle Tree 方式（我們的選擇）
數據位置 Fixed column（setup 時固定） Advice column（每次證明可變）
數據更新 需要重新 setup 只更新 root（instance）
VK / PK 隨數據變化 固定不變
每筆查詢約束 ~1 行（lookup 非常便宜） ~726 行（完整 Merkle 路徑）
適用場景 靜態數據（範圍檢查、S-box） 動態數據（每日價格、利率）

8跨樹約束

在 Proof #1 中，Position 樹和 Market Data 樹之間存在語義關聯：每個持倉的市場數據必須與持倉的合約參數匹配。

8.1 跨樹一致性約束

電路中強制以下等式：

Position 欄位 $=$ Market Data 欄位 意義
underlying_id $=$ underlying_id 同一標的資產
k $=$ strike 同一履約價
expiry_date $=$ expiry_date 同一到期日
asset_type $=$ asset_type 同一合約類型

這些約束確保基金不能為某個持倉「搭配」錯誤的市場數據 — 例如用低波動度的數據來隱瞞高風險部位。

8.2 公開輸入綁定

兩棵樹的根雜湊分別綁定到 Proof #1 的公開輸入：

Position Merkle 驗證 → root_cell → instance[0]
state_root
Market Data Merkle 驗證 → root_cell → instance[1]
market_data_root

constrain_instance(root_cell, instance, 0) 將電路內計算出的根雜湊與公開的 state_root 綁定。驗證者只需檢查 instance 值 — 無需知道樹的內容。
