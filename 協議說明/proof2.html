Proof #2: State Transition

持倉狀態轉移證明 — 技術細節

BabyJubjub 曲線 · EdDSA-Poseidon 簽名 · 3-bit Op_Type · 批次認證

各位好，接下來我要介紹的是 Proof #2 — 狀態轉移證明。

如果說 Proof #1 回答的是「基金現在的風險合不合規」，那 Proof #2 回答的就是「基金的持倉是怎麼從上一期變成這一期的」。

這個證明的核心目標是：確保每一筆持倉變動都經過交易所的合法授權 — 基金不能自己憑空修改帳上的部位。我們在零知識電路裡面驗證 EdDSA 數位簽名，來達成這個目標。

0技術選型動機

Proof #2 的每一項技術選擇都受到同一個核心約束驅動：所有密碼學運算必須在 ZK 電路內完成。這意味著每個元件的成本不是「CPU 上跑多快」，而是「轉換成算術約束需要多少列」。以下是主要設計決策的全景：

層級選擇核心理由被淘汰的替代方案

統一的成本度量：所有選擇都以「halo2 PLONKish rows」為單位衡量。一個看似更安全或更通用的方案，如果約束數膨脹 10 倍，就會導致 k 增加 3-4 級（prove time 增長 8-16 倍）。Proof #2 的設計原則是：在保證 128-bit 安全性的前提下，最小化約束數。
在開始技術細節之前，我想先解釋一下整體的設計哲學。

Proof #2 的每一個技術選擇，都圍繞著一個核心問題：「在 ZK 電路裡面，這個操作需要多少約束？」我們不是在選「最好」的密碼學方案，而是在選「在算術電路裡面最便宜、同時安全性足夠」的方案。

舉個例子：SHA-256 在 CPU 上很快、非常成熟，但轉換成算術約束大約需要 25,000 列 — 因為它大量使用 bit rotation 和 XOR，這些操作在有限域的算術電路中非常昂貴。相反，Poseidon 是為有限域設計的，它的 S-box 就是一個簡單的 x5 運算，自然映射到多項式約束，一次 permutation 只需要 66 列。

同樣的邏輯適用於簽名曲線（BabyJubjub 的域恰好是 BN254 的原生域）、簽名方案（EdDSA 不需要 modular inverse）、甚至批次結構（用 copy constraint 而非額外的 EdDSA 驗證）。每一個選擇都是約束成本的最優化。

2狀態轉移模型

Proof #2 證明從 State(T−1) 到 State(T) 的轉換完全由合法交易驅動，沒有未經授權的狀態變更。每個批次處理 T_MAX = 4 筆交易。

為什麼 T_MAX = 4？每筆交易需要 ~3,838 列（EdDSA 驗證 + Poseidon hash），4 筆 × 3,838 + 198（批次 hash）≈ 15,550 列，恰好落在 k = 14（16,384 列）的範圍內，利用率約 95%。增加到 8 筆需要 k = 15（prove time 約翻倍）；減少到 2 筆則浪費近半電路空間。4 筆在效能和實用性之間取得平衡 — 不足 4 筆時用 NOP（空操作）填充，NOP 的 EdDSA 仍然會被驗證（使用 dummy 簽名），確保電路結構固定、不洩漏實際交易筆數。

這是整個處理流程。從左到右：上期的持倉狀態進來，每一筆交易先驗證 EdDSA 簽名，然後根據 3-bit 的操作類型碼來決定葉子怎麼變，最後更新 Merkle root。四筆交易依序處理，root 像鏈子一樣串起來。

為什麼是 4 筆？因為每筆交易需要約 3,838 列（EdDSA 驗證 + Poseidon hash），4 筆加上批次 hash 共約 15,550 列，恰好放在 k=14 的電路裡（16,384 列），利用率約 95%。如果增加到 8 筆就需要 k=15，prove time 大約翻倍。4 是成本和實用性的甜蜜點。

為什麼必須逐筆依序處理而不是平行？因為 root chaining — 後一筆交易的 old root 必須等於前一筆的 new root。這個序列性是設計上刻意的：它保證了交易執行順序的不可篡改性。

2.1 操作類型 — 3-bit 編碼

操作類型使用 3-bit 編碼：op = b0 + 2 × b1 + 4 × b2。電路以布林約束強制每一位為 0 或 1，並以多項式約束拒絕保留代碼。

為什麼用 3-bit 布林分解而不用 Lookup Table？Lookup table 雖然更靈活（可以任意映射輸入到輸出），但需要專門的 lookup column 和額外的 table 初始化列數。3-bit 方案只用 3 個布林約束（每個 bi(1−bi) = 0）加上 5 個 selector gate，總共 8 列。更關鍵的是：每個 bit 可以直接復用作 selector — 例如 b1 同時扮演「是否為 CLOSE 或 EXPIRY」的指示器（因為 b1 = 1 時包含代碼 2, 3, 6, 7），省去額外的條件分支。這種 bit 復用技巧讓約束矩陣既緊湊又可讀。
代碼b2b1b0名稱語義 0000TRADE全欄位替換（新開倉） 1001EXERCISE行權：到期日歸零，w 可調 2010CLOSE平倉：w 歸零 3011EXPIRY到期：w 歸零 4100CORPORATE_ACTION公司行動：K/w 比例調整 5101RESERVED電路拒絕 6110RESERVED電路拒絕 7111NOP空操作：所有欄位不變
這裡是 3-bit 操作類型的完整編碼表。用三個 bit 可以表達 8 種代碼，我們定義了 6 種合法操作，保留了 2 種（代碼 5 和 6）作為未來擴展。

電路用一個巧妙的多項式來拒絕保留代碼：b2 × (b0 + b1 − 2 × b0 × b1) = 0。這個表達式只在代碼 5 和 6 時不為零。

比較有趣的設計是：CLOSE 和 EXPIRY 的差別只在語義層面 — 電路約束是一樣的，都是 w 歸零。但區分它們可以在鏈下的審計記錄中提供更豐富的資訊。

每種操作類型對 5 個葉子欄位施加不同的約束。「自由」表示新值不受限，「不變」表示必須等於舊值，「=0」表示必須為零：

這張表是整個 op_type 約束系統的核心。我來解讀幾個重點：

TRADE 是唯一可以自由修改所有五個欄位的操作 — 因為它代表全新的開倉。

EXERCISE 讓 w 自由改變、到期日必須歸零 — 因為行權就是消滅選擇權，改成結算後的部位。注意 K 不能變，因為行權是按原始履約價結算的。

CLOSE 和 EXPIRY 都強制 w 歸零 — 代表這個部位被清空了。

CORPORATE_ACTION 可以改 w 和 K — 典型例子是股票分拆，一股變兩股的同時履約價減半。但到期日和資產類型不能動。

NOP 什麼都不能改 — 它只是用來填充批次的。

這些約束用了 8 列、5 個 selector gate，靠巧妙地復用 bit 組合來實現不同的「OR」邏輯。
靜態葉節點原則：每片葉子一旦建立就不可隨意修改。「部分平倉」實現為 CLOSE 舊葉子 + TRADE 新葉子（reduced w）的兩步操作，確保 Merkle root 的唯一性和可追溯性。

2.3 Root 鏈接

交易按順序逐筆套用。每筆交易同時提供 old Merkle path 和 new Merkle path，電路驗證兩條路徑的 siblings 和 position bits 完全相同：

root0 = state_root_prev   （公開輸入 instance[0]）
root1 = MerkleUpdate(root0, idx1, old_leaf1 → new_leaf1)
root2 = MerkleUpdate(root1, idx2, old_leaf2 → new_leaf2)
root3 = MerkleUpdate(root2, idx3, old_leaf3 → new_leaf3)
root4 = state_root   （公開輸入 instance[1]）

電路約束：每筆交易的 old_root = 前一筆的 new_root（copy constraint）
Root 鏈接是 Proof #2 的骨幹。4 筆交易依序處理，每一筆產生一個新的 Merkle root。電路用 copy constraint 強制連續性 — 第二筆的 old root 必須等於第一筆的 new root。

最後一個 root 必須等於公開輸入的 state_root。這就形成了一條不可篡改的「root chain」。任何人跳過一筆交易、重新排序、或修改中間狀態，都會導致 root 不匹配。

3BabyJubjub 曲線

EdDSA 簽名需要橢圓曲線運算。本協議使用 BabyJubjub — 一條定義在 BN254 scalar field 上的 twisted Edwards 曲線。

3.1 為什麼選 BabyJubjub

Native 運算：BabyJubjub 的基域 = BN254 的 scalar field (Fr)。這意味著所有曲線運算都是有限域的原生運算 — 不需要非原生域模擬（non-native field arithmetic），省去大量約束。相比之下，在 halo2 內驗證 secp256k1 需要模擬 256-bit 非原生域，成本高約 10 倍。
為什麼選 BabyJubjub？核心原因就一個字：效率。

我們的 ZK 電路運行在 BN254 的 scalar field 上，也就是 halo2 的原生域。BabyJubjub 恰好就定義在這個域上 — 所以每一次曲線上的加法、乘法，都直接對應到電路原生的有限域運算，不需要任何額外的 overhead。

如果我們用 secp256k1（就是以太坊用的那條曲線），每個域元素的運算都需要用多個原生域元素來模擬 — 這會讓約束數膨脹 10 倍以上。

3.2 為什麼用 Twisted Edwards 形式

橢圓曲線有多種表示方式。Twisted Edwards 形式的核心優勢是 complete addition law（完備加法公式）：

Complete Addition Law：同一組公式可以處理所有情況 — 包括 P + Q、P + P（doubling）、P + O（identity）。這在電路中至關重要：我們不需要「if P == Q then double else add」的分支邏輯。分支在 ZK 電路中非常昂貴（需要 conditional gate），而 complete 公式讓每一步都只走一條路，消除所有邊界情況的約束成本。

為什麼用 twisted Edwards 而不是更常見的 Weierstrass 形式？

關鍵在於 complete addition law。Weierstrass 曲線的加法公式在 P = Q（需要用 doubling 公式）和 P = -Q（結果是無窮遠點）時需要特殊處理。在 CPU 上用 if-else 就解決了，但在 ZK 電路裡，分支意味著你要同時計算兩條路徑再用 conditional select 挑一條 — 成本翻倍。

Twisted Edwards 的加法公式在所有情況下都一樣。BabyJubjub 的 a = 168700 是 BN254 Fr 中的二次剩餘，所以加法公式是 complete 的。我們可以用完全相同的 gate 處理 double、add、甚至加上 identity point，不需要任何分支。

4EdDSA-Poseidon 簽名驗證

每筆交易都需要交易所的 EdDSA 簽名。電路在內部驗證簽名，確保交易不可偽造。

4.1 為什麼選 EdDSA（而非 ECDSA 或 BLS）

數位簽名方案的選擇直接影響電路成本。以下比較三種主流方案在 ZK 電路內驗證的差異：

方案驗證方程電路瓶頸估計列數
EdDSA
(Schnorr 變體)		 S×B = R + h×A 2 次 scalar mul + 1 次 point add ~2,238
ECDSA R = (h×s−1)×G + (r×s−1)×A 需要 modular inverse s−1 mod l（在電路中需 ~1,000+ 列的擴展歐幾里德算法） ~5,000+
BLS e(S, G2) = e(H(m), pk) 需要 pairing 運算（Miller loop + final exponentiation，在電路中不可行） >100,000
EdDSA 的兩個關鍵優勢：
(1) 無 modular inverse：驗證方程 S×B = R + h×A 只需加法和乘法，不需要在電路中計算模逆元（ECDSA 需要）。模逆元在算術電路中極昂貴，因為需要用擴展歐幾里德算法或 Fermat 小定理（約 254 次模乘）。
(2) 確定性 nonce：EdDSA 的 nonce r = Hash(sk, M) 是確定性的，不依賴隨機數產生器。這消除了 nonce 重用攻擊的風險 — Sony PS3 破解（2010 年）和多起比特幣盜竊事件都源於 ECDSA 的 nonce 重用。
為什麼選 EdDSA 而不是更常見的 ECDSA？有兩個核心原因。

第一是電路成本。ECDSA 的驗證方程需要計算 s 的模逆元 — 也就是在 BabyJubjub 的 subgroup order 下求 s−1。這在 CPU 上很快，但在 ZK 電路裡，你需要用擴展歐幾里德算法或 Fermat 小定理來實現，大約需要 1,000 多列。EdDSA 的方程 S·B = R + h·A 完全沒有除法，只有乘法和加法，直接省了這些成本。

第二是安全性。ECDSA 需要一個隨機的 nonce k，如果兩次簽名使用了相同的 k，私鑰就會洩漏 — 這不是理論上的風險，Sony PS3 的 ECDSA 私鑰就是這樣被破解的。EdDSA 的 nonce 是確定性計算的：r = Hash(sk, M)，同樣的訊息永遠產生同樣的簽名，不需要隨機數產生器，從根本上消除了這類攻擊。

至於 BLS 簽名 — 它支持聚合簽名，理論上更優雅，但驗證需要 pairing 運算。pairing 在電路裡面的成本是天文數字，超過 10 萬列，完全不可行。

4.2 Hash 函數：為什麼選 Poseidon

EdDSA 需要一個 hash 函數來計算 challenge h 和 commitment M。標準 EdDSA 使用 SHA-512 — 這在 ZK 電路中極度昂貴。本協議使用 Poseidon hash 替代：

Hash 函數設計目標電路成本不選的原因
Poseidon 為算術電路設計（S-box = x5） ~66 rows/perm —（選用）
SHA-256 為 CPU 設計（bit rotation, XOR, AND） ~25,000 rows bit 操作在算術電路中需要拆解為布林約束，成本極高
MiMC 為 R1CS 設計（S-box = x3） ~15 rows/perm R1CS 下很省，但 PLONKish 支持 degree-5 gate，Poseidon 的 x5 S-box 可以用更少 round
Rescue 為算術電路設計 ~12 rows/perm 需要 S-box 和 inverse S-box 交替，密碼分析文獻較少
Pedersen Hash 基於橢圓曲線 DLP ~750 rows/hash 每次 hash 需要一次 scalar mul，而且安全性依賴額外的 DLP 假設（與 EdDSA 曲線安全性疊加）
Hash 函數的選擇同樣受電路成本驅動。

SHA-256 每次 hash 需要約 25,000 列 — 因為它的內部操作（bit rotation、XOR、32-bit modular addition）在有限域的算術電路中需要拆解成大量的布林約束。一個 32-bit XOR 就需要把兩個操作數拆成 32 個 bit、逐 bit 做 XOR、再組合回來 — 每個 XOR 需要 1 列。

Poseidon 完全不同。它的核心操作是 x5（在有限域裡就是一次乘法鏈），加上矩陣向量乘法（也是有限域原生操作）。一次完整的 Poseidon permutation 在我們的 t=3 配置下只需要 66 列。這是近 380 倍的差距。

MiMC 也是 ZK-friendly 的，但它用 x3 S-box — 在 R1CS 系統下很好，但我們用的 halo2 支持 degree-5 的自訂 gate，所以 Poseidon 的 x5 可以用更少的 round 達到同等安全性。

4.3 簽名方程與電路步驟

簽名生成（off-circuit）：
r = Poseidon(sk, M) mod l   （確定性 nonce — 消除隨機數重用風險）
R = r × B
h = Poseidon(R.x, R.y, A.x, A.y, M, 0)   （sponge 3 perms，吸收 6 個域元素）
S = (r + h × sk) mod l

簽名 = (R, S)   公鑰 A = sk × B
簽名驗證（in-circuit）：
h = Poseidon(R.x, R.y, A.x, A.y, M, 0)   （3 次 Poseidon permutation）
S × B   ?=   R + h × A

電路步驟：
  1. Poseidon sponge 算 h（3 perms × 66 rows = 198 rows）
  2. Decompose S into 254 bits（running sum, 255 rows）
  3. Compute S × B（scalar mul, 763 rows）
  4. Decompose h into 254 bits + constrain 回 sponge output（255 rows）
  5. Compute h × A（scalar mul, 763 rows）
  6. Compute R + h × A（point add, 1 row）
  7. Constrain S × B == R + h × A（2 copy constraints）
EdDSA-Poseidon 的驗證方程式是 S × B = R + h × A。

在電路內部，我們先用 Poseidon sponge 計算 challenge hash h，然後做兩次 scalar multiplication — 一次算 S 乘以 generator B，一次算 h 乘以公鑰 A。最後把 R 加上 h × A，檢查是否等於 S × B。

整個過程最花約束的就是兩次 scalar multiplication，各需要 763 列。加上 bit decomposition 和 Poseidon sponge，每筆簽名驗證總共大約 2,238 列。

有一個技術細節值得提：我們在做 bit decomposition 的時候，使用了 running sum 技巧 — 從 MSB 開始累加，每一步乘以 2 再加上當前的 bit，最後結果必須等於原始的 scalar 值。這保證了 bit 分解的正確性。

4.4 被簽名的訊息 = Trade Commitment

被簽名的訊息 M 是交易參數的 Poseidon sponge hash（4 次 permutation）：

M = PoseidonSponge(
  (op_type, leaf_index),
  (w, K),
  (expiry_date, asset_type),
  (underlying_id, exchange_nonce)
)

共 8 個域元素，分 4 對吸收。Commitment 綁定了完整的交易語義。
為什麼使用 Sponge 而非直接 Hash？Poseidon sponge（rate=2, capacity=1）提供了比直接拼接更強的安全性。直接把 8 個域元素拼成一個向量再 hash，需要 t ≥ 9 的 Poseidon 配置（更多 column、更大的 MDS 矩陣）。Sponge 模式用 t=3 的小配置分 4 輪吸收，capacity 位在整個過程中累積內部狀態，防止長度擴展攻擊和碰撞。
為什麼綁定所有 8 個欄位（含 nonce）？
• 完整綁定：Commitment 包含 op_type 和 leaf_index — 即使兩筆交易的新葉子值相同，如果它們的操作類型或目標位置不同，commitment 也不同。這防止「替換攻擊」（把 TRADE 操作偽裝成 NOP）。
• exchange_nonce 防重放：每筆交易附帶交易所產生的唯一 nonce。即使基金重新提交一筆已簽名的舊交易，nonce 不同會導致 commitment 不同，簽名驗證失敗。這是防止重放攻擊的關鍵機制。

5Per-Exchange 批次認證

電路不僅驗證每筆交易的 EdDSA 簽名，還強制同一子批次內的交易必須來自同一家交易所。

5.1 Sub-Batch 結構

4 筆交易固定分成 2 個 sub-batch：

Sub-batch 0：trades[0], trades[1]  ←  Exchange A 簽署
Sub-batch 1：trades[2], trades[3]  ←  Exchange B 簽署

電路約束（copy constraints，零額外列數）：
  pk0.x == pk1.x   pk0.y == pk1.y
  pk2.x == pk3.x   pk2.y == pk3.y

batch_root = Poseidon(Poseidon(C0, C1), Poseidon(C2, C3))
其中 Cj = Commit(txj)，形成二叉 Merkle tree
批次認證是 Proof #2 的第五階段加入的功能。

4 筆交易固定分成兩個 sub-batch — 前兩筆必須用同一把公鑰簽名，後兩筆必須用同一把公鑰簽名。但前後兩組可以用不同的公鑰 — 這就實現了多交易所的支持。

最優雅的地方是：這個約束只需要 4 個 copy constraint，完全不增加電路列數。我們直接複用 EdDSA 驗證時已經 assign 在 advice column 裡的公鑰座標，讓 halo2 的 permutation argument 來強制相等。

batch root 則是把 4 個交易 commitment 組成一棵二叉 Poseidon hash tree。這個值作為公開輸入，讓任何人都可以向交易所確認「你確實授權了這些交易嗎？」。
為什麼用二叉 Merkle Tree 作為 Batch Commitment？替代方案是 sequential hash chain：Cfinal = H(H(H(C0, C1), C2), C3)。兩者的 hash 次數相同（3 次），但二叉樹結構使得未來若需要個別交易的 inclusion proof，只需 log2(4) = 2 個 sibling（而 chain 結構需要最多 3 個）。此外，二叉結構的對稱性天然對應 sub-batch 分組：左子樹 = sub-batch 0，右子樹 = sub-batch 1。

5.2 安全意義

防止基金偽造交易：即使基金控制了自己的系統，它無法偽造交易所的 EdDSA 簽名。batch_root 作為公開輸入，任何人都可以向交易所詢問「你確認了這個 batch 嗎？」形成不可否認的審計軌跡。
Sub-batch 公鑰約束的作用：防止基金混入未經授權的交易。如果 trade[0] 由 exchange A 簽署，trade[1] 就不可能由 exchange B 簽署 — 公鑰不匹配會被電路拒絕。

6.1 為什麼選 halo2 (PSE fork) + KZG

證明系統的選擇決定了電路的表達能力和效能上限：

證明系統電路模型Proof SizeVerify Time不選的原因

Groth16 R1CS 128 B ~5 ms 需要 per-circuit trusted setup（每次改電路都要重新 ceremony）；R1CS 不支持自訂 gate
Plonky2 + FRI PLONKish ~100 KB ~2 ms Proof size 太大不適合鏈上；使用 Goldilocks 域（64-bit），與 BN254 生態不相容

PLONKish 自訂 Gate 的威力：halo2 允許定義 degree-5 的自訂多項式 gate。這意味著 twisted Edwards 加法的兩條約束（degree-5 多項式）可以放在同一列用同一個 gate 檢查 — 在 R1CS 系統中，同樣的運算需要拆成多個 degree-2 約束，列數增加 3-5 倍。

為什麼選 halo2？兩個關鍵理由。

第一，PLONKish 的自訂 gate。我們的 twisted Edwards 加法公式是 degree-5 的多項式 — 在 R1CS 系統（像 Groth16 用的）裡，每個約束只能是 degree-2（a × b = c），所以一個 degree-5 的約束需要拆成多個中間變數和多列。halo2 允許我們直接定義 degree-5 的 gate，一列搞定。

第二，KZG commitment。它給了我們恆定的 proof size（3.2 KB）和毫秒級的驗證時間。Groth16 的 proof 更小（128 bytes），但每次改電路就要重做 trusted setup ceremony — 對一個還在迭代的研究原型來說，這是不可接受的。

至於 Plonky2 — 它用 FRI 作為 polynomial commitment，不需要 trusted setup，但 proof size 大約 100 KB，放到鏈上的 gas 成本太高。而且它用 64-bit 的 Goldilocks 域，和 BN254 生態不相容 — 我們的 BabyJubjub 曲線是定義在 BN254 Fr 上的。

7安全性質

性質保證機制
交易不可偽造 EdDSA 簽名在電路內驗證。偽造簽名需要破解 BabyJubjub 上的離散對數問題。
狀態不可篡改 Root 鏈接：每筆交易的 old root 必須等於前一筆的 new root（copy constraint）。最終 root 必須等於公開輸入。
交易不可遺漏 batch_root 承諾所有 4 筆交易。跳過任何一筆都會導致 batch root 不匹配。
交易不可重排 Root 鏈接強制交易的執行順序。改變順序 = 改變中間 roots = 驗證失敗。
操作語義正確 3-bit op_type 約束矩陣強制每種操作只能修改允許的欄位。保留代碼被多項式約束拒絕。
交易所不可否認 Sub-batch 公鑰約束 + batch_root 公開輸入 = 交易所簽署了什麼是公開可查的。
狀態連續性 state_root_prev 連接到上一期。Genesis 由監管機構簽名錨定。完整歷史可追溯。
安全性質的設計遵循「多層防禦」的思路。

即使攻擊者控制了基金的全部系統，他仍然無法偽造交易所的簽名 — 除非他同時也破解了 BabyJubjub 的離散對數。

即使攻擊者知道了前後兩個 state root，他也無法自由構造中間的交易 — root chain 強制了唯一的交易序列。

最後，3-bit 的操作類型約束確保了語義正確性。比如一個 CLOSE 操作不可能偷偷把 K 也改了 — 電路不允許。

第一，為什麼用 halo2 而不是 Groth16？Groth16 的 proof 更小（128 bytes），驗證更快，但它需要 per-circuit trusted setup — 每次我們修改電路，就得重做一次 ceremony。對一個還在開發中的系統，這是不可接受的。halo2 + KZG 用的是 universal SRS，一次生成就可以被所有電路共用。

第二，為什麼用 EdDSA 而不是 ECDSA？核心原因是 modular inverse。ECDSA 的驗證需要算 s−1 mod l，這在電路裡大約需要 1,000 多列。EdDSA 的方程 S×B = R + h×A 完全沒有除法，少了將近一半的成本。

第三，Proof #1 和 #2 的組合方式。我們沒有用 recursive proof composition（在一個電路裡面驗證另一個 proof），因為那需要在電路內實作一個完整的 verifier — 大約 10 萬列。我們的做法簡單得多：兩個 proof 各自獨立生成，但都把 state_root 作為公開輸入，驗證者只需檢查兩個 state_root 相等就好。
