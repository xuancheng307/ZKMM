2.1 操作類型 (op_type)

代碼名稱操作葉子變化

1CLOSE完全平倉：清除葉子舊部位 → dummy

2.2 Root 鏈接

交易按順序逐筆套用。每筆交易修改一片 Merkle 葉子，產生新的 root：

root0 = state_root_prev
root1 = MerkleUpdate(root0, idx1, old_leaf1 → new_leaf1)
root2 = MerkleUpdate(root1, idx2, old_leaf2 → new_leaf2)
⋮
rootM = state_root

電路約束：rootM == instance[1]（state_root 公開輸入）

3BabyJubjub 曲線

EdDSA 簽名需要橢圓曲線運算。本協議使用 BabyJubjub — 一條定義在 BN254 scalar field 上的 twisted Edwards 曲線。

3.1 為什麼選 BabyJubjub

Native 運算：BabyJubjub 的基域 = BN254 的 scalar field (Fr)。這意味著所有曲線運算都是有限域的原生運算 — 不需要非原生域模擬（non-native field arithmetic），省去大量約束。

3.2 曲線定義

Twisted Edwards 形式：
a × x² + y² = 1 + d × x² × y²

a = −1
d = 12181644023421730124874158521699555681764249180949974110617291017600649128846

Base field: Fr(BN254) = 21888242871839275222246405745257275088548364400416034343698204186575808495617
Subgroup order: 2736030358979909402780800718157159386076813972158567259200215660948447373041

3.3 Point Operations

在 affine twisted Edwards 座標下：

操作公式約束數
Point Add x3 = (x1y2 + y1x2) / (1 + d × x1x2y1y2)
y3 = (y1y2 − a × x1x2) / (1 − d × x1x2y1y2) 		~6
Point Double 特殊化的 add (P + P) ~6
Scalar Mul Double-and-add（253 bits） ~3,000
實作驗證：20 個 unit tests 通過，包含 curve constants 正確性、identity/inverse 性質、交換律/結合律、以及與 arnaucube/babyjubjub-rs 參考實作的交叉驗證。

4EdDSA 簽名驗證

每筆交易都需要交易所的 EdDSA 簽名。電路在內部驗證簽名，確保交易不可偽造。

4.1 EdDSA with Poseidon

標準 EdDSA 使用 SHA-512 作為 hash — 這在 ZK 電路中極度昂貴。本協議使用 Poseidon hash 替代，利用其 ZK-friendly 特性大幅降低約束數。

簽名生成（off-circuit）：
R = r × B（nonce point）
h = Poseidon(R.x, R.y, A.x, A.y, M)（hash of commitment）
S = r + h × sk (mod subgroup_order)

簽名 = (R, S)
公鑰 = A = sk × B
簽名驗證（in-circuit）：
h = Poseidon(R.x, R.y, A.x, A.y, M)
S × B ?= R + h × A

電路計算左邊 S × B 和右邊 R + h × A，比較兩個點的座標。
兩次 scalar multiplication（~3,000 constraints each）+ 一次 point addition。

4.2 Message = Trade Commitment

被簽名的訊息 M 是交易參數的 Poseidon hash：

M = Poseidon(asset_id, qty, price, direction, K, expiry_date, opt_type, exchange_id)

4.3 約束分析

組件約束數備註
Scalar Mul: S × B~3,000253 doublings + conditional adds
Scalar Mul: h × A~3,000同上
Point Add: R + hA~6twisted Edwards 加法
Equality check2比較 x, y 座標
Poseidon hash (h)~3205 個輸入的 sponge
每筆簽名總計~6,300—
實作驗證：EdDSA off-circuit 8 tests 通過（sign/verify、tamper detection、多 key/多 message）。EdDSA circuit 6 tests 通過（ec_add, scalar_mul MockProver 驗證）。

5Per-Exchange 批次認證

交易所不只簽署個別交易 — 每家交易所還對自己的交易批次進行集體認證（batch attestation）。

5.1 機制

交易按交易所分組：
Exchange A: {tx1, tx3, tx5} → sub_rootA
Exchange B: {tx2, tx4} → sub_rootB

batch_root = Poseidon(sub_rootA, sub_rootB, ...)

每家交易所簽署自己的 sub_root，公開發布。
驗證者可獨立向每家交易所確認 batch 內容。

5.2 安全意義

防止基金偽造交易：即使基金控制了自己的系統，它無法偽造交易所的 EdDSA 簽名。batch_root 作為公開輸入，任何人都可以向交易所詢問「你確認了這個 batch 嗎？」形成不可否認的審計軌跡。

6實作進度

Proof #2 的實作分為 7 個階段。以下是目前的進度（108 tests green）：

Phase 0: BabyJubjub 曲線算術 DONE

Curve constants (a, d, B, subgroup order)、point add/double/negate（affine twisted Edwards）。20 unit tests cross-verified against arnaucube/babyjubjub-rs。

Phase 1: Scalar Multiplication 電路 DONE

Halo2 double-and-add scalar mul gadget。MockProver 驗證通過，matches off-circuit scalar mul。

Phase 2: EdDSA Verify 電路 DONE

Off-circuit EdDSA (sign + verify) + in-circuit ec_add 和 scalar_mul。8 + 6 tests 通過，包含 tamper detection（篡改 R、S、pubkey、message 都會被偵測）。

Phase 3: 整合到 transition.rs IN PROGRESS

將 EdDSA verify gadget 接入 transition circuit。Trade struct 增加 signature fields (R, S)。每筆交易驗證 EdDSA 簽名。

Phase 4: 擴展 op_types PLANNED

加入 EXERCISE、EXPIRY、CORPORATE_ACTION 操作。移除或重新定義 PARAM_UPDATE（與靜態葉節點原則矛盾）。

Phase 5: Per-Exchange 批次認證 PLANNED

交易按交易所分組，sub-tree root per exchange，交易所簽署 batch attestation。

Phase 6: 整合測試 + KZG 基準 PLANNED

End-to-end：generate signatures → prove → verify。KZG proof 生成時間和大小基準測量。

7架構決策

決策選擇原因替代方案
簽名曲線 BabyJubjub Base field = BN254 Fr，native arithmetic secp256k1（non-native，10x 成本）
Hash for EdDSA Poseidon ~64 constraints/permutation vs SHA-256 ~25,000 SHA-256, Blake2（ZK-unfriendly）
Scalar Mul 方法 Double-and-add 簡單、可靠、~3,000 constraints Windowed（更快但更複雜）
State 更新模型 增量式 Merkle 逐筆更新，root 鏈接自然形成 Full tree rebuild（不可行）
Proof 組合 獨立驗證 簡單、無遞歸開銷 Recursive composition（複雜）
已知問題 (ISSUE-003)：原型中的 PARAM_UPDATE op_type 允許修改葉子參數，與靜態葉節點設計矛盾。Phase 4 將解決 — 選項：(1) 完全移除；(2) 重新定義為需要管理員簽名的行政修正。

8安全性質

性質保證機制
交易不可偽造 EdDSA 簽名在電路內驗證。偽造簽名需要破解離散對數問題。
狀態不可篡改 Root 鏈接：每筆交易的 old root 必須等於前一筆的 new root。最終 root 必須等於公開輸入。
交易不可遺漏 batch_root 承諾所有交易。跳過任何一筆都會導致 root 不匹配。
交易不可重排 Root 鏈接強制交易的執行順序。改變順序 = 改變中間 roots。
狀態連續性 state_root_prev 連接到上一期。Genesis 由監管機構簽名錨定。完整歷史可追溯。
