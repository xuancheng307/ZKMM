1問題與動機

金融監管機構需要驗證對沖基金的 Delta 風險是否在合規範圍內。傳統做法要求基金向監管披露完整的持倉明細 — 這會洩露基金的交易策略，造成不可逆的競爭劣勢。

核心矛盾：監管機構需要驗證合規，但基金需要保護商業機密。傳統方法中，這兩個需求不可能同時滿足。
零知識解決方案：基金生成一個密碼學證明（proof），使監管機構在不看到任何持倉資訊的情況下，確信基金的 Delta 風險在法定範圍內。證明的存在本身就代表合規 — 無法生成 proof 就是不合規。

本協議保證什麼

  1. Delta 合規 — 投資組合的加權 Delta 總和在監管門檻 τ 以內
  2. 持倉真實性 — 每筆持倉都可追溯到交易所簽名的合法交易
  3. 市場參數正確 — 計算所用的股價、利率、股息率來自公開可驗證的數據源
  4. 隱波動率合理 — 隱含波動率 σ 由結算價格鎖定，基金無法捏造
  5. 完全隱私 — 監管機構不知道基金操作哪些標的、持倉多少、策略為何

2三方角色

🏢 對沖基金 (Fund)

角色：受監管的證明者（Prover）

職責：維護持倉狀態、生成兩個 ZK 證明

隱私：持倉明細、交易策略永不外洩

計算量：5-10 分鐘/次（每日一次）

🏛 監管機構 (Regulator)

角色：驗證者（Verifier）

職責：驗證 proof、執行隨機抽查

所見：僅 Merkle roots + 門檻值

計算量：< 30 毫秒驗證

💱 交易所 (Exchanges)

角色：數據簽名者

職責：簽署每筆交易（EdDSA）

所見：僅自己執行的交易

計算量：微秒級簽名

隱私分離原則：沒有任何一方能看到基金的完整持倉。基金知道自己的全貌，但只提交加密承諾。交易所各自只看到自己執行的部分交易。監管機構只看到 Merkle root（幾個 hash 值）和一個 proof。

3四層信任模型

協議的安全性建立在四層互相支撐的機制上。每一層獨立運作，即使某一層被突破，其他層仍提供保護。

L1

Trust Anchor（信任錨點）

監管機構簽署 Genesis 承諾，定義初始狀態（空持倉）和參與規則。State(0) = empty tree root，由監管機構的 EdDSA 簽名保證。

L2

Cryptographic Integrity（密碼學完整性）

交易所的 EdDSA 簽名保證每筆交易的真實性。Poseidon Merkle tree 保證持倉和市場數據的完整性。偽造任何數據在計算上不可行。

L3

Computational Correctness（計算正確性）

兩個零知識證明（Proof #1 + Proof #2）保證所有計算邏輯的正確性。電路在數學上保證：如果 proof 通過驗證，那麼聲明一定為真。

L4

Deterrence（嚇阻）

監管機構可隨時要求基金開示特定持倉進行抽查。若抽查結果與 proof 承諾的狀態不一致，基金面臨法律後果。這一層是唯一需要信任的部分（人的行為），但有法律約束力。

4資料結構：三棵 Merkle Tree

整個協議建立在三棵 Poseidon Merkle tree 上。每棵樹承諾不同類型的數據，由不同角色維護。

4.1 Position State Tree（持倉狀態樹）

記錄基金在每個結算時點的所有持倉。每片葉子代表一筆部位。

欄位型別意義性質
wfp24 signed持倉數量（正=多，負=空，0=dummy）靜態
Kfp24履約價格（equity 為 0）靜態
expiry_dateinteger到期日（days since epoch，equity 為 0）靜態
asset_typeinteger0 = EU call, 1 = EU put, 2 = equity靜態
underlying_idinteger標的資產識別碼靜態
靜態葉節點原則：每片葉子從建立到移除，所有欄位永不改變。葉節點只存開倉時就確定的合約條款。部分平倉 = Proof #2 移除舊葉子 + 建立新葉子（reduced w）。這保證了 Merkle root 的唯一性和可驗證性。

規格：深度 7，最大 128 片葉子（Nmax），不足以 dummy（w=0）填充，Poseidon hash。

4.2 Market Data Table（市場數據表）

提供市場參數的公開可驗證來源，消除「基金填假市場數據」的信任缺口。

欄位型別意義更新頻率
underlying_idinteger標的資產識別碼固定
strikefp24履約價格（合約查詢 key）固定
expiry_dateinteger到期日（合約查詢 key）固定
asset_typeinteger選擇權類型（call/put 有不同結算價）固定
spotfp24標的股價 S（收盤價）每日
rfp24無風險利率每日
qfp24股息率每日
settle_pricefp24選擇權結算價格每日
查詢 key = (underlying_id, strike, expiry_date, asset_type)。同一 (K, expiry_date) 的 call 和 put 有不同的結算價格，因此 asset_type 是 key 的一部分。

規格：深度約 20（百萬級條目），每日收盤後重新發布，Poseidon hash。電路結構和 VK 不受影響 — 僅 root 值更新。

4.3 Trade Action Tree（交易動作樹）

記錄結算期內所有已簽名的交易動作。由 Proof #2 使用，驅動 Position State Tree 的更新。

每片葉子 = 一筆已簽名交易，包含交易參數 + 交易所 EdDSA 簽名。交易所各自只簽署自己執行的交易 — 沒有任何一家交易所看到基金的完整組合。

三棵樹的關係：Trade Action Tree 驅動 Position State Tree 的更新（由 Proof #2 保證）。Position State Tree 的每筆部位透過 underlying_id + 合約條款連結到 Market Data Table（由 Proof #1 內的 Merkle 驗證保證）。三棵樹的 root 全部作為公開輸入 — 驗證者只看到三個 hash 值，看不到任何內容。

本協議使用兩個獨立的 ZK 證明，各自負責不同的安全保證。兩者獨立生成、獨立驗證，不需要遞歸證明組合。

5.1 完整的零知識陳述

Proof #1：「我知道一組持倉和對應的 Merkle paths，使得：
 (1) 每筆持倉都在 state_root 承諾的 Merkle tree 中；
 (2) 每筆持倉的市場參數 (S, r, q, settle_price) 來自 market_data_root 承諾的公開數據表；
 (3) 每筆 σi 滿足 BS(S, K, T, r, q, σ) ≈ settle_price（σ 由結算價鎖定）；
 (4) Ti 在電路內由 (expiry_datei − today) / 365 計算；
 (5) |Σ wi · Δi| ≤ τ。」
Proof #2：「我知道一批交易 Trades(T)，使得：
 (1) 每筆交易都有合法交易所的 EdDSA 簽名；
 (2) 所有交易的承諾 = 交易所公布的 batch_root；
 (3) 依序套用交易到 State(T−1) 得到 State(T)。」

5.2 Proof 之間的銜接

6每日結算流程

交易所

交易所簽署每筆交易

基金在各交易所執行交易。每筆交易由執行交易所用 EdDSA (BabyJubjub + Poseidon) 簽名。各交易所只看到自己執行的交易。

基金

建構 Trade Action Tree

收集本期所有已簽名交易，組織成 Merkle tree，計算 batch_root。

基金

計算狀態轉移 State(T) = State(T−1) + apply(Trades)

將交易效果套用到上期持倉，得到本期持倉狀態樹。

基金

生成 Proof #2（State Transition）

證明每筆交易都有交易所簽名，且 State(T) = State(T−1) + Trades(T)。

數據源

發布 Market Data Table

數據提供者在收盤後發布當日市場數據（S, r, q, settle_price），建構 Merkle tree 並公布 market_data_root。

基金

計算 σ 並生成 Proof #1（Delta Compliance）

用 Newton-Raphson 反推每筆持倉的 σ，在電路內驗證 BS(σ) ≈ settle_price，計算 Delta，驗證 |Δtotal| ≤ τ。

基金

提交給監管機構

提交：Proof #1 + Proof #2 + 公開輸入（roots, τ, today）。

監管

驗證（毫秒級）

Proof #1 通過 → Delta 合規。Proof #2 通過 → 持倉真實。兩者的 state_root 一致 → 合規確認。

7隱私保證

角色能看到什麼看不到什麼
監管機構 state_root, market_data_root, τ, today, proof 驗證結果 持倉明細、交易歷史、標的資產、持倉數量、Delta 數值
交易所 A 自己執行的交易 其他交易所的交易、基金完整持倉、其他標的
數據提供者 市場公開數據（本就公開） 基金查詢了哪些條目（哪些標的被使用）
外部觀察者 無 一切
零知識的核心價值：合規驗證不需要看到任何敏感資訊。proof 的存在本身就是合規的數學證據。即使監管機構的驗證系統被入侵，攻擊者也只能看到幾個 hash 值 — 無法還原任何持倉資訊。

攻擊向量描述防禦機制
偽造 σ 基金提供讓 Delta 最小的假 σ σ 正向驗證：BS(σ) ≈ settle_price，偏差超過容許值則 proof 失敗
