1ZK Statement（形式化陳述）

Proof #1 proves:
∃ positions {p1, ..., pN} ⊂ State(T), ∃ {σ1, ..., σN} such that:

(a) ∀i: MerkleVerify(pi, pathi, state_root) = true （注：Proof #1 不強制葉索引唯一性 — 由 Proof #2 的 State(T) 完整性保證）
(b) ∀i: MerkleVerify(mkti, mkt_pathi, market_data_root) = true
(c) ∀i: |BS(S, Ki, Ti, r, q, σi) − settle_pricei| ≤ εσ
(d) ∀i: Ti × 365 = expiry_datei − today
(e) |Σi wi · ΔBS(S, Ki, Ti, r, q, σi)| ≤ τ

各位老師好，接下來我要介紹我們協議中的第一個零知識證明 — Proof #1: Delta Compliance。這個證明的目標是：讓基金公司在不揭露任何持倉細節的前提下，向監管機構證明其投資組合的 Delta 風險在合規門檻之內。

為什麼用零知識證明，而不是其他隱私技術？三個替代方案各有致命缺點：(1) 加密 + 可信第三方：需要一個機構能看到全部持倉明文，這正是我們要避免的 — 基金公司不願意讓任何人看到交易策略。(2) 安全多方計算（MPC）：需要多方在線協作，每次驗證都是一輪互動協議，延遲高且基礎設施複雜。(3) 同態加密（FHE）：可以在密文上計算，但 Black-Scholes 涉及超越函數（ln, erf, exp），FHE 的乘法深度限制讓這類非線性計算極為昂貴。ZK proof 的優勢是：prover 在本地一次性計算，生成一個常數大小的 proof（~1 KB），verifier 在毫秒內驗證。非互動、可公開驗證、不洩漏任何私密輸入。

— 停頓，指向 ZK Statement —

為什麼拆成兩個 proof？我們的協議用 Proof #1 做合規計算（Delta 有沒有超標），用 Proof #2 做狀態完整性（State Tree 是正確更新的、每個葉節點唯一）。拆分的理由是關注點分離：Proof #1 處理金融數學（BS 公式、多項式逼近），Proof #2 處理資料結構（tree update、batch transition）。合在一起的話，單一電路會過大、開發難度高、且不同更新頻率的邏輯被綁死 — 例如市場數據每日變動但 tree 結構規則不常改。

Proof #1 要證明五件事。（逐條指）

條件 (a)：每筆持倉確實存在於已承諾的 State Tree 裡 — 透過 Merkle proof 驗證。這裡有一個跨 proof 的設計取捨：Proof #1 不強制葉索引唯一性，也就是理論上同一筆持倉可以被重複提交。但這不是漏洞 — 因為 Proof #2 保證了 State Tree 的完整性和葉索引唯一性，兩個 proof 合在一起就確保了每個持倉恰好被檢查一次。為什麼不在 Proof #1 也檢查唯一性？因為唯一性需要排序或全域比對，這在固定大小的電路中代價很高（O(N²) 的 pairwise check），而 Proof #2 的 tree update 邏輯天然就包含了這個保證。

條件 (b)：市場數據同樣通過 Merkle 驗證，確保使用的是公開的官方數據。

條件 (c)：用隱含波動率 σ 算出的 BS price 要接近官方結算價 — 這是 σ 正向驗證。

條件 (d)：到期時間 T 在電路內計算，不信任 prover。

條件 (e)：所有持倉的加權 Delta 總和的絕對值不超過門檻 τ。

公開輸入只有 4 個 — state root、market data root、門檻 τ、和今天的日期。都是可公開驗證的值。為什麼只有 4 個？設計原則是：公開輸入越少，隱私性越強。所有持倉細節（權重、strike、到期日、σ）都是 private witness，只有可公開驗證的「錨點」才暴露。

Part B（Delta 管線）是本論文的核心貢獻 — 使用 Minimax 多項式逼近 + Horner 求值法，在 ZK 電路內高效計算 Black-Scholes Delta 所需的三個超越函數（ln, erf, exp）。這一步驟僅佔總電路的約 3%，但是整個協議能夠處理選擇權（而非僅限於現貨）的關鍵。

在看電路細節之前，先說明為什麼選擇 Halo2 + KZG 作為證明系統。

我們需要的是：(1) 可程式化的電路 — 因為 Delta 計算涉及複雜的多項式求值管線；(2) 高效的 lookup table — 用於 range reduction 和 byte decomposition；(3) 成熟的工具鏈。Halo2 基於 PLONKish 算術化，支援自訂閘和 lookup table，而 KZG 多項式承諾提供了常數大小的 proof（~1 KB）和快速的驗證。相比之下，Groth16 不支援 lookup；STARKs 的 proof 太大（~100 KB）且驗證成本高。

另一個基礎選擇是定點算術（fixed-point, pp=24）。為什麼不用浮點數？因為浮點數需要分別追蹤指數和尾數，每次運算都要做正規化 — 在 ZK 電路中代價極高。定點算術把所有數乘以 224 ≈ 1,600 萬，用整數運算模擬小數。pp=24 提供約 7 位十進位精度，足夠金融計算的需求，而且電路中只需要標準的乘法和加法。

整體電路分為五個部分。（逐一指） Part A、A' 做 Merkle 驗證。Part A'' 驗證 σ。Part B 是核心貢獻 — Delta 計算。Part C 做合規檢查。每個部分對每一筆持倉獨立執行，最多 128 slots，最後在 Part C 匯總。

Part B 只佔整體電路的 3% — 這正好說明了我們方法的高效性。

3Part A: State Merkle 驗證

驗證每筆持倉確實存在於 state_root 承諾的 Merkle tree 中。

3.1 葉節點 Hash

5 個欄位透過 Poseidon sponge 壓縮為單一 hash：

leaf_hash = PoseidonSponge([w, K, expiry_date, asset_type, underlying_id])

Sponge 模式：rate=2, capacity=1, 每次吸收 2 個欄位，5 個欄位需要 3 次 permutation。

3.2 Merkle 路徑驗證

從葉節點往上走 7 層（depth = 7），每層：

  1. 根據 position_bit 決定 (left, right) 順序
  2. 計算 parent = Poseidon(left, right)
  3. 最終 root 必須等於 instance[0]（state_root）

每條 Merkle 路徑約 528 rows/position（7 層 × ~64 rows/Poseidon + 葉 hash）。

4Part A': Market Data Merkle 驗證

驗證每筆持倉使用的市場參數 (S, r, q, settle_price) 確實來自 market_data_root 承諾的公開數據表。

4.1 Market Data 葉節點

8 個欄位的 Poseidon sponge hash（4 次 permutation）：

mkt_hash = PoseidonSponge([underlying_id, strike, expiry_date, asset_type, spot, r, q, settle_price])

4.2 Cross-Tree 約束

Position leaf 和 Market Data leaf 透過 8 條 copy constraints 連結，確保 Proof #1 使用的市場數據與持倉的合約條款一致：

Position leaf 欄位約束Market Data leaf 欄位
underlying_id=underlying_id
K=strike
expiry_date=expiry_date
asset_type=asset_type
K=delta.strike
mkt.spot= delta.stock (S)
mkt.r= delta.r
mkt.q= delta.q

前 4 條確保合約條款（underlying、strike、expiry、type）一致；後 4 條將市場參數 (S, r, q) 和持倉 K 直接連結到 Delta 管線的輸入 cells。

為什麼 Market Data 用 Merkle 而不用 Lookup Table？Market Data 每日更新 — 如果用 Halo2 fixed column lookup，每次更新都需要重新產生 setup 和 VK。使用 Merkle tree 讓電路結構完全固定，只有 root 值（公開輸入）每日變化。

Part A' 是整個電路的主要成本來源，約佔 55%（~1,386 rows/position），因為 Market Data 有 8 個欄位（需要 4 次 Poseidon permutation）且 tree depth ~20。

Part A 是標準的 Merkle 驗證。每個持倉有 5 個欄位：持倉量 w、履約價 K、到期日、資產類型、和標的資產 ID。

為什麼用 Merkle tree 做狀態承諾，而不是 KZG 多項式承諾或 Pedersen vector commitment？三個原因：第一，Merkle tree 支援高效的成員證明 — 只需 O(log N) 個 hash 就能證明某筆持倉存在於樹中，而不需要揭露其他持倉。第二，Merkle tree 的更新也是 O(log N)，適合每日結算時只更改部分持倉。第三，Merkle tree 的概念簡單，安全性分析直觀 — 只要 hash 函數抗碰撞，承諾就是 binding 的。

為什麼選 Poseidon hash 而不是 SHA-256 或 Pedersen hash？SHA-256 是位元運算（AND、XOR、rotation），在算術電路中非常昂貴 — 一次 SHA-256 需要約 25,000 個約束，而 Poseidon 只需約 64 個。Pedersen hash 雖然也是代數友好的，但它基於橢圓曲線標量乘法，成本介於兩者之間。Poseidon 是專門為 ZK 電路設計的代數 hash 函數，直接在有限域 Fp 上運算，每次 permutation 只需要幾十個乘法閘。這個選擇讓 Merkle 驗證的成本降低了兩個數量級。

為什麼用 sponge 結構？Poseidon sponge（rate=2, capacity=1）可以吸收任意長度的輸入。5 個欄位每次吸收 2 個，需要 3 次 permutation。相比之下，如果用固定寬度的 Poseidon，就需要先 pad 到特定長度，浪費 rows。

— 翻到 Part A' —

Part A' 結構類似，但處理市場數據。市場數據葉有 8 個欄位。

這裡有一個重要的設計決策：為什麼市場數據不用 Halo2 的 lookup table？Lookup table 是 Halo2 的強大功能 — 可以把任意函數值預先存在 fixed column 中，電路直接查表。但問題是：fixed column 是 setup 的一部分。市場數據每天都在變，如果用 lookup，每天都要重新做 trusted setup、重新產生 proving key 和 verification key。這完全不切實際。改用 Merkle tree，電路結構一次 setup 永久有效，每天只有 root 這個公開輸入在變。

Part A 和 A' 之間透過 8 條 copy constraints 連結。（指向表格） 為什麼用 copy constraint 而不是重新 hash 驗證？Copy constraint 是 PLONK 系統的原生機制 — 它強制兩個 cell 的值完全相等，零成本。如果改用重新 hash 來驗證一致性，就需要額外的 Poseidon 電路，白白增加 rows。前 4 條確保合約條款一致，後 4 條把 S、r、q、K 直接綁定到 Delta 管線的輸入 cells。

Part A' 佔了整個電路的 55%，是主要成本來源 — 因為 8 個欄位需要 4 次 Poseidon permutation。

5Type Decomposition 與 Equity MUX

5.1 類型分解閘 (q_type_decomp)

將 asset_type（0/1/2）分解為兩個布林值：

asset_type = is_put + 2 × is_equity

asset_type = 0 (EU Call)→is_put = 0, is_equity = 0
asset_type = 1 (EU Put)→is_put = 1, is_equity = 0
asset_type = 2 (Equity)→is_put = 0, is_equity = 1
閘約束：is_equity × (1 − is_equity) = 0（確保布林）。is_put 的布林性由 Delta 管線隱式保證。

5.2 Equity MUX 閘 (q_equity_mux)

Equity 部位不需要 Black-Scholes — Delta 固定為 1.0：

effective_delta = is_equity × 1.0 + (1 − is_equity) × bs_delta

Equity: Δ = 1.0（持倉量 w 就是 Delta 貢獻）
Option: Δ = BS Delta（經過完整 ln/erf/exp 管線計算）

6Ti 電路內計算

到期時間 Ti 不是由 prover 直接提供的 — 電路在內部計算並驗證。

6.1 Time Verify 閘 (q_time_verify)

Ti × 365 = expiry_datei − today

• expiry_datei 來自 Merkle-verified Position leaf
• today 來自公開輸入 instance[3]
• Ti 是 prover 提供的 witness，電路驗證其正確性
• 選擇權部位（type 0/1）額外檢查 Ti > 0（防止除以零）
為什麼不直接在電路內做除法？ZK 電路中的除法 = 乘法逆元，成本較高。改為 prover 在電路外算好 Ti = (expiry − today) / 365，電路只需驗證 Ti × 365 == expiry − today（一個乘法閘）。

在進入 Delta 計算之前，電路需要做兩件前處理工作。

第一是類型分解。我們的系統支援三種資產：歐式 call、歐式 put、和 equity。asset_type 被分解為 is_put 和 is_equity 兩個旗標。

為什麼用 MUX 閘而不是條件分支？ZK 電路沒有 if-else — 所有路徑都必須執行。MUX 的做法是：兩個值都算出來（bs_delta 和 1.0），然後用 is_equity 做加權選擇：effective_delta = is_equity × 1.0 + (1 − is_equity) × bs_delta。這在電路中只需要 2 個乘法和 1 個加法。對 equity 來說 Delta 永遠是 1（持倉量等於 Delta 貢獻），不需要 BS 計算。

第二是到期時間驗證。為什麼不讓 prover 直接提供 T 就好？因為 T 可以被操縱 — 較短的 T 會改變 Delta 計算結果。所以電路自己驗證：T × 365 = expiry_date − today，兩邊都是 Merkle-verified 或公開輸入的值。為什麼用乘法驗證而不是電路內除法？ZK 電路中的除法需要乘法逆元，成本較高。改為 prover 在電路外算好 T，電路只做一個乘法閘來驗證 — 這是 ZK 中常見的 「prover hint + verifier check」設計模式。

7Part A'': σ 正向驗證

隱含波動率 σ 是由 prover 在電路外用 Newton-Raphson 法從結算價反推出來的。電路的任務是驗證這個 σ 值是否合理 — 使用它計算一次 BS price，檢查是否接近結算價。

7.1 BS Price 計算

d1 = [ln(S/K) + (r − q + σ²/2) × T] / (σ × √T)
d2 = d1 − σ × √T

Call price = S × e−qT × N(d1) − K × e−rT × N(d2)
Put price = K × e−rT × N(−d2) − S × e−qT × N(−d1)

7.2 電路內驗證

q_d2_sub gate

計算 d2 = d1 − σ√T（帶符號減法處理）

q_bs_price gate

驗證 BS price ≈ settle_price。使用加法形式避免除法：

bs + is_put × s_disc_q + term2 = term1 + is_put × k_disc_r
其中 term1 = stock × Δcall / S（= S·e−qT·N(d1) / S），term2 = strike × disc_r × N(d2) / S，
s_disc_q = stock × discount / S，k_disc_r = strike × disc_r / S（皆為 fp24 定點值）

容許偏差：SIGMA_TOL = 0.02 × 224 = 335,544（絕對容差 0.02，非百分比）

σ 驗證的安全意義：如果不驗證 σ，基金可以填入讓 Delta 最小的假 σ（例如極高波動率會讓所有 Delta → 0.5，降低加權效果）。結算價鎖定使 σ 成為隱函數的唯一解— 基金無法操縱。

σ 正向驗證是 Proof #1 的一個關鍵安全機制。

隱含波動率 σ 是 prover 在電路外用 Newton-Raphson 法從結算價反推出來的。但電路不能直接信任 prover 給的 σ — 為什麼？因為如果不驗證，基金可以填入讓 Delta 趨近於零的假 σ，從而規避合規檢查。例如，極高的波動率會讓所有 Delta 趨向 0.5，降低加權效果。

— 停頓 —

為什麼用「正向驗證」而不是在電路內直接求解 σ？因為求解 σ 需要 Newton-Raphson 迭代 — 這涉及迴圈和條件收斂判斷，在 ZK 電路中無法實現（電路是固定結構的，不支援動態迭代）。但反過來，給定 σ 去算一次 BS price 只需要前向計算 — 這正好就是我們 Delta 管線已經能做的事。所以策略是：prover 在電路外做困難的反向求解，電路只負責簡單的正向驗證。

容許偏差 SIGMA_TOL = 0.02。為什麼用絕對容差而不是相對容差？因為相對容差需要除法（|error / price|），而除法在 ZK 電路中成本高。絕對容差只需要一個 range check：|bs_price − settle_price| ≤ 0.02，電路成本更低。0.02 的容差足夠小，不會影響 σ 的唯一性。

BS price 為什麼要改寫成加法形式？（指向 q_bs_price gate） 標準的 BS 公式是 Call = S·e−qT·N(d₁) − K·e−rT·N(d₂)，裡面有減法。在有限域中，減法結果可能 wrap around 成巨大的值，無法直接做 range check。改寫為加法形式：把減號兩邊移到等式的不同側，所有項都是正的，就可以安全比較了。

因為結算價是公開的、不可篡改的，σ 就成了隱函數的唯一解 — 基金無法操縱。

電路使用 MUX：is_put = 0 → Call Δ，is_put = 1 → Call Δ − discount

現在進入本論文的核心貢獻 — Delta 計算管線。

Black-Scholes Delta 公式包含三個超越函數：自然對數 ln、誤差函數 erf、和指數函數 exp。在一般程式語言中，這些函數由 CPU 的浮點運算單元直接支援。但在 ZK 電路中，我們只有加法和乘法 — 沒有浮點數，沒有條件分支，沒有迭代。

— 停頓，讓聽眾理解這個挑戰 —

在 ZK 中逼近超越函數有幾種可能的方案，讓我解釋為什麼我們選擇了 Minimax：

為什麼不用 Taylor 展開？Taylor 展開在展開點附近精度很高，但離展開點越遠誤差增長很快。要在整個輸入區間達到足夠精度，Taylor 需要很高的次數 — 而在 ZK 電路中，每增加一個次數就多一個乘法閘。Minimax 多項式是在整個區間上均勻分配誤差的最優解 — 由 Chebyshev 等振盪定理保證。同樣 degree 8，Minimax 比 Taylor 精度高 100 到 1000 倍。

為什麼不用 lookup table？Halo2 的 lookup table 可以直接查表得到函數值，但問題是精度。要達到 24-bit 定點精度，lookup table 需要 224 ≈ 1,600 萬個 entries — 這會讓 proving key 和 setup 的大小爆炸。多項式逼近只需要 9 個係數。

為什麼用 degree 8？這是精度和成本的平衡點。Degree 8 在我們的輸入區間上可以達到 24-bit 以上的精度（相對誤差 < 2-24），而每個多項式求值只需要 8 個乘法閘。更低的 degree 精度不夠，更高的 degree 成本增加但精度提升已經邊際遞減。

搭配 Horner 求值法。（指向 17 步表格） 為什麼不用直接展開 c₀ + c₁x + c₂x² + ...？因為直接展開每一步乘法都會產生中間結果需要截斷，而 Horner 法把多項式改寫為 c₀ + x(c₁ + x(c₂ + ...))，整個過程只需要在最後做一次截斷。這把截斷的額外 rows 從 O(d) 降到 O(1)。

為什麼需要 range reduction？Minimax 多項式只在特定區間上有保證精度。例如 ln 的多項式是為 [1, 2) 設計的。所以我們先把輸入映射到這個區間（x = 2k × r，r ∈ [1,2)），求值後再還原：ln(x) = ln(r) + k × ln(2)。Range reduction 用 ByteTable lookup 實現，成本很低。

整個管線 17 步，每個持倉只需約 58 rows。相比之下，光是 Merkle 驗證就要將近 2,000 rows。Delta 管線只佔總電路的 3%。

對於 put option，我們利用 put-call parity：Put Delta = Call Delta − e−qT。這是金融數學的恆等式，不是近似 — 所以只需一個減法閘，不必重新計算整個管線。

9Part C: 合規累加與檢查

9.1 帶符號累加

每筆持倉貢獻 wi × Δi，w 有正負（多/空）：

w_sign XOR is_put → 路由到 positive_sum 或 negative_sum

positive_sum = Σ(正向貢獻的 |w| × Δ)
negative_sum = Σ(負向貢獻的 |w| × Δ)

9.2 雙向合規檢查

Delta 合規不只檢查淨值，而是雙向檢查：

τ + negative_sum ≥ positive_sum
τ + positive_sum ≥ negative_sum

等價於：|positive_sum − negative_sum| ≤ τ
使用 range check 而非直接比較，避免有限域中的負數問題。

τ 來自公開輸入 instance[2]（threshold）。

最後一步是合規檢查。電路把所有持倉的 wi 乘以 Deltai 累加起來。因為 w 有正有負 — 代表多倉和空倉 — 我們把正向貢獻和負向貢獻分開累加。

為什麼要分開累加，而不是直接加總？因為 ZK 電路在有限域 Fp 上運算，負數不存在 — 一個「負值」在有限域中其實是一個非常大的正數。如果直接做 a - b，結果可能 wrap around 變成一個巨大的值，range check 就會失敗。所以我們把正項和負項分別累加，最後用雙向 range check 來驗證。

具體做法：τ + negative_sum ≥ positive_sum，同時 τ + positive_sum ≥ negative_sum。兩邊都是非負數相加，不會出現 underflow 問題。兩個都成立，就等價於 |net delta| ≤ τ。

這個技巧是 ZK 電路中處理帶符號比較的標準做法 — 把「比大小」轉化為「驗證一個差值落在合法範圍內」。
