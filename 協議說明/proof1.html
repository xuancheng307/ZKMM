Proof #1: Delta Compliance

投資組合 Delta 風險合規證明 — 技術細節

電路架構、自訂閘、約束管線的完整說明

0金融背景：什麼是 Delta 合規

0.1 什麼是 Delta

Delta（Δ）衡量一個金融部位對標的資產價格變動的敏感度。Δ = 0.6 代表股價漲 1 元，這個部位大約漲 0.6 元。

資產類型Delta 範圍說明
股票（做多）+1.0股價漲 1 元 → 部位漲 1 元
股票（做空）−1.0股價漲 1 元 → 部位虧 1 元
歐式 Call0 ~ +1由 Black-Scholes 公式計算
歐式 Put−1 ~ 0由 put-call parity 推導

0.2 Black-Scholes Delta 公式

Step 1 — 計算 d1：
d1 = [ln(S/K) + (r − q + σ²/2) × T] / (σ × √T)

Step 2 — Call Delta：
Δcall = e−qT × N(d1)

Step 3 — Put Delta（由 put-call parity）：
Δput = Δcall − e−qT

其中 N(x) = 0.5 + 0.5 × erf(x / √2)（常態分佈累積函數）
符號意義來源
S標的資產現價市場數據（Market Data Tree）
K履約價合約條款（Position State Tree）
T距到期的年數電路內計算：(expiry − today) / 365
r無風險利率市場數據
q股息殖利率市場數據
σ隱含波動率由結算價反推（Newton-Raphson，電路外）

0.3 合規檢查公式

|Σi wi × Δi| ≤ τ

wi = 第 i 筆持倉的數量（正=做多，負=做空）
Δi = 第 i 筆持倉的 Delta（股票=1，選擇權=BS Delta）
τ = 合規門檻（由 LPA 或監管規定）
這個公式本身很簡單 — 任何 Excel 都能算。困難的是在零知識電路裡面算，因為公式包含三個超越函數：ln（自然對數）、erf（誤差函數）、exp（指數函數）。ZK 電路只有加法和乘法，沒有這些函數 — 這就是本論文要解決的核心問題。

在進入 Proof #1 的技術細節之前，先用兩分鐘說明我們要證明什麼。

Delta 衡量一個金融部位對標的資產價格變動的敏感度。股票的 Delta 固定是 1，選擇權的 Delta 則由 Black-Scholes 公式動態計算，取決於市價、履約價、波動率、到期時間等參數。

一個投資組合可能同時持有股票、call option、put option。合規檢查要求：所有持倉的加權 Delta 總和的絕對值不超過門檻 τ。公式本身很簡單，但 BS Delta 裡面包含三個超越函數 — ln、erf、exp。這些函數在一般程式語言中有 CPU 的浮點單元直接支援，但零知識電路的運算核心是算術電路，只有有限域上的加法和乘法，沒有浮點數、沒有迴圈、沒有這些函數。

如何在零知識電路中精確計算這三個超越函數，就是本論文在協議層面要解決的核心工程問題。接下來我會從 ZK statement 開始，逐步展開整個電路的設計。

1ZK Statement（形式化陳述）

Proof #1 proves:

了解了金融背景，來看形式化陳述。Proof #1 的目標：讓基金公司在不揭露任何持倉細節的前提下，證明 Delta 風險在合規門檻之內。

為什麼用零知識證明？三個替代方案各有致命缺點：可信第三方需要揭露持倉；MPC 需多方在線協作、延遲高；FHE 的乘法深度限制讓超越函數極為昂貴。ZKP 讓 prover 在本地一次性計算，verifier 毫秒內驗證，不洩漏任何私密輸入。

— 指向 ZK Statement —

為什麼拆成兩個 proof？Proof #1 做合規計算，Proof #2 做狀態完整性。金融數學和資料結構邏輯獨立、更新頻率不同，拆開能獨立升級。

Proof #1 要證五件事。（逐條指） (a)(b) 持倉和市場數據透過 Merkle 驗證存在性。(c) σ 正向驗證 — 用 σ 算出的 BS price 接近結算價。(d) 到期時間 T 電路內計算。(e) 加權 Delta 總和的絕對值不超過 τ。

公開輸入只有 4 個 — state_root、market_data_root、threshold、today。公開輸入越少，隱私性越強。所有持倉細節都是 private witness。

在看各部分細節之前，先說明兩個基礎設計選擇。

為什麼選 Halo2 + KZG？Delta 計算需要可程式化的自訂閘、高效的 lookup table（用於 range reduction 和 byte decomposition）、以及成熟的工具鏈。Halo2 的 PLONKish 架構支援這些需求，KZG 提供常數大小的 proof（~1 KB）和毫秒級驗證。Groth16 不支援 lookup，STARKs 的 proof 太大（~100 KB）。

第二個選擇是定點算術 pp=48。浮點數在 ZK 電路中每次運算都要做指數對齊和正規化，代價極高。定點算術把所有數乘以 S = 248 ≈ 281 兆，用整數運算模擬小數。pp=48 提供約 14.4 位十進位精度，端對端 BS Delta 絕對誤差達 ~2×10−14，遠超金融需求。

— 指向管線圖 —

電路分為五個部分。（逐一指） Part A 和 A' 做 Merkle 驗證 — 確認持倉和市場數據的存在性。Part A'' 驗證隱含波動率 σ。Part B 是核心 — Delta 計算管線。Part C 做最終合規判定。每個部分對每筆持倉獨立執行，最多 128 slots，最後在 Part C 匯總。各部分的成本佔比等看完技術細節後再一起分析。

3Part A: State Merkle 驗證

驗證每筆持倉確實存在於 state_root 承諾的 Merkle tree 中。

3.1 葉節點 Hash

5 個欄位透過 Poseidon sponge 壓縮為單一 hash：

leaf_hash = PoseidonSponge([w, K, expiry_date, asset_type, underlying_id])

Sponge 模式：rate=2, capacity=1, 每次吸收 2 個欄位，5 個欄位需要 3 次 permutation。

3.2 Merkle 路徑驗證

從葉節點往上走 7 層（depth = 7），每層：

  1. 根據 position_bit 決定 (left, right) 順序
  2. 計算 parent = Poseidon(left, right)
  3. 最終 root 必須等於 instance[0]（state_root）

每條 Merkle 路徑約 528 rows/position（7 層 × ~64 rows/Poseidon + 葉 hash）。

4Part A': Market Data Merkle 驗證

驗證每筆持倉使用的市場參數 (S, r, q, settle_price) 確實來自 market_data_root 承諾的公開數據表。

4.1 Market Data 葉節點

8 個欄位的 Poseidon sponge hash（4 次 permutation）：

mkt_hash = PoseidonSponge([underlying_id, strike, expiry_date, asset_type, spot, r, q, settle_price])

4.2 Cross-Tree 約束

Position leaf 和 Market Data leaf 透過 8 條 copy constraints 連結，確保 Proof #1 使用的市場數據與持倉的合約條款一致：

Position leaf 欄位約束Market Data leaf 欄位
underlying_id=underlying_id
K=strike
expiry_date=expiry_date
asset_type=asset_type
K=delta.strike
mkt.spot= delta.stock (S)
mkt.r= delta.r
mkt.q= delta.q

前 4 條確保合約條款（underlying、strike、expiry、type）一致；後 4 條將市場參數 (S, r, q) 和持倉 K 直接連結到 Delta 管線的輸入 cells。

為什麼 Market Data 用 Merkle 而不用 Lookup Table？Market Data 每日更新 — 如果用 Halo2 fixed column lookup，每次更新都需要重新產生 setup 和 VK。使用 Merkle tree 讓電路結構完全固定，只有 root 值（公開輸入）每日變化。

Part A' 是整個電路的主要成本來源，約佔 61%（~1,386 rows/position），因為 Market Data 有 8 個欄位（需要 4 次 Poseidon permutation）且 tree depth ~20。

Part A 是標準的 Merkle 驗證。每個持倉有 5 個欄位，透過 Poseidon sponge 壓縮為葉 hash，再沿 7 層路徑驗證到 state_root。

為什麼選 Poseidon hash？SHA-256 是位元運算，一次需要 ~25,000 個約束；Poseidon 是專為 ZK 設計的代數 hash，直接在有限域上運算，一次 permutation 只需 ~64 個約束 — 差兩個數量級。這個選擇直接決定了整個系統是否可行。

— 翻到 Part A' —

Part A' 結構類似但更重，處理市場數據（8 個欄位，4 次 Poseidon permutation，tree depth ~20）— 它會是整個電路中最大的部分。

一個關鍵設計取捨：為什麼市場數據不用 Halo2 lookup table？因為 fixed column 是 setup 的一部分 — 市場數據每天變動，如果用 lookup 就要每天重新 setup。改用 Merkle tree，電路結構一次 setup 永久有效，只有 root 值每日更新。

兩棵樹之間透過 8 條 copy constraints 連結 — 這是防止資產錯配的核心機制。（指向表格） 前 4 條強制 Position leaf 和 Market Data leaf 的合約條款完全一致：underlying_id、strike、expiry_date、asset_type — 這四個欄位合起來就是合約的複合唯一鍵。後 4 條把市場參數（S, K, r, q）直接綁定到 Delta 管線的輸入 cells。

舉個具體例子：prover 持有 AAPL 的選擇權，能不能拿 NVDA 的市場數據來騙？不能。Position leaf 的 underlying_id = AAPL 已經被 Poseidon hash 鎖進 state_root — 改了就重建不出 root。Market Data leaf 的 underlying_id = NVDA 同樣被 hash 鎖進 market_data_root。constrain_equal 強制兩邊的 underlying_id 相等 — AAPL ≠ NVDA，約束失敗，proof 根本產生不出來。同樣的道理，prover 也不能用同一標的的不同合約數據混用 — strike、expiry、type 三個欄位也同時被鎖死。

而且這全部用 PLONK 原生的 cell equality 機制實作，零額外約束成本 — copy constraint 不消耗 gate，不佔用 row。

5Type Decomposition 與 Equity MUX

5.1 類型分解閘 (q_type_decomp)

將 asset_type（0/1/2）分解為兩個布林值：

asset_type = is_put + 2 × is_equity

asset_type = 0 (EU Call)→is_put = 0, is_equity = 0
asset_type = 1 (EU Put)→is_put = 1, is_equity = 0
asset_type = 2 (Equity)→is_put = 0, is_equity = 1
閘約束：is_equity × (1 − is_equity) = 0（確保布林）。is_put 的布林性由 Delta 管線隱式保證。

5.2 Equity MUX 閘 (q_equity_mux)

Equity 部位不需要 Black-Scholes — Delta 固定為 1.0：

effective_delta = is_equity × 1.0 + (1 − is_equity) × bs_delta

Equity: Δ = 1.0（持倉量 w 就是 Delta 貢獻）
Option: Δ = BS Delta（經過完整 ln/erf/exp 管線計算）

6Ti 電路內計算

到期時間 Ti 不是由 prover 直接提供的 — 電路在內部計算並驗證。

6.1 Time Verify 閘 (q_time_verify)

Ti × 365 = expiry_datei − today

• expiry_datei 來自 Merkle-verified Position leaf
• today 來自公開輸入 instance[3]
• Ti 是 prover 提供的 witness，電路驗證其正確性
• 選擇權部位（type 0/1）額外檢查 Ti > 0（防止除以零）
為什麼不直接在電路內做除法？ZK 電路中的除法 = 乘法逆元，成本較高。改為 prover 在電路外算好 Ti = (expiry − today) / 365，電路只需驗證 Ti × 365 == expiry − today（一個乘法閘）。

進入 Delta 計算之前，電路做兩件前處理。

第一是類型分解。三種資產（call、put、equity）被分解為 is_put 和 is_equity 兩個旗標。ZK 電路沒有 if-else，所以用 MUX 閘：兩個值都算出來，然後加權選擇。Equity 的 Delta 固定是 1，不需要 BS 計算。

第二是到期時間驗證。T 不信任 prover — 因為操縱 T 會改變 Delta。電路自己驗證：T × 365 = expiry_date − today。這是 ZK 中常見的「prover hint + verifier check」模式：prover 在電路外算好 T，電路只做一個乘法閘驗證正確性，避免昂貴的電路內除法。

7Part A'': σ 正向驗證

隱含波動率 σ 是由 prover 在電路外用 Newton-Raphson 法從結算價反推出來的。電路的任務是驗證這個 σ 值是否合理 — 使用它計算一次 BS price，檢查是否接近結算價。

7.1 BS Price 計算

d1 = [ln(S/K) + (r − q + σ²/2) × T] / (σ × √T)
d2 = d1 − σ × √T

Call price = S × e−qT × N(d1) − K × e−rT × N(d2)
Put price = K × e−rT × N(−d2) − S × e−qT × N(−d1)

7.2 電路內驗證

q_d2_sub gate

計算 d2 = d1 − σ√T（帶符號減法處理）

q_bs_price gate

驗證 BS price ≈ settle_price。使用加法形式避免除法：

bs + is_put × s_disc_q + term2 = term1 + is_put × k_disc_r
其中 term1 = stock × Δcall / S（= S·e−qT·N(d1) / S），term2 = strike × disc_r × N(d2) / S，
s_disc_q = stock × discount / S，k_disc_r = strike × disc_r / S（皆為 fp48 定點值）

容許偏差：SIGMA_TOL = 0.02 × 248 ≈ 5.6 兆（絕對容差 0.02，非百分比）

σ 驗證的安全意義：如果不驗證 σ，基金可以填入讓 Delta 最小的假 σ（例如極高波動率會讓所有 Delta → 0.5，降低加權效果）。結算價鎖定使 σ 成為隱函數的唯一解— 基金無法操縱。

σ 正向驗證是關鍵安全機制。如果不驗證 σ，基金可以填入讓 Delta 趨近於零的假 σ 來規避合規。

為什麼用「正向驗證」而不是電路內求解 σ？求解需要 Newton-Raphson 迭代 — ZK 電路不支援動態迭代。但給定 σ 去算一次 BS price 只需前向計算，正好就是 Delta 管線已經能做的事。這又是「prover hint + verifier check」模式的應用。

容許偏差 SIGMA_TOL = 0.02（pp=48 下 = 0.02 × 248 ≈ 5.6 兆）。為什麼用絕對容差？相對容差需要除法；絕對容差只需一個 range check。

BS price 公式改寫成加法形式：把減號兩邊移到等式不同側，所有項為正，避免有限域 wrap-around。結算價是公開的、不可篡改的，所以 σ 成了隱函數的唯一解。

8.3 Put Delta 調整

Put Δ = Call Δ − e−qT（由 put-call parity 推導）

電路使用 MUX：is_put = 0 → Call Δ，is_put = 1 → Call Δ − discount

現在進入核心貢獻 — Delta 計算管線。BS Delta 包含三個超越函數，我們對每個使用不同的最優策略。

— 停頓 —

ln 和 erf：Minimax 分段多項式 + Per-step Horner。

為什麼選 Minimax 而不是 Taylor？Taylor 在展開點附近好，但離開展開點誤差快速增長。Minimax 在整個區間上均勻分配誤差，由等振盪定理保證最優。同樣次數下，精度高數億至數百億倍（pp=48 d=14：erf 2.6×108×、ln 6.8×1010×）。

為什麼用分段逼近？單一多項式在寬區間上會遇到「量化飽和」— 高次係數的定點量化誤差被放大，增加次數反而惡化精度。切成窄段（ln 4 段 × 0.25，erf 12 段 × 0.5），每段只需 d=12 就能達到 ~1.2 ULP 的 kernel 精度。d=12 是經過 degree sweep 實驗找到的最優點 — 逼近誤差和截斷累積的 U 曲線最低點。

Per-step Horner 是求值法的關鍵。（指向管線表格） 把多項式改寫為巢狀形式 c₀ + x(c₁ + x(c₂ + ...))，每一步都做截斷（除以 S = 248），讓中間值始終保持在 S² 以內。截斷驗證用 ByteTable lookup — 把值拆成 6 個 byte，每個 byte 查一次 256-entry 的共用表。

exp：Method B 查表乘法。exp 利用 ea+b = ea × eb 的分解性質。把 48-bit 輸入拆成 6 個 byte，每個位置預先算好 256-entry 查表，然後 5 次乘法-截斷相乘。6 張表共 1,536 entries，只需 6 行電路 — 比 d=12 Horner 的 12 行更緊湊，精度 ~8 ULP。

Range reduction：每個函數先做定義域映射。ln 把 S/K 映射到 [1, 2)，還原時加回 k × ln(2)。erf 用 segment index 選段。都透過 ByteTable lookup 實現，成本極低。

整個管線 17 個邏輯步驟，端對端 BS Delta 精度達 ~6 ULP（~2×10−14）。Put option 用 put-call parity 只需一個減法：Put Δ = Call Δ − e−qT。

9Part C: 合規累加與檢查

9.1 帶符號累加

每筆持倉貢獻 wi × Δi，w 有正負（多/空）：

w_sign XOR is_put → 路由到 positive_sum 或 negative_sum

positive_sum = Σ(正向貢獻的 |w| × Δ)
negative_sum = Σ(負向貢獻的 |w| × Δ)

9.2 雙向合規檢查

Delta 合規不只檢查淨值，而是雙向檢查：

τ + negative_sum ≥ positive_sum
τ + positive_sum ≥ negative_sum

等價於：|positive_sum − negative_sum| ≤ τ
使用 range check 而非直接比較，避免有限域中的負數問題。

τ 來自公開輸入 instance[2]（threshold）。

最後一步是合規檢查。w 有正有負（多倉和空倉），但有限域沒有負數 — 直接做 a − b 可能 wrap around。所以正項和負項分別累加，最後用雙向 range check：τ + negative_sum ≥ positive_sum，同時 τ + positive_sum ≥ negative_sum。兩邊都是非負數相加，不會 underflow。兩個同時成立就等價於 |net delta| ≤ τ。

這是 ZK 中處理帶符號比較的標準做法 — 把「比大小」轉化為「驗證一個差值落在合法範圍內」。pp=48 下 range check 使用 7 個 byte（56-bit）的分解驗證，足以涵蓋所有可能的累加值。

Proof #1 電路（PortfolioCircuit）共註冊 31 個自訂閘（create_gate），分布在 6 個 Rust 模組中。在 Halo2 的 PLONKish 架構中，每個閘定義一組多項式約束 — 閘啟用時（selector = 1），該 row 上的所有約束方程必須同時為零。一個邏輯操作（如 Poseidon permutation）需要跨多行重複啟用閘。

Part A、A' 的 Merkle 路徑驗證核心。t=3 狀態向量，S-box 為 x5。

3 個元素全部施加 S-box + MDS 矩陣乘法。9 constraints/row（3 × sq 驗證 + 3 × p=sq²·a 驗證 + 3 × MDS state update）。

只對第一個元素施加 S-box，其餘為線性層。5 constraints/row。每次 permutation = 8 full + 57 partial = 65 rows。

Poseidon sponge 吸收：rate=2，將 2 個欄位加到 state 上。3 constraints/row（state[0..2] 更新）。

本論文核心貢獻。用於 ln（4-segment piecewise）和 erf（12-segment piecewise）的 Minimax 多項式求值。

horner_per_step

巢狀求值每一步：accnext × S + rem = acc × r + coeff，同時做截斷的 6-byte 分解驗證。2 constraints/row。d=12 多項式需要 12 次啟用 = 12 rows。

Range reduction 輸入和 Horner 輸出的 6-byte 分解驗證。b0 + 256·b1 + … + 2565·b5 = value。每個 byte 經 ByteTable lookup 確保 ∈ [0, 255]。1 constraint/row each。

這裡要區分兩個層次。Gate 是硬體層：1 gate = 1 row，selector 啟用時該行的所有約束必須同時為零 — 例如 fp_mul 在一行上同時施加乘法正確性和餘數 byte 分解兩條約束。Gadget 是邏輯層：1 操作 = 多 rows — Poseidon permutation 需要 65 行（8 full + 57 partial rounds），Horner d=12 需要 12 行，Method B exp 需要 11 行（5 次 fp_mul + 6 lookup）。

— 停頓 —

再看按管線階段分類的四層結構：（逐層指出）

第一階段 — Merkle 驗證（4 gates、20 constraints）：poseidon_full_round 一個 gate 就有 9 條約束（3 元素 × S-box + MDS），是約束密度最高的 gate。poseidon_partial_round 5 條，sponge_absorb 和 merkle_swap 各 3 條。這 4 個 gate 佔全部約束的 30%。

第二階段 — 前處理（2 gates、6 constraints）：type_decomp 做 asset_type 布林分解，time_verify 單閘 4 條約束（is_equity 布林 + t_days 計算 + T 換算 + rem 分解）是 portfolio.rs 中密度最高的。

第三階段 — Delta 計算（19 gates、29 constraints）：這是主體，佔 gate 數的 61%。kernel.rs 3 個 gate 做 Horner 求值（horner_per_step 2 constraints + r/w_byte_decomp 各 1）。gadgets/ 3 個 gate 做 range reduction（lead_bit 3、int_div 2、sign_extend 2）。delta.rs 13 個 gate 進一步分為四小類：定點算術 5 個（fp_mul、fp_div、arith_add、arith_sub、arith_div2）、超越函數還原 3 個（exp_recon、ln_recon、erf_flip）、控制流 3 個（bool_check、mux、sqrt_verify）、帶符號處理 2 個（signed_num、phi_signed）。

第四階段 — 合規判定（6 gates、11 constraints）：equity_mux 做 equity bypass，d2_sub 算 d2，bs_price 驗 σ，route_accum 3 條約束做帶符號累加，compliance_range 和 wide_range 各 2 條約束做 range check。

— 停頓 —

Lookup 表使用也值得一提。ByteTable 是整個電路最核心的安全機制 — 幾乎每個涉及 range check 的 gate 都用它：fp_mul 的 6-byte rem、fp_div 的 8-byte gap、Horner 的 6-byte 截斷、wide_range 的 7-byte 分解。byte decomposition + ByteTable lookup 是鎖死 prover 的萬能工具。另一個是 Pow2Table，只用在 exp_recon 查 2k。

回顧整個設計：核心挑戰是在只有加法和乘法的算術電路中精確計算超越函數。結果是端對端精度 ~2×10−14（57.6 ULP 理論上界，實測最差 6.04 ULP），Delta 管線僅佔電路的 13%。31 個 gate 中 61% 服務於 Delta 計算，但大部分行數被 Poseidon 佔據（每次 Merkle 路徑驗證 65 行 × 路徑深度）。

接下來看 Proof #2 如何保證 State Tree 的完整性。

12ZK Proof of Knowledge 論證

前述 Sections 2–11 描述了電路「怎麼蓋的」。本節證明這個電路構成一個 zero-knowledge proof of knowledge (ZK-POK)：prover 確實知道一組合規的持倉，且 verifier 從證明中學不到任何額外資訊。

12.0 假設與安全模型

Assumption 1 (State Completeness). State Tree 包含投組的全部 N 筆持倉，且每筆恰好出現一次。
此假設由 Proof #2 保證（State(T) 完整性 + 葉索引唯一性）。Proof #1 不重複驗證。

Assumption 2 (Collision-Resistant Hash). Poseidon hash 是 collision-resistant：找到兩個不同的輸入映射到相同 hash 在計算上不可行。

Assumption 3 (PLONK Soundness). 底層 PLONK 協議（KZG commitment + random oracle）在代數群模型（AGM）下是 knowledge-sound 的：
若 prover 以不可忽略的機率使 verifier accept，則存在一個多項式時間的 extractor 可從 prover 的 commitment 中萃取滿足所有電路約束的 witness。
參考：Gabizon, Williamson, Ciobotaru, "PLONK: Permutations over Lagrange-bases for Oecumenical Noninteractive arguments of Knowledge", 2019.

12.1 Relation 定義

定義 Proof #1 的 relation R1：

Public input x = (state_root, market_data_root, τ, today)

Witness w = ({pi, pathi, mkti, mkt_pathi, σi, wi}i=1..N)

(x, w) ∈ R1 ⇔ 條件 (a)–(e) 全部成立（Section 1）

12.2 Theorem

Theorem (Proof #1 is a ZK-POK for R1).
在 Assumptions 1–3 下，Proof #1 的 PLONK 電路 C1 構成 relation R1 的 zero-knowledge proof of knowledge，滿足：

(I) Completeness：若 (x, w) ∈ R1，則 Prove(pk, x, w) 產生的 π 必被 Verify(vk, x, π) 接受。
(II) Knowledge Soundness：若 PPT prover P* 以機率 ε 使 verifier accept，則存在 extractor E 能萃取 w 使得 (x, w) ∈ R1，成功機率 ≥ ε − negl(λ)。
(III) Zero-Knowledge：存在 PPT simulator S，使得 {π : π ← Prove(pk, x, w)} 與 {π : π ← S(vk, x)} 在計算上不可區分。

12.3 Completeness 證明

目標：合法 witness 代入電路，所有 31 gates 的約束方程值為零。

設 (x, w) ∈ R1（即條件 (a)–(e) 成立）。電路 synthesize 過程按以下順序執行：

模組對應條件完備性論證
Poseidon + Merkle
(4 gates)		 (a), (b) 由 (a)(b) 成立，Merkle path 合法。poseidon_full/partial_round 是 Poseidon permutation 的正確實作（round constants + MDS 矩陣），sponge_absorb 正確吸收，merkle_swap 按 position_bit 正確交換。合法路徑代入必然重建出 state_root / market_data_root。

因此所有 31 gates 的約束在合法 witness 下均為零，PLONK 證明驗證必通過。□

12.4 Soundness 證明（核心）

目標：若所有約束被滿足，則 witness 必然滿足 relation R1 的全部五個條件。

由 Assumption 3（PLONK soundness），verifier accept ⇒ 存在 extractor 萃取出滿足所有電路約束的 assignment。以下逐條從約束推導出 R1 的條件。

12.4.1 條件 (a)(b): Merkle 成員性

Claim. merkle_swap + poseidon gates 滿足 ⇒ pi ∈ State(state_root)。

Proof sketch. merkle_swap 在每層強制 (left, right) = swap(current, sibling, bit)，接著 poseidon gates 計算 hash(left, right)。逐層從 leaf hash 重建到 root。約束強制最終 root = public input state_root（via constrain_instance）。

若 pi ∉ State(state_root)，則 prover 必須找到一條偽造路徑使 Poseidon 重建到同一 root — 這等價於找到 Poseidon 的 collision 或 second preimage，在 Assumption 2 下不可行。

同理適用於 market_data_root（條件 (b)）。□

12.4.2 條件 (d): 時間正確性

Claim. time_verify gate 滿足 ⇒ Ti × 365 = expiry_datei − today。

Proof. time_verify 施加四條約束：(1) is_equity 布林、(2) t_days = expiry − today、(3) T × 365 + rem = t_days × S、(4) rem 的 2-byte decomposition（確保 0 ≤ rem < 216）。
today 來自 public input（constrain_instance），prover 無法篡改。expiry_date 來自 Merkle 驗證過的葉節點。2-byte 允許 rem ≤ 65535，理論上 prover 可偏移 T 至多 ⌊65535/365⌋ = 179 個定點單位。在 pp=48 下，179/S = 6.4×10−13 年（< 0.02 毫秒），對 Delta 的影響遠小於 1 ULP，可忽略。因此 T 被確定至精度 179/S 以內。□

12.4.3 條件 (c): σ 正向驗證

Claim. bs_price gate 滿足 ⇒ |BS(S, K, T, r, q, σ) − settle_price| ≤ εσ。

Proof sketch. d2_sub gate 約束 d2 與 d1 的關係（d2 = d1 − σ√T），bs_price gate 將 BS price 的加法形式與 settle_price 做 range check（容差 SIGMA_TOL = 0.02 × S）。

BS price 的所有子計算（N(d1)、N(d2)、discount）都透過 Delta pipeline 的約束閘（fp_mul、kernel、exp_recon 等）強制正確。settle_price 來自 Merkle 驗證過的 market data 葉節點，prover 無法篡改。

因此 σ 被鎖定為使 BS(params) ≈ settle_price 的值 — 即市場隱含波動率。prover 無法偽造使 Delta 趨近零的假 σ。

安全邊界：SIGMA_TOL = 0.02（絕對值，非百分比）足夠窄。BS price 對 σ 的 Vega 敏感度通常 > 0.1，因此 σ 被限制在真值 ± 0.2 以內，對 Delta 影響 < 0.02。□

12.4.4 條件 (e): Delta 合規

這是 soundness 的核心：prover 能否偽造一個假的 Delta 值繞過合規檢查？

Claim. 若所有 kernel + delta + portfolio gates 滿足，則 |Σ wi · Δi| ≤ τ，且每個 Δi 是 BS Delta 的定點近似（誤差 ≤ 57.6 ULP）。

Proof sketch. 分三層論證。

Layer 1 — 算術閘不可繞。

fp_mul 約束 a × b = w × S + rem，其中 rem 經 byte decomposition 強制 0 ≤ rem < S。如果 prover 提供錯誤的 w（乘積偽造），則 rem = a×b − w×S 不在 [0, S) 內，byte decomposition 失敗（某個 byte > 255，lookup table 拒絕）。fp_div、arith_add、arith_sub 同理。

Layer 2 — 超越函數不可繞。

ln、erf 透過 horner_per_step gate 求值。每步約束 accnext × S + rem = acc × r + coeff — 係數 coeff 是電路常數（hardcoded），prover 無法更改。每步的截斷 rem 經 byte decomposition 驗證。因此 prover 被鎖定在正確的 Horner 求值路徑上，僅能偏離截斷的量化誤差（≤ 1 ULP / step，d=12 步累積 ≤ 12 ULP）。

exp 透過 exp_recon gate 約束 result = method_b_output × 2k，2k 查 pow2 table。目前狀態：method_b_output 為 witness，其 Method B 內部約束（byte 分解 → 6 次查表 → 5 次 fp_mul 鏈）尚未在電路中實作（計畫強化項目）。當前防線為 bs_price gate 的容差檢查：如果 exp 結果偏離真值，最終 BS price 不等於結算價，bs_price gate 拒絕。完整的 Method B 約束鏈實作後，exp 將與 ln/erf 同樣被算術閘鎖死。

Layer 3 — 合規判定不可繞。

route_accum 按符號分別累加到 pos_sum 和 neg_sum。wide_range gate 約束 gap = τ + neg_sum − pos_sum，gap 做 7-byte decomposition（56-bit）強制 gap ≥ 0，等價於 τ ≥ pos_sum − neg_sum。同理反向。因此 |pos_sum − neg_sum| ≤ τ 被強制。（注：合規判定使用 wide_range 而非 compliance_range，因為 pp=48 下 τ 和加權 Delta 的規模超過 32-bit。）

τ 來自 public input constrain_instance — prover 無法篡改。pos_sum 和 neg_sum 由前面的閘鏈唯一確定。因此條件 (e) 成立。□

12.5 Zero-Knowledge 證明

Claim. Verifier 從 π 中學不到任何 witness 資訊（除 x 本身隱含的）。

Proof. 直接由 PLONK 的 zero-knowledge 性質保證。Halo2 的 PLONK 實作在 witness polynomial commitments 上加入 blinding factors（隨機多項式），使得 KZG commitment 在統計上獨立於 witness。

具體地：advice column 的多項式 w(X) 被替換為 w(X) + b(X)，其中 b(X) 是隨機多項式（degree ≤ blinding rows 數）。由 KZG hiding 性質，commitment [w(X) + b(X)]1 不洩漏 w(X) 的資訊。

Proof #1 的 private witness 包含：持倉明細（K, w, expiry, type）、隱含波動率 σ、Merkle 路徑、所有中間計算值。這些全部在 advice columns 中，受 blinding 保護。

Public input 僅有 4 個值（state_root, market_data_root, τ, today），verifier 本就知道。□

12.6 Knowledge Extraction

Claim. 若 prover 能使 verifier accept，則 prover 「知道」滿足 R1 的 witness。

Proof. 由 PLONK 在 AGM（Algebraic Group Model）下的 knowledge soundness（Assumption 3），存在 extractor E：
給定 prover 的多項式 commitments 和 evaluation proofs，E 可萃取出所有 advice column 的完整 assignment。

由 Section 12.4 的 soundness 論證，這組 assignment 滿足 R1 的全部條件。因此 extracted witness 即為 R1 的合法 witness。

形式地：Pr[(x, E(π)) ∈ R1 | Verify(x, π) = 1] ≥ 1 − negl(λ)。□

12.7 精度保證與 Soundness 的關係

Soundness gap 由精度誤差產生：電路計算的 Δcircuit 與理想 ΔBS 存在 ≤ 57.6 ULP 的偏差（Section 5 上界），即 |Δcircuit − ΔBS| ≤ 57.6 × 2−48 ≈ 2 × 10−13。

因此 Proof #1 實際證明的是：|Σ wi · Δcircuit,i| ≤ τ，而非精確的 |Σ wi · ΔBS,i| ≤ τ。
對於 N-position portfolio，最大偏差 ≤ N × max|wi| × 57.6 ULP。在典型 20 持倉、max|wi| ≤ S 的場景下：

偏差 ≤ 20 × 57.6 × 2−48 ≈ 4 × 10−12

遠小於任何合理的監管門檻 τ（通常 ≥ 0.01）。soundness gap 在實務上可忽略。

12.7.1 已知邊界與計畫強化

以下不影響 Proof #1 的數學 soundness，但屬於工程層面的精進方向：

  1. bs_price 容差模式：現行為固定絕對容差（|gap| ≤ 0.02）。對低價選擇權（settle ≈ $0.01），相對誤差較寬鬆。計畫改為混合容差：gap ≤ max(ABS_MIN, REL_TOL × settle_price)。
  2. 近到期部位過濾：現行 OP_EXPIRY 排除 T ≤ 0 的已到期部位。極近到期（T < 1 day）的 ATM 部位 Delta 變化劇烈（pin risk），snapshot 合規的實質意義減弱。計畫加入 Tmin 下限，將此類部位排除或強制全額備兌。
  3. σ 來源：現行 σ 為 prover witness，由 bs_price gate 正向驗證。終極設計方向為 σ 由 Market Data Tree 直接提供，消除 prover 對 σ 的 witness 自由度。

最後一個 section 是 ZK Proof of Knowledge — 把前面所有的電路設計串成一個完整的安全論證。整個結構是：3 假設 → 5 條件 → 3 層 soundness → 10 攻擊向量全覆蓋。

— 停頓 —

首先是三個假設。(1) State Tree 完整且正確 — 由 Proof #2 保證，是跨 Proof 分工的核心；(2) Poseidon collision-resistant — 標準密碼學假設，128-bit 安全性；(3) PLONK 在 AGM 下 knowledge-sound — 也是標準假設，KZG commitment 在代數群模型下成立。注意 (1) 是系統特有的（依賴我們自己的 Proof #2），(2)(3) 是業界共用的密碼學基礎。

接著定義 Relation R1 — 五個條件：（逐一指出） (a) 每筆持倉在 State Tree、(b) 市場資料在 Market Data Tree、(c) σ 滿足 BS price 驗證、(d) 時間 T 正確計算、(e) 加權 Delta 合規。五個條件對應電路的五個模組群 — 這就是 completeness 和 soundness 的證明結構。

— 停頓 —

Completeness 是最簡單的部分 — 合法 witness 代入電路，31 個 gate 的 66 條約束方程都是等式，直接代入驗證。五個模組群的映射：Poseidon+Merkle（4 gates、20 constraints）保證 (a)(b)，Type+Time（2 gates、6 constraints）保證 (d)，σ 驗證（2 gates）保證 (c)，Kernel+Delta（19 gates、29 constraints）是 (e) 的前置計算，Portfolio 累加+range check（4 gates、8 constraints）完成 (e) 的合規判定。

— 停頓 —

Soundness 是核心。 反方向：假設 verifier accept，推出 witness 必然滿足 relation。我們用三層結構論證：（指向 12.4）

Layer 1 — 算術閘不可繞。 fp_mul 的 a×b = w×S + rem，如果 w 偽造，rem 超出 [0, S)，byte decomposition 的 lookup 拒絕。這是最基礎的安全機制：byte decomp + ByteTable lookup，結構簡單到不可能繞過。delta.rs 的 13 個 gate 中有 5 個算術閘（fp_mul、fp_div、arith_add、arith_sub、arith_div2）全部用這個機制。

Layer 2 — 超越函數不可繞。 horner_per_step 每一步都有約束 + byte decomposition — 係數是 hardcoded 常數，prover 改不了。d=12 的多項式 prover 需要 12 步全部通過，truncation 被 byte table lookup 鎖定。偏差上界：每步 ≤ 1 ULP，12 步累積 ≤ 12 ULP。ln 和 erf 都被這個機制鎖死。

exp 比較特殊 — Method B result 目前是 witness，內部的 lookup+multiply 約束鏈是計畫強化項目。當前安全由bs_price gate 兜底：如果 exp 結果偏離，算出的 BS price 不等於結算價，gate 拒絕。完整的 Method B 約束實作後，exp 將跟 ln/erf 一樣被算術閘鎖死。三個超越函數的安全等級總結：ln ✓ 完全約束、erf ✓ 完全約束、exp ⚠ 間接約束（bs_price 兜底）。

— 停頓 —

Layer 3 — 合規判定不可繞。 route_accum 按符號分流，wide_range 做 7-byte 分解（56-bit）強制 gap ≥ 0。τ 來自 public input（constrain_instance），prover 無法篡改。

三層之間的邏輯鏈：Layer 1 保證每一步算術正確 → Layer 2 保證超越函數求值正確（建立在 Layer 1 的 fp_mul 上）→ Layer 3 保證合規判定正確（建立在 Layer 2 的 Delta 值上）。攻擊者必須從 Layer 1 開始破解 — 而 Layer 1 的 byte decomposition 是最堅固的防線。

— 停頓 —

Under-constrained 分析列出了十個潛在攻擊向量。（指向表格） 按攻擊目標分四類：

操縱輸入參數（2 個）：偽造 σ、deep OTM 的 σ 容差操縱。操縱計算結果（3 個）：偽造 Horner 結果、偽造 exp、偽造符號。操縱投組組成（2 個）：重複計入、遺漏持倉 — 這兩個是Proof #1 不防的，by design 交給 Proof #2。利用數值弱點（2 個）：range check overflow、T→0 pin risk。另外 τ/today 是 public input（verifier 提供），prover 本來就不控制 — constrain_instance 確保電路內部一致，屬於架構保證而非需要防禦的攻擊。

按防禦機制分：Gate 約束直接擋住 4 個（偽造 σ → bs_price、偽造 Horner → horner_per_step、偽造符號 → sign_extend、overflow → wide_range byte decomp）。數學與經濟學天然防堵 2 個（deep OTM 的 Vanna-Vega 聯動 — 大幅改 σ 會讓 price 暴增過不了閘，T→0 pin risk 在 snapshot 時間點數學正確）。跨 Proof 分工 2 個（重複/遺漏 → Proof #2 的 State Tree 完整性保證）。間接約束 + 計畫強化 1 個（exp 的 bs_price 兜底 + Method B 約束鏈 TODO）。架構保證 1 個（τ/today 為 verifier 提供的 public input，不可篡改）。

— 停頓 —

Zero-knowledge 直接 delegate 給 PLONK — Halo2 的 blinding factors 確保 KZG commitment 不洩漏 witness。Private witness 包含持倉明細、σ、Merkle 路徑、所有中間值 — 全在 advice columns，受 blinding 保護。Public input 只有 4 個值（state_root, market_data_root, τ, today）。隱私保護範圍：verifier 知道「投組合規」但不知道任何一筆持倉的內容。

最後一個重點：精度 vs soundness。電路算的 Δ 和理想 BS Δ 有 ≤ 57.6 ULP 的理論偏差上界（Section 5 區間算術證明），實測最差僅 6.04 ULP。對 20 持倉的投組，累積偏差 ≤ 4×10−12 — 比任何合理的 τ（通常 ≥ 0.01）小 10 個數量級。soundness gap 在實務上完全可忽略。

總結整個安全論證：3 假設 → 5 條件 → 3 層 soundness → 10 攻擊向量全覆蓋 → 精度 gap 可忽略。Proof #1 構成一個完整的 ZK-POK — prover 知道一組合規的持倉，精度保證 ≥ 42 有效位元（理論 57.6 ULP）/ 45 有效位元（實測 6.04 ULP），verifier 學不到超出 4 個公開值的任何資訊。三個計畫強化方向（bs_price 混合容差、Tmin 下限、σ 由 Market Data Tree 提供）屬於工程精進，不影響核心數學 soundness。
