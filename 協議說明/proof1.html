Proof #1: Delta Compliance

投資組合 Delta 風險合規證明 — 技術細節

電路架構、自訂閘、約束管線的完整說明

0金融背景：什麼是 Delta 合規

0.1 什麼是 Delta

Delta（Δ）衡量一個金融部位對標的資產價格變動的敏感度。Δ = 0.6 代表股價漲 1 元，這個部位大約漲 0.6 元。

資產類型Delta 範圍說明
股票（做多）+1.0股價漲 1 元 → 部位漲 1 元
股票（做空）−1.0股價漲 1 元 → 部位虧 1 元
歐式 Call0 ~ +1由 Black-Scholes 公式計算
歐式 Put−1 ~ 0由 put-call parity 推導

0.2 Black-Scholes Delta 公式

Step 1 — 計算 d1：
d1 = [ln(S/K) + (r − q + σ²/2) × T] / (σ × √T)

Step 2 — Call Delta：
Δcall = e−qT × N(d1)

Step 3 — Put Delta（由 put-call parity）：
Δput = Δcall − e−qT

其中 N(x) = 0.5 + 0.5 × erf(x / √2)（常態分佈累積函數）
符號意義來源
S標的資產現價市場數據（Market Data Tree）
K履約價合約條款（Position State Tree）
T距到期的年數電路內計算：(expiry − today) / 365
r無風險利率市場數據
q股息殖利率市場數據
σ隱含波動率由結算價反推（Newton-Raphson，電路外）

0.3 合規檢查公式

|Σi wi × Δi| ≤ τ

wi = 第 i 筆持倉的數量（正=做多，負=做空）
Δi = 第 i 筆持倉的 Delta（股票=1，選擇權=BS Delta）
τ = 合規門檻（由 LPA 或監管規定）
這個公式本身很簡單 — 任何 Excel 都能算。困難的是在零知識電路裡面算，因為公式包含三個超越函數：ln（自然對數）、erf（誤差函數）、exp（指數函數）。ZK 電路只有加法和乘法，沒有這些函數 — 這就是本論文要解決的核心問題。

在進入 Proof #1 的技術細節之前，先用兩分鐘說明我們要證明什麼。

Delta 衡量一個金融部位對標的資產價格變動的敏感度。股票的 Delta 固定是 1，選擇權的 Delta 則由 Black-Scholes 公式動態計算，取決於市價、履約價、波動率、到期時間等參數。

一個投資組合可能同時持有股票、call option、put option。合規檢查要求：所有持倉的加權 Delta 總和的絕對值不超過門檻 τ。公式本身很簡單，但 BS Delta 裡面包含三個超越函數 — ln、erf、exp。這些函數在一般程式語言中有 CPU 的浮點單元直接支援，但零知識電路的運算核心是算術電路，只有有限域上的加法和乘法，沒有浮點數、沒有迴圈、沒有這些函數。

如何在零知識電路中精確計算這三個超越函數，就是本論文在協議層面要解決的核心工程問題。接下來我會從 ZK statement 開始，逐步展開整個電路的設計。

1ZK Statement（形式化陳述）

Proof #1 proves:
∃ positions {p1, ..., pN} ⊂ State(T), ∃ {σ1, ..., σN} such that:

(a) ∀i: MerkleVerify(pi, pathi, state_root) = true （注：Proof #1 不強制葉索引唯一性 — 由 Proof #2 的 State(T) 完整性保證）
(b) ∀i: MerkleVerify(mkti, mkt_pathi, market_data_root) = true
(c) ∀i: |BS(S, Ki, Ti, r, q, σi) − settle_pricei| ≤ εσ
(d) ∀i: Ti × 365 = expiry_datei − today
(e) |Σi wi · ΔBS(S, Ki, Ti, r, q, σi)| ≤ τ

了解了金融背景之後，來看 Proof #1 的形式化陳述。這個證明的目標是：讓基金公司在不揭露任何持倉細節的前提下，證明投資組合的 Delta 風險在合規門檻之內。

為什麼用零知識證明？三個替代方案各有致命缺點：(1) 可信第三方需要揭露持倉明文；(2) MPC 需要多方在線協作，延遲高；(3) FHE 的乘法深度限制讓超越函數計算極為昂貴。ZK proof 的優勢：prover 在本地一次性計算，生成 ~1 KB 的 proof，verifier 在毫秒內驗證 — 非互動、可公開驗證、不洩漏私密輸入。

— 指向 ZK Statement —

為什麼拆成兩個 proof？Proof #1 做合規計算，Proof #2 做狀態完整性。拆分的理由是關注點分離：金融數學和資料結構的邏輯獨立，更新頻率也不同。

Proof #1 要證明五件事。（逐條指）

條件 (a)：每筆持倉存在於 State Tree — 透過 Merkle proof。這裡有個設計取捨：Proof #1 不檢查葉索引唯一性，因為唯一性需要 O(N²) 的比對，由 Proof #2 的 tree update 邏輯天然保證。

條件 (b)：市場數據通過 Merkle 驗證。條件 (c)：σ 正向驗證 — 用 σ 算出的 BS price 接近結算價。條件 (d)：到期時間 T 在電路內計算。條件 (e)：加權 Delta 總和的絕對值不超過 τ。

公開輸入只有 4 個。設計原則：公開輸入越少，隱私性越強。所有持倉細節都是 private witness。

在看各部分細節之前，先說明兩個基礎選擇。

為什麼選 Halo2 + KZG？Delta 計算需要可程式化的自訂閘、高效的 lookup table（用於 range reduction 和 byte decomposition）、以及成熟的工具鏈。Halo2 基於 PLONKish 算術化，支援這些需求，KZG 提供常數大小的 proof（~1 KB）和毫秒級驗證。Groth16 不支援 lookup，STARKs 的 proof 太大（~100 KB）。

第二個選擇是定點算術 pp=48。浮點數需要分別追蹤指數和尾數，每次運算都要做正規化，在 ZK 電路中代價極高。定點算術把所有數乘以 S = 248 ≈ 281 兆，用整數運算模擬小數。pp=48 提供約 14.4 位十進位精度，端對端 BS Delta 的絕對誤差達到 ~2×10−14，遠超金融計算的需求。

— 指向管線圖 —

整體電路分為五個部分。（逐一指） Part A、A' 做 Merkle 驗證。Part A'' 驗證 σ。Part B 是核心貢獻 — Delta 計算。Part C 做合規檢查。每個部分對每一筆持倉獨立執行，最多 128 slots，最後在 Part C 匯總。即使在最高精度 pp=48 下，Part B 也僅佔整體電路的 ~13%。

3Part A: State Merkle 驗證

驗證每筆持倉確實存在於 state_root 承諾的 Merkle tree 中。

3.1 葉節點 Hash

5 個欄位透過 Poseidon sponge 壓縮為單一 hash：

leaf_hash = PoseidonSponge([w, K, expiry_date, asset_type, underlying_id])

Sponge 模式：rate=2, capacity=1, 每次吸收 2 個欄位，5 個欄位需要 3 次 permutation。

3.2 Merkle 路徑驗證

從葉節點往上走 7 層（depth = 7），每層：

  1. 根據 position_bit 決定 (left, right) 順序
  2. 計算 parent = Poseidon(left, right)
  3. 最終 root 必須等於 instance[0]（state_root）

每條 Merkle 路徑約 528 rows/position（7 層 × ~64 rows/Poseidon + 葉 hash）。

4Part A': Market Data Merkle 驗證

驗證每筆持倉使用的市場參數 (S, r, q, settle_price) 確實來自 market_data_root 承諾的公開數據表。

4.1 Market Data 葉節點

8 個欄位的 Poseidon sponge hash（4 次 permutation）：

mkt_hash = PoseidonSponge([underlying_id, strike, expiry_date, asset_type, spot, r, q, settle_price])

4.2 Cross-Tree 約束

Position leaf 和 Market Data leaf 透過 8 條 copy constraints 連結，確保 Proof #1 使用的市場數據與持倉的合約條款一致：

Position leaf 欄位約束Market Data leaf 欄位
underlying_id=underlying_id
K=strike
expiry_date=expiry_date
asset_type=asset_type
K=delta.strike
mkt.spot= delta.stock (S)
mkt.r= delta.r
mkt.q= delta.q

前 4 條確保合約條款（underlying、strike、expiry、type）一致；後 4 條將市場參數 (S, r, q) 和持倉 K 直接連結到 Delta 管線的輸入 cells。

為什麼 Market Data 用 Merkle 而不用 Lookup Table？Market Data 每日更新 — 如果用 Halo2 fixed column lookup，每次更新都需要重新產生 setup 和 VK。使用 Merkle tree 讓電路結構完全固定，只有 root 值（公開輸入）每日變化。

Part A' 是整個電路的主要成本來源，約佔 55%（~1,386 rows/position），因為 Market Data 有 8 個欄位（需要 4 次 Poseidon permutation）且 tree depth ~20。

Part A 是標準的 Merkle 驗證。每個持倉有 5 個欄位，透過 Poseidon sponge 壓縮為單一 hash，再沿 7 層路徑驗證到 state_root。

為什麼用 Merkle tree 而不是 KZG 多項式承諾？三個原因：O(log N) 的成員證明和更新效率、概念簡單的安全性分析、以及只要 hash 函數抗碰撞，承諾就是 binding 的。為什麼選 Poseidon hash？SHA-256 是位元運算，一次需要 ~25,000 個約束；Poseidon 是專為 ZK 設計的代數 hash，直接在有限域上運算，一次 permutation 只需 ~64 個約束 — 兩個數量級的差距。這個選擇直接決定了整個系統的可行性。

— 翻到 Part A' —

Part A' 結構類似，處理市場數據（8 個欄位，4 次 Poseidon permutation）。

一個關鍵設計：為什麼市場數據不用 Halo2 lookup table？因為 fixed column 是 setup 的一部分 — 市場數據每天變，如果用 lookup 就要每天重新 setup。改用 Merkle tree，電路結構一次 setup 永久有效，只有 root 每日更新。

兩棵樹之間透過 8 條 copy constraints 連結 — PLONK 原生機制，零成本強制 cell 相等。前 4 條確保合約條款一致，後 4 條把市場參數直接綁定到 Delta 管線。Part A' 佔了整個電路的 ~61%，是最主要的成本來源。

5Type Decomposition 與 Equity MUX

5.1 類型分解閘 (q_type_decomp)

將 asset_type（0/1/2）分解為兩個布林值：

asset_type = is_put + 2 × is_equity

asset_type = 0 (EU Call)→is_put = 0, is_equity = 0
asset_type = 1 (EU Put)→is_put = 1, is_equity = 0
asset_type = 2 (Equity)→is_put = 0, is_equity = 1
閘約束：is_equity × (1 − is_equity) = 0（確保布林）。is_put 的布林性由 Delta 管線隱式保證。

5.2 Equity MUX 閘 (q_equity_mux)

Equity 部位不需要 Black-Scholes — Delta 固定為 1.0：

effective_delta = is_equity × 1.0 + (1 − is_equity) × bs_delta

Equity: Δ = 1.0（持倉量 w 就是 Delta 貢獻）
Option: Δ = BS Delta（經過完整 ln/erf/exp 管線計算）

6Ti 電路內計算

到期時間 Ti 不是由 prover 直接提供的 — 電路在內部計算並驗證。

6.1 Time Verify 閘 (q_time_verify)

Ti × 365 = expiry_datei − today

• expiry_datei 來自 Merkle-verified Position leaf
• today 來自公開輸入 instance[3]
• Ti 是 prover 提供的 witness，電路驗證其正確性
• 選擇權部位（type 0/1）額外檢查 Ti > 0（防止除以零）
為什麼不直接在電路內做除法？ZK 電路中的除法 = 乘法逆元，成本較高。改為 prover 在電路外算好 Ti = (expiry − today) / 365，電路只需驗證 Ti × 365 == expiry − today（一個乘法閘）。

進入 Delta 計算之前，電路做兩件前處理。

第一是類型分解。三種資產（call、put、equity）被分解為 is_put 和 is_equity 兩個旗標。ZK 電路沒有 if-else，所以用 MUX 閘：兩個值都算出來，然後加權選擇。Equity 的 Delta 固定是 1，不需要 BS 計算。

第二是到期時間驗證。T 不信任 prover — 因為操縱 T 會改變 Delta。電路自己驗證：T × 365 = expiry_date − today。這是 ZK 中常見的「prover hint + verifier check」模式：prover 在電路外算好 T，電路只做一個乘法閘驗證正確性，避免昂貴的電路內除法。

7Part A'': σ 正向驗證

隱含波動率 σ 是由 prover 在電路外用 Newton-Raphson 法從結算價反推出來的。電路的任務是驗證這個 σ 值是否合理 — 使用它計算一次 BS price，檢查是否接近結算價。

7.1 BS Price 計算

d1 = [ln(S/K) + (r − q + σ²/2) × T] / (σ × √T)
d2 = d1 − σ × √T

Call price = S × e−qT × N(d1) − K × e−rT × N(d2)
Put price = K × e−rT × N(−d2) − S × e−qT × N(−d1)

7.2 電路內驗證

q_d2_sub gate

計算 d2 = d1 − σ√T（帶符號減法處理）

q_bs_price gate

驗證 BS price ≈ settle_price。使用加法形式避免除法：

bs + is_put × s_disc_q + term2 = term1 + is_put × k_disc_r
其中 term1 = stock × Δcall / S（= S·e−qT·N(d1) / S），term2 = strike × disc_r × N(d2) / S，
s_disc_q = stock × discount / S，k_disc_r = strike × disc_r / S（皆為 fp48 定點值）

容許偏差：SIGMA_TOL = 0.02 × 248 ≈ 5.6 兆（絕對容差 0.02，非百分比）

σ 驗證的安全意義：如果不驗證 σ，基金可以填入讓 Delta 最小的假 σ（例如極高波動率會讓所有 Delta → 0.5，降低加權效果）。結算價鎖定使 σ 成為隱函數的唯一解— 基金無法操縱。

σ 正向驗證是關鍵安全機制。如果不驗證 σ，基金可以填入讓 Delta 趨近於零的假 σ 來規避合規。

為什麼用「正向驗證」而不是電路內求解 σ？求解需要 Newton-Raphson 迭代 — ZK 電路不支援動態迭代。但給定 σ 去算一次 BS price 只需前向計算，正好就是 Delta 管線已經能做的事。策略是：prover 電路外做反向求解，電路只負責正向驗證。

容許偏差 SIGMA_TOL = 0.02（絕對容差，pp=48 下 = 0.02 × 248 ≈ 5.6 兆）。為什麼用絕對容差？相對容差需要除法，成本高；絕對容差只需一個 range check。

BS price 改寫成加法形式以避免有限域 wrap-around — 把減號兩邊移到等式的不同側，所有項都是正的。結算價是公開的、不可篡改的，所以 σ 成了隱函數的唯一解。

8.3 Put Delta 調整

Put Δ = Call Δ − e−qT（由 put-call parity 推導）

電路使用 MUX：is_put = 0 → Call Δ，is_put = 1 → Call Δ − discount

現在進入核心貢獻 — Delta 計算管線。BS Delta 包含三個超越函數：ln、erf、exp。我們對每個函數使用不同的最優策略。

— 停頓 —

ln 和 erf：Minimax 分段多項式 + Per-step Horner。

為什麼選 Minimax 而不是 Taylor？Taylor 在展開點附近精度高，但離開展開點誤差快速增長。Minimax 多項式在整個區間上均勻分配誤差，由 Chebyshev 等振盪定理保證最優。同樣的次數，Minimax 比 Taylor 精度高 2,000–4,000 倍。

為什麼用分段逼近？單一多項式在寬區間上會遇到「量化飽和」— 增加次數反而惡化精度，因為高次係數的定點量化誤差被放大。分段逼近把區間切成窄段（ln 4 段 × 0.25，erf 12 段 × 0.5），每段只需低次多項式（d=12）就能達到 ~1.2 ULP 的 kernel 精度。

Per-step Horner 是求值法的關鍵。（指向管線表格） 把多項式改寫為巢狀形式 c₀ + x(c₁ + x(c₂ + ...))，每一步都做截斷（除以 S = 248），讓中間值始終保持在 S² 以內。截斷驗證用 ByteTable lookup — 把值拆成 6 個 byte，每個 byte 查一次 256-entry 的共用表。配合分段逼近（segment width < 1），截斷誤差呈幾何衰減，收斂到常數地板。

exp：Method B 查表乘法。exp 不用多項式，而是利用 ea+b = ea × eb 的分解特性。把 48-bit 輸入拆成 6 個 byte，每個 byte 位置預先算好一張 256-entry 的 ebyte×28k/S×S 查表，然後 5 次乘法-截斷把 6 個中間值相乘。6 張表共 1,536 entries，只需 6 行電路 — 比 d=12 Horner（12 行）更緊湊，精度 ~8 ULP。

Range reduction：每個函數的多項式只在特定區間上有效。例如 ln 先把 S/K 映射到 [1, 2)：x = 2k × r，求值後還原 ln(x) = ln(r) + k × ln(2)。erf 用 segment index 選擇對應段的係數。Range reduction 用 ByteTable lookup 實現，成本很低。

整個管線 17 個邏輯步驟，pp=48 配置下每個持倉約 298 rows，佔總電路 ~13%。相比之下，Merkle 驗證佔 ~85%。端對端 BS Delta 精度達 ~6 ULP（絕對誤差 ~2×10−14）。

Put option 利用 put-call parity：Put Δ = Call Δ − e−qT，只需一個減法閘。

9Part C: 合規累加與檢查

9.1 帶符號累加

每筆持倉貢獻 wi × Δi，w 有正負（多/空）：

w_sign XOR is_put → 路由到 positive_sum 或 negative_sum

positive_sum = Σ(正向貢獻的 |w| × Δ)
negative_sum = Σ(負向貢獻的 |w| × Δ)

9.2 雙向合規檢查

Delta 合規不只檢查淨值，而是雙向檢查：

τ + negative_sum ≥ positive_sum
τ + positive_sum ≥ negative_sum

等價於：|positive_sum − negative_sum| ≤ τ
使用 range check 而非直接比較，避免有限域中的負數問題。

τ 來自公開輸入 instance[2]（threshold）。

最後一步是合規檢查。w 有正有負（多倉和空倉），但有限域沒有負數 — 直接做 a − b 可能 wrap around。所以正項和負項分別累加，最後用雙向 range check：τ + negative_sum ≥ positive_sum，同時 τ + positive_sum ≥ negative_sum。兩邊都是非負數相加，不會 underflow。兩個同時成立就等價於 |net delta| ≤ τ。

這是 ZK 中處理帶符號比較的標準做法 — 把「比大小」轉化為「驗證一個差值落在合法範圍內」。pp=48 下 range check 使用 7 個 byte（56-bit）的分解驗證，足以涵蓋所有可能的累加值。

11自訂閘清單與約束方程

以下列出 Proof #1 電路的 10 種自訂閘，每個閘對應一組多項式約束。Halo2 的 PLONKish 架構允許 degree ≤ 5 的自訂閘，每個閘在啟用時（selector = 1）強制其約束方程為零。

q_poseidon — Poseidon Permutation

位置：Part A（State Merkle）、Part A'（Market Data Merkle）

約束：對 t=3 的狀態向量 (s0, s1, s2) 施加完整的 Poseidon round — S-box（x5）+ MDS 矩陣乘法 + round constant 加法。每次 permutation 需要完整的 full rounds + partial rounds。

每次：~66 rows/permutation

q_merkle_swap — Merkle 路徑選擇

位置：Part A、Part A'

約束：根據 position_bit 決定 hash 順序
bit × (1 − bit) = 0（布林約束）
left = bit × sibling + (1 − bit) × current
right = bit × current + (1 − bit) × sibling

每次：1 row/層

q_type_decomp — 資產類型分解

位置：每持倉一次

約束：
asset_type = is_put + 2 × is_equity（值分解）
is_equity × (1 − is_equity) = 0（布林約束）
is_put 的布林性由 Delta 管線隱式保證

每次：1 row

q_time_verify — 到期時間驗證

位置：每持倉一次

約束：
Ti × 365 = expiry_datei − today（乘法驗證）
Prover 在電路外計算 Ti，電路只驗證正確性 — 避免電路內除法

每次：1 row

q_equity_mux — Equity/Option MUX

位置：每持倉一次

約束：
effective_delta = is_equity × 1.0 + (1 − is_equity) × bs_delta
Equity 部位 Δ = 1.0（跳過 BS 計算），Option 部位使用完整管線結果

每次：1 row（2 乘法 + 1 加法）

q_d2_sub — d2 帶符號減法

位置：Part A''（σ 驗證）

約束：
d2 = d1 − σ × √T（帶符號，結果可能為負）
使用符號位 + 絕對值表示法，避免有限域 wrap-around

每次：1 row

q_bs_price — BS Price 正向驗證

位置：Part A''（σ 驗證）

約束：
bs + is_put × s_disc_q + term2 = term1 + is_put × k_disc_r（加法形式 — 避免減法 wrap-around）
|bs_price − settle_price| ≤ SIGMA_TOL（range check，容差 0.02 × 224）
同一個閘用 is_put 選擇 Call/Put 公式

每次：~4 rows（含 range check）

q_kernel_* — Per-step Horner 多項式求值（ln / erf / exp）

位置：Part B（Delta 管線）— 本論文核心貢獻

約束：Per-step Horner 求值鏈
acc0 = cd（最高次係數）
acck+1 = (acck × x + cd−k−1) / S（每步乘法 + 加法 + 截斷）
每步截斷讓中間值保持在 S² 以內，避免有限域溢出
截斷驗證使用 ByteTable lookup（256-entry 共享表）

每個函數：~8–16 rows（degree 6–14 依精度等級，+ 截斷 + range reduction）

q_sign_extend — 符號擴展

位置：Part C（合規累加）

約束：
w_sign × (1 − w_sign) = 0（布林約束）
w = w_sign × (−|w|) + (1 − w_sign) × |w|（2's complement 還原）
將 w 分解為符號位和絕對值，供後續正負分離累加使用

每次：1 row

q_compliance — 雙向合規 Range Check

位置：Part C（合規檢查）— 全電路最後一步

約束：
τ + negative_sum ≥ positive_sum（range check #1）
τ + positive_sum ≥ negative_sum（range check #2）
兩者同時成立 ⇔ |positive_sum − negative_sum| ≤ τ
使用加法形式避免有限域中的負數 wrap-around 問題

每次：2 rows（2 個 range check）

閘與管線的對應關係：每個閘對應 BS Delta 計算的一個步驟。資料從 Merkle 驗證（q_poseidon, q_merkle_swap）→ 前處理（q_type_decomp, q_time_verify）→ 金融計算（q_kernel_*, q_d2_sub, q_bs_price）→ 合規判定（q_equity_mux, q_sign_extend, q_compliance），形成一條完整的約束管線。

以上是 Proof #1 的全部技術細節。10 種自訂閘構成一條完整的端對端管線。

總結技術選擇：Halo2 + KZG 提供可程式化的 PLONK 框架和常數大小的 proof；Poseidon hash 讓 Merkle 驗證可行（比 SHA-256 快 100 倍）；Merkle tree 讓電路結構一次 setup 永久有效；分段 Minimax + Per-step Horner 讓 ln 和 erf 在 ZK 中達到 ~1.2 ULP kernel 精度；Method B 查表乘法讓 exp 只需 6 行電路；pp=48 定點算術提供 ~2×10−14 的端對端絕對精度。

每一個選擇都有明確的理由。精度等級的選擇經過了系統性的 degree sweep 實驗驗證 — d=12 是逼近誤差和截斷累積的 U 曲線最低點，分段數量（ln 4、erf 12）則對應各函數的曲率特徵。
