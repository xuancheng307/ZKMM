Motivation03 / 87

金融合規與持倉隱私的張力

對沖基金必須向監管證明風險合規，但持倉本身是商業機密 — 公開 = 策略外洩、front-running。

傳統做法

基金交出完整持倉 → 監管自己算 → 策略曝光，競爭對手可複製

ZK 合規

基金提交 proof → 監管毫秒驗證 → 持倉永遠保密，proof 存在 = 合規

ZK 是唯一能同時做到「即時可驗證 + 永久隱私」的技術；MPC 需多方協作延遲高，FHE 在乘法深度下對超越函數過於昂貴。

ZK Value06 / 87

ZK 能提供什麼

「我能證明某件事是真的，但不透露為什麼是真的。」

  • Completeness — 誠實 prover 總能通過
  • Knowledge Soundness — 惡意 prover 不能偽造通過（且真的「知道」witness）
  • Zero-knowledge — 驗證者除了「命題為真」外什麼都學不到

關鍵：ZK 電路裡只有有限域上的加法和乘法 — 沒有 if、沒有迴圈、沒有浮點。這是本論文要解決的核心工程問題。

"Zero-knowledge proofs are defined as those proofs that convey no additional knowledge other than the correctness of the proposition in question."

— Goldwasser, Micali, Rackoff, The Knowledge Complexity of Interactive Proof Systems, SIAM J. Comput. 18(1):186–208 (1989), Abstract [1]

"We say that V is a knowledge verifier for the relation R with knowledge error κ if … there exists a probabilistic oracle machine K such that … if p(x) > κ(x) then, on input x and access to oracle Px, machine K outputs a string from the set R(x) within an expected number of steps bounded by |x|c / (p(x)−κ(x)). The oracle machine K is called a universal knowledge extractor, and κ is called the knowledge error function."

— Bellare, Goldreich, On Defining Proofs of Knowledge, CRYPTO '92, Definition 3.1 [2]

PF1 Statement15 / 87

PF1 形式化陳述 — NP relation RPF1

DEF. NP relation R ⊆ {0,1}* × {0,1}*：public statement x 與 witness w 的多項式時間可驗證關係；語言 LR = {x : ∃w, (x,w) ∈ R}。Prover 要證 x ∈ LR 且不洩漏 w。

Statement x = (state_root, market_data_root, τ, today)；witness w = ({leafi, pathi, mkti, mkt_pathi, σi, ...})。(x, w) ∈ RPF1 iff 對所有 i ∈ [1, N]：

  • (a) MerkleVerify(pi, pathi, state_root) = true
  • (b) MerkleVerify(mkti, mkt_pathi, market_data_root) = true
  • (c) |BS(S,Ki,Ti,r,q,σi) − settle_pricei| ≤ εσ（σ 正向驗證）
  • (d) Ti × 365 = expiry_datei − today（到期期限一致性）
  • (e) |Σ wi · ΔBS(S,Ki,Ti,r,q,σi)| ≤ τ（合規門檻）

DEF. Merkle tree（Merkle 1979）：高度 d 的完全二元樹，每個內部節點儲存 h(left ∥ right)，h 為抗碰撞雜湊。Authentication path = 從葉到根的 d 個 sibling hash，允許 verifier 以 O(d) 成本確認某葉屬於 commitment root。在 SNARK 電路內驗證一條 path 要 d 次 hash gate evaluation。

為什麼用 Merkle 而非 Halo2 lookup？ Market data 每日變 — 若用 fixed column lookup 每日得重做 preprocessing + verification key。Merkle 讓電路結構一次 preprocessing 永久有效，只有 root 值每日更新。

"we describe a method called 'tree authentication' which can be used to authenticate any Yi of any user quickly and easily, but which requires minimal storage."

— Merkle, A Certified Digital Signature (1979), §6 Tree Authentication [11]

Recap: Groth16 / R1CS22 / 87

先回到大家熟悉的 — Groth16 / R1CS / QAP

DEF. zk-SNARK（succinct non-interactive argument of knowledge）for NP relation R：三元組 (Setup, Prove, Verify) 滿足 (i) completeness — 誠實 prover 對 (x,w)∈R 必能讓 verifier 接受；(ii) knowledge soundness — 任一 PPT prover 若能讓 verifier 接受，則存在 extractor 可抽出合法 w（except with negligible probability）；(iii) zero-knowledge — 存在 simulator 不用 w 即可產生與真實 proof 不可區分的 transcript；(iv) succinctness — proof size 與 verify time 對 circuit size 為 poly(λ, log|C|)。
  • R1CS (Rank-1 Constraint System)：constraint 形式固定為 (A·z) ⊙ (B·z) = C·z，每條 constraint 一個 row
  • QAP (Quadratic Arithmetic Program)：把 R1CS 轉成多項式 identity A(X)·B(X) − C(X) = H(X)·Z(X)，用 bilinear pairing 一次驗證
  • Trusted setup（CRS）：每個 circuit 各做一次 multi-party ceremony（toxic waste），circuit 改了重做
  • Proof size：3 G1 群元素 ≈ 200 bytes（constant）
  • 無 lookup argument：range check 需位元分解（8-bit → 8 條 constraints）

"We construct a pairing-based (preprocessing) SNARK for arithmetic circuit satisfiability, which is an NP-complete language, where a proof is only 3 group elements and verification consists of checking a single pairing product equation."

— Groth, On the Size of Pairing-based Non-interactive Arguments, EUROCRYPT 2016, Abstract [3]

這些是 Halo2 不一樣的地方的對照基準。

Plonkish23 / 87

Halo2 = Plonkish arithmetization

DEF. Arithmetization：把 NP computation 編碼成代數 / 多項式可驗證形式（R1CS、AIR、Plonkish 皆屬之）。Plonkish = 矩形 trace table（rows × columns）+ 多元多項式 gate constraint + selector column 開關。每個 constraint 是對 cell 值在 𝔽p 上的多項式等式，滿足的條件是「所有啟用 gate 的 row 上 polynomial = 0」。

直覺：整個電路是一張「表格」。列 = 一行計算；欄 = 不同類型的變數；gate 是欄之間的多項式關係。

R1CS (Groth16)

固定 a·b=c 形式

一條 constraint 一個 row

一個 row 只有一條 constraint

→

Plonkish (Halo2)

自訂高次多項式 gate

一個 gate 可跨多列多欄

一個 row 可同時啟用多條 constraint

"PLONKish circuits are defined in terms of a rectangular matrix of values. (…) A PLONKish circuit configuration requires a sequence of polynomial constraints. These are multivariate polynomials over F that must evaluate to zero for each row." A gate is "a set of constraints controlled by a set of selector columns."

— The halo2 Book, §Concepts › Proof systems › Arithmetization [5]

Custom Gates25 / 87

Custom gate — 自訂高次多項式

R1CS 的 gate 固定是 bilinear；Halo2 可以寫任意 degree 的 constraint。

meta.create_gate("horner", |meta| {
  let q    = meta.query_selector(q_horner);
  let acc  = meta.query_advice(acc, Rotation::cur());
  let next = meta.query_advice(acc, Rotation::next());
  let c    = meta.query_fixed(coeff, Rotation::cur());
  let x    = meta.query_advice(x, Rotation::cur());
  vec![q * (next - (acc * x + c))]   // degree-3 約束
});

白話：當 q=1，下一列的 acc = 這列 acc × x + coeff — Horner 一步到位。R1CS 要拆成兩條 constraint + 一個中間變數。

[Paraphrase — 非原文] PLONK 原論文定義的 gate 形式為 qL·a + qR·b + qO·c + qM·a·b + qC = 0（degree ≤ 2）。Halo2 將此推廣為任意多變數、任意 rotation 的多項式約束，本頁的 Horner gate 即為 degree-3 範例。

— 依 Gabizon, Williamson, Ciobotaru, PLONK (ePrint 2019/953) 通用 gate 形式改寫 [4]
Why high degree26 / 87

為什麼 Halo2 能 degree 高？

PLONK 最早只允許 degree-2。Halo2 靠更高次商多項式 + rotation + lookup，gate 實務 degree 可達 8–10。

  • 好處：一條 constraint 幹以前 5–10 條 R1CS 的事
  • 代價：prover FFT 範圍放大到 ≥ 2·degree·n，但仍 quasi-linear
  • Poseidon x⁵ S-box 就是高 degree 的直接受益者
Copy Constraints27 / 87

Copy constraint — permutation argument

DEF. Permutation argument（PLONK 原論文 §5）：prover 要證明兩個序列 (a1,...,an)、(b1,...,bn) 互為某 permutation σ 的 re-ordering，透過 grand product argument ∏(ai+γ)/∏(bi+γ) = 1（γ 為 Fiat-Shamir challenge）證明。Copy constraint：把多個 cell 宣告屬於同一 equivalence class，permutation argument 強制它們取相同值，無需額外 gate。

R1CS

同一變數在多條 constraint 共用 z[i] — 天然 same-value，不需額外機制

↔

Halo2

不同 cell (row, col) 之間透過 permutation argument 綁定 = 同值；零額外 gate 成本

Part V 的 Sigma-to-MDT 8 條約束、pipeline 中介值銜接，本質都是 copy constraint；具體 prover 成本在 grand product poly（+1 column）不在 gate layer。

Lookup28 / 87

Lookup argument — R1CS 沒有的東西

DEF. Lookup argument（Plookup, Halo2）：prover 要證明 advice cell (f1,...,fn) 之多集合是 table (t1,...,tm) 多集合的子集（as multisets）。實作為 logarithmic-derivative 或 grand-product argument：∏i(γ+fi) · ∏j(γ+tj)cj−1 = ∏j(γ+tj)cj' 之類的 identity。prover 成本 O(n log n) table scan，gate 上零代價。

直覺：「這個 advice 值一定落在這張預定義表裡。」

  • 用途：range check (0 ≤ x < 2⁸)、S-box、非線性查表
  • R1CS 要做 8-bit range check 得拆 8 個位元 = 8 條 constraint
  • Halo2 一次 lookup 就好，table 可由多 circuit 共用
  • 本 PF1 的 ByteTable（256 entries）是整個電路 range check 的萬能工具 — 83 lookup arguments 共享此表
Universal SRS29 / 87

Universal SRS — 不是 per-circuit ceremony

DEF. Structured Reference String (SRS) / Common Reference String (CRS)：trusted setup 產生之公開參數，供 prover/verifier 共同使用。典型 KZG SRS ({gτi}i=0d, g2, g2τ)，τ 為 trapdoor（toxic waste，多方 ceremony 後銷毀）。Universal SRS：對所有 degree ≤ d 的 circuit 通用；updatable：任一參與者可 append 新貢獻重新隨機化（Groth–Kohlweiss–Miers–Miers–Sarkar 等方案）。

Prove/Verify30 / 87

Prover / Verifier 流程（Plonk-style）

DEF. Polynomial Commitment Scheme (PCS)：三元組 (Commit, Open, Verify)。Commit(f) 對多項式 f ∈ 𝔽p[X] 產生簡短承諾 c；Open(f, z) 產生在點 z 的 opening proof π；Verify(c, z, y, π) 驗證 f(z) = y。安全性要求 (i) binding：prover 無法對同 z 開出兩個不同 y；(ii) hiding（選配）：commitment 不洩漏 f 資訊。KZG（Kate–Zaverucha–Goldberg 2010）：c = gf(τ)，opening 為 π = g(f(X)−y)/(X−z)|X=τ，常數大小、pairing-based。
DEF. Fiat–Shamir transform（Fiat & Shamir, CRYPTO '86）：將任一 public-coin interactive proof 轉為 non-interactive — prover 把每輪 verifier 的 random challenge c 替換為 c := H(transcriptso far)（H 建模為 random oracle）。在 Random Oracle Model (ROM) 下保留 soundness，transcript 可被任意第三方獨立驗證。Halo2 的 non-interactivity 即基於 Fiat–Shamir，H 由 Blake2b transcript 實現。

Verifier work O(1)：~5 個 pairing + 一些 𝔽p 算術 ≈ 8 ms（本電路 N=128 實測）。Proof size 31 KB 跨 N invariant。

Halo2 能表達什麼 · 不能表達什麼

Why not SHA36 / 87

為何不用 SHA / Keccak — algebraic hash

DEF. Algebraic / arithmetization-friendly hash（或 SNARK-friendly hash）：設計目標為在大素數有限域 𝔽p 上原生運算、用 low-degree polynomial 可表達的 hash function。對比傳統 hash（SHA-256 / Keccak）以 bit-level boolean 運算為主，編碼到 SNARK 需將每個 bit 分解為 𝔽p 元素 ⇒ 大量 boolean constraint b·(1−b)=0。代表族：Poseidon、Rescue、Griffin、Reinforced Concrete；共通特徵為 S-box 用 xα（α 小常數，如 3、5、7）+ MDS 擴散。
  • SHA-256 arithmetised：~25,000 約束 / hash（boolean 拆位 + carry chain）
  • Poseidon ~64 約束 / row × 65 rows/permutation — 代數層運算，直接在 𝔽p 上
  • Zcash / Filecoin / Polygon zkEVM 生產驗證，業界事實標準
  • 兩者差約 兩個數量級 — 選擇決定整個系統 feasibility

PF1 Merkle (Part A + A') 佔總 rows ~85%。SHA-arithmetised 代價下整個電路直接爆表。

Sponge37 / 87

Sponge 結構

Poseidon 是固定長度的 permutation（t=3）。任意長度輸入透過 sponge 封裝：

  • State vector：t=3 個 field element，分為 rate=2 + capacity=1
  • Absorb：把輸入 chunk 加到 rate 部分 → 跑一次 permutation
  • Squeeze：從 rate 讀 output chunk → 跑一次 permutation

PF1 使用：Position leaf（5 欄位 → 3 次 permutation）、Market Data leaf（9 欄位 → 5 次 permutation，zero-padding on odd tail）。Merkle 每層 hash(left, right) = 1 次 permutation。

"In the absorbing phase, the r-bit input message blocks are XORed into the first r bits of the state, interleaved with applications of the function f. When all message blocks are processed, the sponge construction switches to the squeezing phase."

— Bertoni, Daemen, Peeters, Van Assche, Cryptographic Sponge Functions (CSF v0.1, 2011), §1.2 [8]

P(x) = c_n xⁿ + ... + c_0，巢狀寫法：P(x) = c_0 + x(c_1 + x(c_2 + ...))

// 每一步約束：acc_next · S + rem = acc · x + coeff

meta.create_gate("horner_per_step", |meta| {
  let q    = meta.query_selector(q_horner);

Q1：為什麼不用 lookup table 直接做 exp/ln/erf？

  • Lookup 的代價：每個函數都要建 65K+ 專用表，setup / verification key 隨函數改
  • Poly 的優勢：一套 256-entry ByteTable 通吃所有函數，換函數只換 Remez 係數（< 200 Fr），circuit 不變
  • Scale 可控：精度要求提高，lookup table 指數爆炸；多項式 d 加幾項就夠
  • 不 claim 更快 prove time — claim 的是工具鏈簡化 + function-independent circuit

細節：Part V slide 46-48。

Q2A2 / A10

Q2：N(d₁) 實測 7 ULP 超過 claim 6.04 ULP 16%，怎麼說？

  • 6.04 是 per-transcendental conservative bound：exp 2.81 + erf 2.81 + rounding = 5.62 ≈ 6.04
  • 實測 7 ULP 是 single erf call 後的 bit-for-bit 量測，落在 conservative bound ±1 ULP 之內（rounding 方向影響）
  • d₁ 15 ULP 是 多步鏈（ln + FpMul×3 + FpDiv + FpAdd），per-transcendental claim 本來就不直接適用；鏈式累積理論上界 57.6 ULP，實測 15 = 26% of bound，合理
  • 合規意義：7 ULP ≈ 2×10⁻¹⁴，離任何合理門檻 τ ≥ 0.01 有 12 個數量級餘裕
Q3A3 / A10

Q3：N=128 持倉太小，真實基金有 1000+ 怎辦？

  • 本論文 scope：single-batch per-fund primitive，N=128 是 Halo2 stub k=18 的 sweet spot（proof 89s / verify 8 ms / 31 KB）
  • Scaling 路徑：recursive SNARK aggregation（Halo2 verifier-in-circuit）或 folding scheme（Nova / SuperNova / HyperNova）
  • 分區（sharding）：N=1024 = 8 × N=128 分片 + aggregator，proof 線性疊但 verify 還是常數
  • Recursive aggregation 不在論文主 contribution，屬 Future Work
Q4A4 / A10

Q4：為什麼選 Halo2 而不是 Plonky2 / Jolt / STARK？

  • Universal SRS：MDT 每日更新，每日換 setup 不可行；Halo2 KZG 是 universal
  • Lookup + hi-degree gates 都要：ByteTable lookup（83 lookups）+ degree 6 gates（Poseidon, subgroup check）— Plonkish 是最省的
  • Proof size & verify：31 KB / 8 ms — STARK 是 O(log² rows)，小 portfolio 吃虧
  • 工程成熟度：halo2_proofs v0.3.0 穩定，Jolt 還在快速演化 API 常動

Q6A6 / A10

Q6：Column-packing 讓 Δ 變免費，是 planner 機制還是你的設計？

  • 機制層：Halo2 `SimpleFloorPlanner` 做水平 column packing — 這是既有機制
  • 本論文貢獻：(1) 實測發現 Merkle column 是 row bottleneck（T1 獨立 harness）(2) 驗證 Δ pipeline 61 rows/pos 可塞進同 row 範圍對 total rows 貢獻 0
  • Caveat：依賴當前 layout（94 advice cols）；若換 V1Planner 或 column 減少則不成立
  • 「rows-only 免費」≠ prove time 免費 — 94 cols × N rows 仍 scale
Q7A7 / A10

Q7：Proof size N-invariant 31,872 B，是你做的還是 KZG 內建？

  • KZG + SHPLONK 內建：proof size = O(columns)，非 O(rows)
  • 本論文貢獻：(1) 選擇 SHPLONK（vs 其他 multi-open scheme）(2) 實測跨 N=8/32/128 精確 31,872 bytes 相同，bit-for-bit 驗證 N-invariant
  • 對大 portfolio 是 non-trivial 特性：STARK/FRI 會隨 N 增長 O(log² rows)，KZG 不會
  • 不 claim 發明，claim 是「實測驗證 + 正確引用」
Q8A8 / A10

Q8：4 個 external trust assumption，還算 zero-knowledge 嗎？

  • ZK 性質 ≠ trust model：ZK 三性質（completeness / soundness / zero-knowledge）對 circuit 成立；trust assumption 是 protocol 層
  • 本電路的 ZK：prove Δ ≤ τ 不洩持倉具體 w / K / σ — 這層成立
  • 4 項 out-of-circuit assumption（event_root / position registry / trade completeness / inter-batch ordering）是 操作信任，由 fund administrator / exchange 提供
  • 所有實用 ZK 系統都有 trust layer — ceremony / administrator / oracle — 論文誠實列出即可
Q9A9 / A10

Q9：MDT 每日更新靠 administrator，能去中心化嗎？

  • 當前：fund administrator 每日發布 MDT_root + EdDSA 簽章，LP 驗簽
  • 去中心化路徑：
    • Threshold signature：n 家 data provider（Bloomberg / Refinitiv / Xignite）m-of-n 共簽
    • Price oracle（Chainlink-style）：各 provider 個別簽，LP 取 median
    • ZK commitment scheme：data feed 層加 ZK proof 來源正確性
  • 這屬 Open question，論文列為 Future Work
Q10A10 / A10

Q10：能擴展到 Gamma / Vega / Theta 嗎？

  • 共用 kernel 復用率高：Delta 用 N(d₁)，Vega 用 φ(d₁) = exp(−d₁²/2)，Theta/Gamma 用 φ(d₁) — 全共用 exp/ln/erf
  • 估工作量：Vega 加 ~80 rows/pos（exp 一次 + FpMul）、Gamma 加 ~30 rows/pos（已有 φ 重用）、Theta 加 ~120 rows/pos（再算 T 相關項）
  • 四個 Greeks 共用的電路總 rows ≈ 1.5× 當前 — 可行
  • 論文專注 Delta 是為了深度，不為廣度
