第九章：查找論證（Lookup Arguments）

如何證明某一欄中的每個值皆屬於一張預先計算的表格 — 這是在零知識電路中實現高效範圍檢查、位元運算及非線性函數求值的關鍵原語。

關鍵定義

Lookup Argument 查找論證
查找論證是一種協議，用以證明「查詢」欄 $f$ 中的每個元素都包含在「表格」欄 $t$ 之中：$\forall\, i,\; f_i \in \{t_0, t_1, \ldots, t_{N-1}\}$。證明者無需在算術約束中直接計算函數，而是透過預先計算的合法輸入-輸出配對表格，證明該輸出確實來自表格查找。
Lookup Table 查找表
查找表是一欄預先計算的固定值，作為查找論證的參考集合。例如：用於位元組級運算的 256 個條目表 ($\{0, 1, \ldots, 255\}$)、用於 16 位元範圍檢查的 $2^{16}$ 條目表，或是將 $(x, y) \mapsto x \oplus y$ 映射的多欄 XOR 表。
Plookup Plookup
Plookup（Gabizon 與 Williamson, 2020）是開創性的查找論證協議。它採用排序合併方法：將 $f \cup t$ 的聯集排序，然後透過大乘積論證檢查相鄰關係。其核心恆等式確保排序列表中的連續元素要麼重複，要麼對應表格中相鄰的有效條目。
Halo2 Lookup Halo2 查找
Halo2 的內建查找機制。證明者將查詢值排列以匹配排序後的表格順序，並構造置換欄 $(A', S')$。大乘積論證驗證 $A'$ 和 $S'$ 分別是輸入與表格的有效置換，加上相鄰性檢查確保每個 $A'_i$ 等於 $S'_i$ 或 $A'_{i-1}$。
LogUp LogUp
LogUp（對數導數查找）以有理函數之和取代大乘積方法：$\sum_{j} \frac{1}{\gamma + f_j} = \sum_{i} \frac{m_i}{\gamma + t_i}$，其中 $m_i$ 是表格條目 $t_i$ 在查詢中出現的多重度。此方法無需排序，並支援跨子電路的表格共享。
MVLookup MVLookup
MVLookup（多變量查找）將 LogUp 擴展到多欄查找。它不將多欄條目編碼為單一隨機線性組合，而是原生處理每一欄，使得表格具有多個輸入/輸出欄時，能產生更高效的證明。
Multiplicity 多重度
表格條目 $t_i$ 的多重度 $m_i$ 是 $t_i$ 在查詢欄 $f$ 中出現的次數。在 LogUp 中，多重度作為獨立的一欄被顯式追蹤：證明者對每個表格列承諾 $m_i$，協議驗證其一致性。在 Plookup 中，多重度隱含在排序合併之中。
Sorted Concatenation 排序合併
在 Plookup 中，排序合併是將查詢欄 $f$ 和表格欄 $t$ 合併為單一排序序列 $s$ 的過程。排序遵循表格順序：相等的值被分組在一起，連續的不同值對應表格中相鄰的條目。檢查 $s$ 是否具有此結構即可證明 $f \subseteq t$。
Table Size Bottleneck 表大小瓶頸
表大小瓶頸是指傳統查找方案的根本限制：表格必須佔據電路中的固定欄位。一張 $2^{16}$ 條目的表格光是表格資料就需要 $2^{16}$ 行，無論實際執行多少次查找。當多個函數各自需要獨立的表格時，行數會急劇膨脹 — 這是查找密集型電路的主要可擴展性障礙。

9.1 為何需要查找論證？

算術化的瓶頸

回顧第四章，算術化將計算轉換為有限域 $\mathbb{F}_r$ 上的多項式約束。約束系統使用加法門和乘法門。對於天然具有代數結構的運算 — 域算術、線性代數、多項式求值 — 這非常有效，但對於非天然代數的運算則變得極為昂貴。

考慮以下運算在算術約束下的表現：

  • 位元 AND（$a \wedge b$）：計算兩個 32 位元值的 AND 需要先將兩者分解為單獨的位元（每個需 32 個約束做位元分解，加上 32 個範圍檢查約束以驗證每個位元為 0 或 1），然後將對應位元相乘（32 個乘法門），最後重新組合結果（32 個帶冪次的加法）。總計：一個 32 位元 AND 約需 160 個以上的約束。
  • 範圍檢查（$x \in [0, 2^k)$）：標準方法將 $x$ 分解為 $k$ 個位元：$x = \sum_{i=0}^{k-1} b_i \cdot 2^i$，每個 $b_i \in \{0,1\}$ 由 $b_i \cdot (1 - b_i) = 0$ 強制。代價為 $k$ 個乘法約束加 $k$ 個加法約束。當 $k = 16$ 時，一次範圍檢查就需要 32 個約束。
  • SHA-256：SHA-256 的一輪涉及 32 位元字上的位元運算（AND、XOR、NOT、旋轉）。每輪需要數百個約束。完整的 SHA-256（64 輪）大約需要 25,000–30,000 個約束。
  • 非線性函數（ReLU、sigmoid、$\exp$）：機器學習中的這些函數在域算術中沒有封閉形式的表達式。用多項式約束來近似它們需要高次多項式和大量的門。

查找的核心思想

查找論證背後的根本洞見很簡單：如果一個函數的定義域是有界的，就預先計算其所有輸出並存入表格。無需在電路內部計算 $f(x)$，只要證明 $(x, f(x))$ 這對值存在於預先計算的表格中即可。

具體範例：透過查找進行範圍檢查

假設我們要證明 $x \in [0, 256)$（即 $x$ 是一個 8 位元值）。不使用查找時，需要將 $x$ 分解為 8 個位元並約束每個位元，代價為 16 個約束。使用查找：

  1. 預先計算一張表格 $t = (0, 1, 2, \ldots, 255)$，共 256 個條目。
  2. 使用一個查找論證證明 $x \in t$。
  3. 完成。一次查找取代了 16 個約束。

查找論證本身有一些開銷（下面會量化），但它在對同一表格的所有查找上是攤銷的。如果電路針對同一張 256 條目的表格執行 10,000 次範圍檢查，每次檢查的邊際成本基本上只是查詢欄中的一行。

成本比較

下表比較了幾種運算在直接實現與使用查找時的約束成本：

運算 直接約束數 使用查找 節省倍數
範圍檢查：$x \in [0, 2^8)$ 16（8 個位元 + 8 個布林檢查） 1 次查找，256 條目表格 16x
範圍檢查：$x \in [0, 2^{16})$ 32 個約束 1 次查找，$2^{16}$ 條目表格 32x

攤銷成本

在傳統方案中，表格本身佔據電路的行數。一張 256 條目的表格消耗 256 行。但如果執行 $n$ 次查找，每次查找的邊際成本約為 $\approx 1$ 行。損益平衡點在於 $n \times \text{直接成本} > n + \text{表格大小}$。對於 256 條目的表格和 16 個約束的範圍檢查：$16n > n + 256 \implies n > 17$。僅需超過 17 次範圍檢查，查找方法就更為划算。

查找在實務中的應用場景

  • 範圍檢查 — 最常見的用途。zkEVM 每個區塊執行數千次範圍檢查。
  • 位元運算 — ZK 電路中 SHA-256、Keccak、AES 所需的 AND、OR、XOR。
  • 雜湊函數 S-box — AES、Poseidon（部分輪次）、Rescue 的 S-box 查找。
  • 指令碼調度 — zkVM 使用查找來驗證指令操作碼的合法性並擷取其語義。
  • 記憶體一致性 — zkVM 使用類似查找的論證來證明 RAM 讀寫的一致性。
  • 機器學習激活函數 — ReLU、sigmoid 等透過量化精度的查找表來求值（見第 9.6 節）。

9.2 Plookup 協議

設定與符號

令 $t = (t_1, t_2, \ldots, t_N)$ 為表格 — 一個固定的、公開已知的域元素序列，代表所有合法值。令 $f = (f_1, f_2, \ldots, f_n)$ 為查詢（或「見證」）— 證明者聲稱全部包含在 $t$ 中的值。我們要證明：

$$\forall\, j \in [n], \quad f_j \in \{t_1, t_2, \ldots, t_N\}$$

核心思想：排序合併

Plookup 的核心洞見是包含關係可以透過排序來檢驗。考慮將 $f$ 和 $t$ 的合併結果按照 $t$ 定義的順序排序。如果每個 $f_j$ 確實在 $t$ 中，則排序後的序列具有特定結構：連續的元素要麼相等，要麼對應 $t$ 中相鄰的條目。

更精確地說，定義 $s = \text{sort}(f \cup t)$，其中「排序」指按 $t$ 中元素的順序排列（$f$ 中的重複值放置在其匹配的表格條目旁邊）。那麼 $f \subseteq t$ 成立，若且唯若對每對連續元素 $(s_i, s_{i+1})$：

  • $s_i = s_{i+1}$（來自查詢的重複值），或
  • $(s_i, s_{i+1})$ 是 $t$ 中連續條目的一對（「合法步進」）。
實例演練：排序合併

令 $t = (1, 4, 7, 10)$（4 個條目的表格）。令 $f = (4, 1, 7, 4)$（查詢）。所有 $f_j \in t$，因此查找應該通過。

合併結果：$f \cup t = (4, 1, 7, 4, 1, 4, 7, 10)$。按表格順序排序：

$s = (1, 1, 4, 4, 4, 7, 7, 10)$

檢查連續對：$(1,1)$ 相等；$(1,4)$ 在 $t$ 中相鄰；$(4,4)$ 相等；$(4,4)$ 相等；$(4,7)$ 在 $t$ 中相鄰；$(7,7)$ 相等；$(7,10)$ 在 $t$ 中相鄰。全部有效。

現在假設 $f = (4, 1, 5, 4)$，其中 $5 \notin t$。排序後：$s = (1, 4, 4, 5, 7, 10, \ldots)$。$(4, 5)$ 這對既不相等也不是 $t$ 中的相鄰條目（$4$ 之後的下一個表格條目是 $7$，不是 $5$）。檢查失敗。

正式協議

Plookup 使用驗證者抽樣的隨機挑戰 $\beta, \gamma$（在非互動式設定中透過 Fiat-Shamir），將排序合併檢查轉換為多項式恆等式。協議步驟如下：

證明者承諾排序合併。 證明者計算 $s = \text{sort}(f \cup t)$（按表格順序排序，長度為 $n + N$），並將 $s$ 作為多項式進行承諾。
驗證者發送隨機挑戰 $\beta, \gamma$。 （實務中透過 Fiat-Shamir 從通訊記錄中導出。）
大乘積檢查。 證明者構造大乘積多項式 $Z(X)$ 並證明以下恆等式成立： $$\prod_{i=1}^{n} \frac{(1+\beta)(\gamma + f_i)(\gamma(1+\beta) + t_i + \beta \cdot t_{i+1})}{\left(\gamma(1+\beta) + s_i + \beta \cdot s_{i+1}\right)\left(\gamma(1+\beta) + s_{n+i} + \beta \cdot s_{n+i+1}\right)} = 1$$
驗證者檢查。 驗證者確認 $Z(\omega^0) = 1$（乘積從 1 開始）、遞推關係在定義域的每個點上成立，以及 $Z(\omega^{n}) = 1$（乘積在末尾回到 1）。
Plookup 健全性

大乘積恆等式成立（在 $\beta, \gamma$ 隨機選擇下以壓倒性機率），若且唯若：

  1. $s$ 是 $f \cup t$ 按表格順序的有效排序，且
  2. 每個 $f_j \in \{t_1, \ldots, t_N\}$。

健全性誤差至多為 $\frac{(n+N)^2}{|\mathbb{F}|}$，對於密碼學級別的域大小而言是可忽略的。

理解大乘積

大乘積公式看起來令人生畏，但直覺上很簡單。分子中的每個因子編碼一次「移動」— 查詢值 $f_i$ 與表格對 $(t_i, t_{i+1})$ 的配對。分母中的每個因子編碼排序序列中對應的「移動」。如果移動的多重集合相匹配，乘積為 1。

$\beta$ 的作用是將連續的配對「綁定」在一起：$\gamma(1+\beta) + s_i + \beta \cdot s_{i+1}$ 實質上將配對 $(s_i, s_{i+1})$ 編碼為單一域元素。因子 $(1+\beta)$ 用於正規化編碼。挑戰值 $\gamma$ 則是平移所有值以避免除以零。

Plookup 的限制：表格在電路內

在 Plookup 中，表格 $t$ 必須作為約束系統中的一欄存在。一張有 $N$ 個條目的表格佔據 $N$ 行。排序序列 $s$ 的長度為 $n + N$，需要一個額外的同等長度的欄。對於一張 $2^{16}$ 條目的表格，這意味著至少有 $2^{16} = 65{,}536$ 行專門用於表格 — 即使只執行一次查找也是如此。這就是後續方案所要解決的表大小瓶頸。

複雜度

  • 證明者時間：$O((n + N) \log(n + N))$，用於排序和基於 FFT 的多項式運算。
  • 證明大小：增加 $s$ 和 $Z$ 的承諾（2 個額外的多項式承諾）。
  • 驗證者時間：與基礎證明系統相同（固定數量的配對檢查或求值檢查）。

9.3 Halo2 的查找機制

API 介面

Halo2 透過 ConstraintSystem 提供高階查找 API。在電路配置期間使用 meta.lookup() 來宣告查找： 

meta.lookup("range_check", |meta| {
    let value = meta.query_advice(value_col, Rotation::cur());
    let table_val = meta.query_fixed(table_col, Rotation::cur());
    vec![(value, table_val)]
});

閉包回傳一個 (input_expression, table_expression) 配對的向量。每對宣告：在查找有效的每一行上，輸入表達式的值必須出現在表格表達式的欄中某處。多個配對可建立多欄查找（輸入值的元組必須匹配表格值的元組）。

輸入與表格表達式

輸入和表格表達式都可以是電路欄位上的任意多項式表達式，而不僅僅是原始的欄位引用。這非常強大：

  • 輸入表達式可以涉及 advice 欄、fixed 欄、instance 欄和選擇器，並透過加法和乘法進行組合。這讓你可以用選擇器「門控」查找，並在查找前轉換值。
  • 表格表達式通常引用包含預計算表格資料的 fixed 欄。但它們也可以是多項式表達式，從而實現動態表格建構。
多欄查找：XOR 表格

要查找兩個 4 位元值的 XOR，你需定義一張具有三欄 $(a, b, a \oplus b)$ 和 256 行（所有 $16 \times 16$ 組合）的表格： 

meta.lookup("xor", |meta| {
    let a = meta.query_advice(a_col, Rotation::cur());
    let b = meta.query_advice(b_col, Rotation::cur());
    let c = meta.query_advice(result_col, Rotation::cur());
    let ta = meta.query_fixed(table_a, Rotation::cur());
    let tb = meta.query_fixed(table_b, Rotation::cur());
    let tc = meta.query_fixed(table_c, Rotation::cur());
    vec![(a, ta), (b, tb), (c, tc)]
});

查找論證證明：對於每一有效行，三元組 $(a, b, c)$ 都作為行 $(t_a, t_b, t_c)$ 出現在表格中。這以單一 3 欄查找取代了約 24 個約束（位元分解、逐位元 XOR、重組）。

底層原理：置換方法

Halo2 的查找基於 Plookup 的一個變體，適配於 PLONK 風格的約束系統。其機制如下：

壓縮多欄查找。 若查找有多個欄位，驗證者提供隨機挑戰 $\theta$，證明者透過隨機線性組合將每個元組壓縮為單一域元素：$\text{input}_i = a_i + \theta \cdot b_i + \theta^2 \cdot c_i + \cdots$，表格側同理。
構造置換欄 $A'$ 和 $S'$。 證明者建立兩個新欄：
  • $A'$ 是（壓縮後的）輸入值的一個置換，按表格順序排列。
  • $S'$ 是（壓縮後的）表格值的一個置換，條目按輸入中的多重度進行複製。
$A'$ 和 $S'$ 具有相同長度（填充至電路大小）。
置換論證。 大乘積論證（類似於 PLONK 的複製約束）驗證 $A'$ 是輸入欄的置換，$S'$ 是表格欄的置換。
相鄰性檢查。 額外的約束驗證對每一行 $i$： $$A'_i = S'_i \quad \text{or} \quad A'_i = A'_{i-1}$$ 這確保每個輸入值等於對應的排序表格值或前一個輸入值（處理重複）。結合置換論證，即可證明包含關係。

大乘積多項式

大乘積 $Z(X)$ 累積置換檢查。使用隨機挑戰 $\beta$ 和 $\gamma$：

$$Z(\omega^{i+1}) = Z(\omega^i) \cdot \frac{(\text{input}_i + \beta)(\text{table}_i + \gamma)}{(A'_i + \beta)(S'_i + \gamma)}$$

從 $Z(\omega^0) = 1$ 開始，若且唯若 $A'$ 是輸入的置換且 $S'$ 是表格的置換時，乘積應在 $Z(\omega^n) = 1$ 回到 1。結合相鄰性約束，即完成查找的證明。

Halo2 查找 vs. 原始 Plookup

關鍵差異在於 Halo2 將置換檢查與相鄰性檢查分開，這與 PLONK 約束系統的整合更為乾淨。原始 Plookup 使用單一大乘積覆蓋排序合併。Halo2 的方法增加了兩個輔助欄（$A'$、$S'$），但避免了單獨的排序合併欄的需求，使其在 Halo2 證明系統中更加模組化。

實務範例：Halo2 中的範圍檢查

一個完整的 Halo2 範圍檢查電路需要三個部分： 

// 1. Define the table (in configure)
let table_col = meta.fixed_column();
meta.lookup("range", |meta| {
    let value = meta.query_advice(value_col, Rotation::cur());
    let sel = meta.query_selector(range_sel);
    let table = meta.query_fixed(table_col, Rotation::cur());
    // When selector is off, lookup input defaults to 0 (which is in the table)
    vec![(sel * value, table)]
});

// 2. Fill the table (in synthesize)
for i in 0..RANGE {
    region.assign_fixed(|| "table", table_col, i, || Value::known(F::from(i as u64)))?;
}

// 3. Assign values to look up (in synthesize)
region.assign_advice(|| "value", value_col, row, || Value::known(witness_value))?;
range_sel.enable(&mut region, row)?;

選擇器 range_sel 門控查找：在選擇器未啟用的行上，輸入表達式求值為 $0$（該值在表格中），因此查找自動通過。這是 Halo2 中條件查找的標準模式。

常見陷阱：遺漏選擇器

如果省略選擇器而直接寫 vec![(value, table)]，查找將套用於電路中的每一行。在 value 包含不相關見證值（不在範圍表格中）的行上，證明將因晦澀的「lookup input not in table」錯誤而失敗。務必使用選擇器門控查找輸入，或使用一個保證存在於表格中的預設值。

9.4 LogUp：現代方法

從乘積到求和

Plookup 和 Halo2 的查找都依賴大乘積 — 跨所有行累積運行乘積的多項式 $Z(X)$。大乘積有一個眾所周知的數值問題：中間值對溢位很敏感，需要仔細管理定義域。更根本地，排序步驟施加了 $O(n \log n)$ 的成本，且難以輕易並行化。

LogUp（Habok, 2022）以對數導數之和取代大乘積。其關鍵數學洞見是：以乘積表達的多項式恆等式與以有理函數之和表達的恆等式之間存在等價性。

對數導數恆等式

$\mathbb{F}$ 上的兩個多重集合 $\{a_1, \ldots, a_n\}$ 和 $\{b_1, \ldots, b_n\}$ 相等，若且唯若：

$$\sum_{i=1}^{n} \frac{1}{X - a_i} = \sum_{j=1}^{n} \frac{1}{X - b_j}$$

作為 $X$ 的有理函數。這源自恆等式：$\frac{d}{dX} \ln \prod_i (X - a_i) = \sum_i \frac{1}{X - a_i}$。兩個多項式相等若且唯若它們的對數導數相等。

應用於查找

對於查找論證（證明 $f \subseteq t$），我們並非擁有相等的多重集合。每個表格條目 $t_i$ 可能在 $f$ 中出現零次或多次。令 $m_i$ 表示多重度 — $t_i$ 在 $f$ 中出現的次數。那麼查找有效若且唯若：

$$\sum_{j=1}^{n} \frac{1}{\gamma + f_j} = \sum_{i=1}^{N} \frac{m_i}{\gamma + t_i}$$

對隨機挑戰 $\gamma$ 成立。左側對每個查詢求和 $1/(\gamma + f_j)$。右側對每個表格條目求和 $m_i/(\gamma + t_i)$，以其出現次數加權。

實例演練：LogUp 檢查

表格 $t = (3, 5, 7, 9)$，查詢 $f = (5, 3, 5, 7, 5)$。多重度：$m = (1, 3, 1, 0)$（條目 $3$ 出現一次，$5$ 出現三次，$7$ 出現一次，$9$ 出現零次）。

左側：$\frac{1}{\gamma+5} + \frac{1}{\gamma+3} + \frac{1}{\gamma+5} + \frac{1}{\gamma+7} + \frac{1}{\gamma+5}$

$= \frac{1}{\gamma+3} + \frac{3}{\gamma+5} + \frac{1}{\gamma+7}$

右側：$\frac{1}{\gamma+3} + \frac{3}{\gamma+5} + \frac{1}{\gamma+7} + \frac{0}{\gamma+9}$

$= \frac{1}{\gamma+3} + \frac{3}{\gamma+5} + \frac{1}{\gamma+7}$

兩側相等。查找有效。

作為運行和的實現

在 PLONK 風格的約束系統中，求和恆等式透過運行和多項式（類似於大乘積，但為加法形式）來強制執行。定義：

$$\phi(\omega^{k+1}) = \phi(\omega^k) + \frac{1}{\gamma + f_{k+1}} - \sum_{\text{table rows } i \text{ at step } k} \frac{m_i}{\gamma + t_i}$$

協議要求 $\phi(\omega^0) = 0$ 且 $\phi(\omega^n) = 0$（運行和從零開始並回到零）。每一步加入查詢的貢獻並減去表格的貢獻。

在實務中，求和分散在電路的各行中。每行從查詢側貢獻一項。表格側可以用多種方式分配 — 每行一個表格條目（若表格可放入電路），或透過單獨的累加器欄。

LogUp 的優勢

大乘積（Plookup / Halo2）

  • 需要排序（$O(n \log n)$）
  • 排序是順序的 — 難以並行化
  • 每個查找論證一個乘積
  • 表格必須作為單一電路中的一欄
  • 多欄：透過隨機線性組合編碼
  • 中間乘積值可能導致累加器溢位

對數導數（LogUp）

  • 無需排序（$O(n)$）
  • 每項互相獨立 — 完全可並行化
  • 對同一表格的所有查找使用一個總和
  • 表格可在子電路間共享
  • 多欄：透過 MVLookup 擴展原生支援
  • 加法在數值上比乘法更穩定

表格共享

LogUp 最重要的實務優勢是表格共享。在 zkEVM 或 zkVM 中，許多子電路（每個操作碼或每個執行軌跡段各一個）需要在同一表格中查找值（例如位元組範圍檢查表或 Keccak S-box 表）。

使用 Plookup 風格的查找時，每個子電路需要自己的表格欄副本，浪費行數。使用 LogUp，單一表格可以服務所有子電路：

$$\underbrace{\sum_{j \in \text{sub-circuit 1}} \frac{1}{\gamma + f_j}}_{\text{來自子電路 1 的查詢}} + \underbrace{\sum_{j \in \text{sub-circuit 2}} \frac{1}{\gamma + f_j}}_{\text{來自子電路 2 的查詢}} + \cdots = \sum_{i=1}^{N} \frac{m_i}{\gamma + t_i}$$

每個子電路產生一個部分運行和。各部分運行和隨後被聚合。表格條目和多重度由中央統一處理。這是 Polygon zkEVM（Type 1 zkEVM）和 SP1（Succinct 的 zkVM）所使用的架構。

Polygon zkEVM 中的 LogUp

Polygon zkEVM 使用 LogUp 在 12 個以上的子電路間共享單一位元組範圍表（$\{0, \ldots, 255\}$）。在 LogUp 之前，每個子電路需要自己的 256 行表格欄。採用 LogUp 之後，表格僅存在一份，每個子電路向共享的運行和貢獻項。這將總電路大小減少了約 30%。

9.5 表大小問題

根本限制

在迄今討論的所有查找方案（Plookup、Halo2 查找，甚至基本的 LogUp）中，表格佔據電路的行。一張有 $N$ 個條目的表格在電路的固定欄佈局中至少需要 $N$ 行。這造成了嚴重的瓶頸：

表格用途 表格大小 $N$ 消耗行數 是否成問題？
8 位元範圍檢查 $2^8 = 256$ 256 沒問題
16 位元範圍檢查 $2^{16} = 65{,}536$ 65,536 可接受
8 位元 XOR（3 欄） $2^{16} = 65{,}536$ 65,536 可接受
16 位元 XOR（3 欄） $2^{32} \approx 4 \times 10^9$ $2^{32}$ 不可行
ReLU（16 位元精度） $2^{16}$ 65,536 單獨可接受
Sigmoid（16 位元精度） $2^{16}$ 65,536 但合在一起...
GELU + Tanh + Exp + ... 每個函數 $k \times 2^{16}$ $k \times 65{,}536$ 爆炸性增長

「爆炸性增長」的場景對ZKML（零知識機器學習）尤為相關，因為神經網路可能使用 5–10 種不同的非線性激活函數。如果每種函數需要自己的 $2^{16}$ 條目表格，電路僅表格就需要 $5 \times 65{,}536 = 327{,}680$ 行 — 尚未包含任何計算門。

為什麼不直接增大電路？

增大電路大小（行數 $n$）會帶來連鎖成本：

  • 證明者時間在基於 FFT 的系統中以 $O(n \log n)$ 增長。將 $n$ 加倍會使證明者時間超過兩倍。
  • 記憶體使用量為每欄 $O(n)$。一個有 $2^{20}$ 行和 50 欄的電路需要 $50 \times 2^{20} \times 32$ 位元組 $\approx$ 1.6 GB 的域元素。
  • 證明大小可能增加（取決於方案）。
  • 定義域限制：FFT 定義域必須是 2 的冪且整除 $r - 1$。對於 BN254，最大定義域為 $2^{28}$。

表大小問題的解決方案

方案 1：LogUp 搭配共享表格

如第 9.4 節所述，LogUp 允許單一表格服務多個子電路，消除重複的表格欄。這雖不會縮小表格本身的大小，但可防止 $k$ 個子電路都需要同一表格時的 $k$ 倍膨脹。

方案 2：表格分解（ByteTable 方法）

與其儲存大型表格，不如將其分解為小型表格並使用組合。例如，16 位元範圍檢查可以分解為兩個 8 位元範圍檢查：

$$x \in [0, 2^{16}) \quad \iff \quad x = 256 \cdot x_{\text{hi}} + x_{\text{lo}}, \quad x_{\text{hi}} \in [0, 256), \quad x_{\text{lo}} \in [0, 256)$$

這使用單一 256 條目的 ByteTable 和兩次查找，取代一張 $2^{16}$ 條目的表格。取捨：更多查找實例（2 次而非 1 次），但表格小得多（256 vs 65,536）。

此方法可推廣：任何 $b$ 位元的範圍檢查都可以用 $\lceil b/8 \rceil$ 次查找到 256 條目的表格中完成，加上 $\lceil b/8 \rceil - 1$ 個約束來驗證分解。

方案 3：電路外表格（cq、Baloo）

近期研究完全消除了表格必須作為電路欄位的要求。在 cq（Eagen, Fiore, Gabizon, 2023）和 Baloo（Zapico 等, 2022）方案中，表格獨立承諾（例如作為表格多項式的 KZG 承諾），查找論證直接針對該承諾工作，無需將表格具體化為電路行。

這意味著一張 $2^{32}$ 條目的表格在電路行數上零成本 — 僅需承諾設定的一次性 $O(N \log N)$ 成本以及每次查找的證明成本。

方案 4：結構化表格與 Lasso

Lasso（Setty, 2023）利用了一個觀察：許多表格具有結構。例如，範圍檢查表 $\{0, 1, \ldots, 2^{16}-1\}$ 不是任意的域元素集合 — 它是所有 16 位元可表示值的集合。位元 XOR 表具有結構 $t[a, b] = a \oplus b$。

Lasso 將結構化表格分解為透過簡單組合函數連接的子表格。以 XOR 為例：

$$\text{XOR}_{16}(a, b) = \text{XOR}_8(a_{\text{hi}}, b_{\text{hi}}) \cdot 2^8 + \text{XOR}_8(a_{\text{lo}}, b_{\text{lo}})$$

每個子表格（$\text{XOR}_8$）僅有 $256 \times 256 = 65{,}536$ 個條目，而非 $2^{32}$。證明者展示分解過程並對較小的子表格執行查找。Lasso 的關鍵洞見在於，證明者只需為實際被存取的表格條目「付費」— 未使用的子表格條目零成本。

可分解表格（Lasso）

一張表格 $T: \{0,1\}^c \to \mathbb{F}$ 是$(k, \alpha)$-可分解的，若存在子表格 $T_1, \ldots, T_k$，每個大小至多 $2^\alpha$，以及一個可用 $O(k)$ 次域運算計算的組合函數 $g$，使得：

$$T(x) = g(T_1(x_1), T_2(x_2), \ldots, T_k(x_k))$$

其中 $x = (x_1, \ldots, x_k)$ 是將輸入分解為 $k$ 個各 $\alpha$ 位元的區塊。

9.6 ZKML 情境下的查找

非線性問題

機器學習模型由線性運算（矩陣乘法、卷積）和非線性激活函數（ReLU、sigmoid、GELU、tanh、softmax）交替的層組成。線性運算在算術電路中是天然的 — 矩陣乘法只是 $\mathbb{F}_r$ 中的加法和乘法。但非線性函數是瓶頸。

考慮神經網路中常用的函數：

  • ReLU：$\text{ReLU}(x) = \max(0, x)$。需要比較運算（$x \geq 0$？），這在域算術中不是原生的。
  • Sigmoid：$\sigma(x) = \frac{1}{1 + e^{-x}}$。涉及指數運算和除法 — 在電路中極為昂貴。
  • GELU：$\text{GELU}(x) = x \cdot \Phi(x)$，其中 $\Phi$ 是高斯累積分佈函數。沒有封閉形式的域表達式。
  • Softmax：$\text{softmax}(x_i) = \frac{e^{x_i}}{\sum_j e^{x_j}}$。多個指數運算加上一個除法。

ZKML 的查找方法

思路很直接：將輸入定義域量化為固定精度（例如 16 位元定點數），在每個量化輸入處預先計算激活函數，並將結果存入查找表。

例如，如果輸入是 $[-32768, 32767]$ 範圍內的 16 位元有號整數，以 8 個小數位元表示定點值，則 ReLU 表格有 $2^{16} = 65{,}536$ 個條目：

$$\text{ReLU\_table}[x] = \begin{cases} x & \text{if } x \geq 0 \\ 0 & \text{if } x < 0 \end{cases}$$

神經網路中的每個激活運算變為對此表格的單次查找。

規模化問題

單一激活函數表格（$2^{16}$ 個條目）是可管理的。但真實模型使用多種非線性函數，每種都需要自己的表格：

模型組件 激活函數 表格大小
隱藏層 ReLU / GELU $2^{16}$
注意力機制（縮放） $\exp$ $2^{16}$
正規化 $1/\sqrt{x}$ $2^{16}$
輸出層 Softmax（$\exp$ + 除法） $2 \times 2^{16}$
嵌入層 LayerNorm（$\text{mean}$, $\text{var}$） $2 \times 2^{16}$

總表格行數：$\approx 7 \times 65{,}536 = 458{,}752$ — 在任何實際計算之前就消耗了近五十萬行。對於一個典型的 Transformer 層有 100,000 次以上的激活呼叫，查找查詢行也佔據了主導地位。

多項式方法替代方案

第十章提出了一種根本不同的方法：與其使用每個函數各自的表格，不如使用單一共享 ByteTable（256 個條目）搭配多項式分解。核心洞見如下：

  1. 將輸入分解為位元組：$x = x_0 + 256 \cdot x_1$（兩次位元組查找）
  2. 用每個位元組上的多項式近似 $f(x) \approx P_0(x_0) + P_1(x_1)$ 或 $P_0(x_0) \cdot P_1(x_1)$
  3. 在電路中原生求值多項式（加法 + 乘法在算術電路中是免費的）

這以單一 256 條目的 ByteTable 取代了所有函數特定的表格，從而解決了表大小爆炸的問題。取捨在於多項式近似會引入微小誤差，必須界定並在模型精確度分析中加以考量。

查找 vs. 多項式：何時使用哪種

使用查找的情況：函數沒有代數結構（例如 AES S-box）、表格很小（8 位元），或需要精確值（密碼學雜湊內部）。

使用多項式方法的情況：函數是平滑的（例如 sigmoid、GELU）、輸入精度較高（16 位元以上），或多個函數共享相同的定義域分解。完整討論見第十章。

9.7 查找方案比較

發展時間線

自 2020 年以來，在 zkEVM 和 zkVM 的需求驅動下，查找論證發展迅速。每一代都解決了前一代的局限性：

2020 年：Plookup（Gabizon, Williamson）。第一個實用的查找論證。使用排序合併 + 大乘積。透過單一多項式恆等式證明包含關係。限制：表格必須在電路內；排序為 $O(n \log n)$ 且是順序的。
2021 年：Halo2 Lookup（Electric Coin Company / ZCash）。將 Plookup 適配於 PLONK 約束系統。使用置換欄 + 相鄰性檢查取代顯式排序合併。隨 Halo2 發佈並成為基於 Halo2 的電路（Scroll、Axiom 等）的事實標準。
2022 年：LogUp（Habok）。以對數導數求和取代大乘積。消除排序（$O(n)$ 證明者）。支援跨電路表格共享。被 Polygon zkEVM 和 SP1 採用。透過 MVLookup 擴展至多欄表格。
2023 年：cq / Baloo（Eagen, Fiore, Gabizon / Zapico 等）。電路外表格 — 表格獨立承諾，不具體化為電路行。證明者成本 $O(N \log N)$（表格大小為 $N$），但電路成本與表格大小無關。
2023 年：Lasso（Setty）。透過分解利用表格結構。對可分解表格，證明者成本為 $O(n \cdot c)$，其中 $c$ 是子表格查找次數，與完整表格大小無關。證明者僅涉及被存取的子表格條目。用於 Jolt zkVM。

方案 年份 證明者成本 表格限制 多欄支援 應用於
Plookup 2020 $O(n \log n)$ 電路內 透過元組編碼 PLONK 變體

詳細取捨分析

證明者成本

主要的成本差異在於基於排序的方案（$O(n \log n)$）與基於求和的方案（$O(n)$）之間。對於 $n = 2^{20}$（約一百萬次查找），$\log n$ 因子為 20，意味著基於排序的方案每次查找多做約 $20 \times$ 的工作。以絕對值而言，在現代硬體上，Plookup 風格證明者的排序步驟對 $n = 2^{20}$ 需要 2–5 秒，而 LogUp 的運行和需要 0.1–0.3 秒。

輔助欄

每種方案需要不同數量的輔助欄（除輸入和表格欄之外），影響電路寬度和記憶體：

  • Plookup：1 個排序合併欄（$n + N$ 個條目）+ 1 個大乘積欄。
  • Halo2 lookup：2 個置換欄（$A'$、$S'$）+ 1 個大乘積欄。
  • LogUp：1 個多重度欄 + 1 個運行和欄。無排序欄。
  • Lasso：$c$ 個子表格查找欄 + $c$ 個多重度欄 + 運行和。

驗證者成本

所有方案都為驗證者增加固定數量的求值檢查。方案間的驗證者成本差異可忽略 — 主要取決於基礎證明系統的驗證成本（例如 KZG 配對檢查或 IPA 內積檢查）。在所有情況下，驗證者每個查找論證執行 $O(1)$ 的額外工作。

天下沒有白吃的午餐

每種方案針對不同的瓶頸進行最佳化。LogUp 消除了排序但要求證明者知道多重度。cq 將表格從電路中移除但有更高的設定成本。Lasso 利用結構但僅適用於可分解的表格。根據應用場景選擇方案：

  • 具有中等大小表格的通用 ZK 電路：Halo2 lookup 或 LogUp
  • 具有共享表格的 zkEVM / zkVM：LogUp
  • 巨大的無結構表格：cq
  • 結構化表格（範圍、XOR、位移）：Lasso

9.8 實務考量

選擇適當的表格粒度

表格大小的選擇涉及表格開銷與查找次數之間的根本取捨：

大型表格（例如 $2^{16}$ 條目）

  • 每個 16 位元運算一次查找
  • 表格開銷高（65,536 行）
  • 最適合：對同一表格有大量查找
  • 損益平衡：~4000 次查找

小型表格 + 分解（例如 $2^8$ 條目）

  • 每個 16 位元運算兩次查找 + 1 個組合約束
  • 表格開銷低（256 行）
  • 最適合：查找次數較少或有多種函數類型
  • 對混合工作負載始終高效
定量比較

假設我們需要對 16 位元值進行 $n$ 次範圍檢查。比較兩種方法：

方法 A：一張 $2^{16}$ 條目的表格。總行數：$65{,}536 + n$（表格 + 查詢）。

方法 B：一張 $2^8$ 條目的表格，每次檢查兩次查找。總行數：$256 + 2n$（表格 + 每次檢查 2 個查詢，加上 $n$ 個分解約束）。

損益平衡：$65{,}536 + n = 256 + 3n \implies n \approx 32{,}640$。低於約 32,640 次檢查時，小型表格方法更高效。超過該數則大型表格勝出。

在實務中，小型表格方法往往勝出，因為：(1) 256 條目的表格可與其他運算共享（位元組範圍檢查、XOR 位元組等），因此 256 行的開銷已經支付；(2) 電路通常有多種類型的查找，單一共享 ByteTable 優於多張大型表格。

選擇器門控查找

在 Halo2 中，查找套用於電路的每一行（因為約束是在整個定義域上的多項式恆等式）。要使查找有條件，你必須用選擇器門控輸入表達式： 

// WRONG: applies to every row
vec![(value, table)]

// CORRECT: applies only where selector is active
// When sel = 0, input expression = 0, which must be in the table
vec![(sel * value, table)]

// ALSO CORRECT: use Expression::Constant as default
// sel * value + (1 - sel) * default_table_value
let default = Expression::Constant(F::ZERO);
vec![(sel.clone() * value + (Expression::Constant(F::ONE) - sel) * default, table)]

第二種方法更安全，因為它顯式控制非活動行上的預設值，而不是依賴 $0$ 存在於表格中。

處理「找不到」錯誤

當查找證明在開發過程中失敗時，錯誤訊息通常會顯示「lookup input not in table」及行索引。常見原因包括：

  • 遺漏選擇器：查找在不應觸發的行上觸發了（見上文）。
  • 表格未填充：包含表格的 fixed 欄在 synthesize 期間未被填入。
  • 表格範圍的差一錯誤：表格有 $\{0, \ldots, N-1\}$ 的條目，但查詢值為 $N$。
  • 錯誤的欄位：輸入表達式引用了錯誤的 advice 欄。
  • 旋轉錯誤：使用 Rotation::next() 而非 Rotation::cur()，導致查找讀取了錯誤行的資料。
  • 表達式求值為非預期值：涉及多個欄位的多項式表達式因邏輯錯誤而求值為不在表格中的值。

效能調校

對於具有大量查找的生產電路，考慮以下最佳化：

  • 批次查找：將共享同一表格的多個查找論證分組。在 Halo2 中，每個查找論證增加一個大乘積欄。合併相關查找可減少輔助欄數量。
  • 表格重用：如果多個運算可以使用同一表格（例如範圍檢查和位元組分解都使用位元組表），則定義一次表格並共享。
  • 表格欄打包：如果電路的行數多於表格所需，未使用的表格行就被浪費。考慮使用「表格 ID」欄將多個小型表格打包到同一 fixed 欄中加以區分。
  • 動態表格：在某些應用中，表格條目取決於見證（例如 zkVM 中的記憶體表）。Halo2 透過「動態查找」支援這種方式，其中表格表達式引用 advice 欄。成本較高（證明者必須將兩側都視為非固定），但支援重要的使用場景。
與置換論證的關聯

查找論證和置換論證（第七章，PLONK 複製約束）密切相關。置換論證證明兩欄是彼此的重排列。查找論證證明一欄的值是另一欄的子集（允許重複）。事實上，兩者都可以統一在「多重集合檢查」的框架下 — 置換論證是多重集合具有相同大小和相同多重度分佈的特例。

9.9 進階：向量查找與 RAM

向量查找

向量查找（或多欄查找）證明一個值的元組 $(a, b, c)$ 作為多欄表格中的一行存在，而非僅僅單一值。這對於編碼多變量函數至關重要：

$$\forall\, i, \quad (a_i, b_i, c_i) \in \{(t_1^{(1)}, t_1^{(2)}, t_1^{(3)}), \ldots, (t_N^{(1)}, t_N^{(2)}, t_N^{(3)})\}$$

在 Plookup 和 Halo2 中，向量查找透過隨機線性組合壓縮元組來實現：使用挑戰 $\theta$，將 $(a, b, c)$ 編碼為 $a + \theta b + \theta^2 c$，然後對編碼值執行單欄查找。Schwartz-Zippel 引理保證，如果元組不匹配，壓縮後的值將以壓倒性機率不同。

在 LogUp / MVLookup 中，向量查找可以原生處理而無需壓縮，透過擴展對數導數公式以操作欄值的隨機線性組合。

記憶體檢查作為查找

zkVM 需要證明記憶體讀取返回正確的值 — 即最近寫入該地址的值。這是查找的一種形式：「讀取」操作在「記憶體狀態」表格中查找一個 (address, value) 配對。

標準方法使用基於多重集合相等的讀/寫記憶體檢查論證。核心恆等式為：

$$\text{init\_memory} \cup \text{writes} = \text{reads} \cup \text{final\_memory}$$

作為多重集合。這表明：初始記憶體狀態加上所有寫入等於所有讀取加上最終記憶體狀態。這正是多重集合相等檢查，可透過 LogUp 風格的求和或大乘積來實現。

記憶體檢查與查找論證之間的這種關聯，解釋了為何相同的代數機制（大乘積、對數導數）同時出現在兩個場景中。SP1 和 Jolt 等系統將兩者統一在單一的 LogUp / Lasso 框架下。

離線記憶體檢查（Blum 等人）

證明者維護兩個多重集合：$W$（寫入集合：所有寫入的 $(addr, val, timestamp)$ 三元組）和 $R$（讀取集合：所有讀取的 $(addr, val, timestamp)$ 三元組）。記憶體一致性成立若且唯若，對每個讀取 $(addr, v, t_r)$，存在一個寫入 $(addr, v, t_w)$，滿足 $t_w \leq t_r$ 且 $addr$ 在此期間沒有被插入寫入。這透過涉及時間戳的多重集合恆等式來驗證。

查找奇異點

LogUp 中一個微妙的問題是分母 $\gamma + f_j$ 的處理。如果對某個查詢值 $\gamma = -f_j$，項 $1/(\gamma + f_j)$ 就是未定義的（除以零）。由於 $\gamma$ 是在證明者承諾 $f$ 之後隨機選取的，$\gamma = -f_j$ 對任何特定 $j$ 的機率為 $1/|\mathbb{F}| \approx 2^{-254}$ — 可忽略不計。

然而，實現必須處理這種邊界情況。標準方法是檢查所有 $j$ 的 $\gamma + f_j \neq 0$（以壓倒性機率成立），若檢查失敗則中止。在實務中，這永遠不會觸發。

展望未來

查找論證持續是零知識證明研究中最活躍的領域之一。關鍵的開放方向包括：

  • 批次攤銷：當數千次查找共享同一表格時，降低每次查找的成本。
  • 遞迴查找：在遞迴證明組合中使用查找論證，其中表格是外部電路的一部分。
  • 硬體加速：排序（Plookup）和多重度計數（LogUp）是受益於 GPU / FPGA 加速的瓶頸。
  • 最優查找：尋找查找論證的資訊理論最優成本 — 對任意表格是否可以實現 $O(n)$ 且不依賴表格大小？

對本研究的意義

查找表：論文要解決的核心問題

查找論證是論文技術的關鍵組件，同時也是論文要克服的瓶頸：

  • ByteTable 依賴查找：我們的 range check（截斷驗證中的 $0 \leq r < S^d$）通過 byte decomposition 實現——每個 byte 需要一次 lookup 到 256 行的 ByteTable。
  • 表大小瓶頸是核心問題：傳統方法用 $O(S)$ 行的大型查找表直接存儲函數值。當 $S = 2^{16} = 65536$ 時，查找表本身就佔用了 65536 行——這對多數實際電路來說太大了。
  • 多項式方法的突破：我們用 degree-$d$ 的 Minimax 多項式計算函數值，而非查找。查找表只用於 byte-level 的 range check（256 行，與精度無關）。
方法查找表大小精度提升時函數無關？
直接查找表 $O(S) = O(2^{pp})$ 行 表指數增長（$pp: 16 \to 20$ 意味著 $65536 \to 1048576$ 行） ✗ 每個函數一張表
多項式方法 + ByteTable 256 行（固定） 不變——精度由多項式次數控制 ✓ 同一電路適用所有函數
這就是「精度可擴展性」

傳統查找方法中，精度（$pp$）和電路大小強耦合——提高精度必然增加表大小。 多項式方法將兩者解耦：精度由多項式次數 $d$ 和 $pp$ 共同控制，而查找表大小恆為 256。 這種精度可擴展性（Precision Scalability）是論文的核心貢獻之一。
