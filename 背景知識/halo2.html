第八章：Halo2

Zcash 的 PLONKish 證明系統 — 透過 IPA 承諾實現透明設置、自定義閘、查找論證、 基於區域的電路佈局，以及利用橢圓曲線循環進行遞迴證明組合。

關鍵定義

Halo2 Halo2
Zcash 的 PLONKish 證明系統，建立在 Inner Product Argument (IPA) 多項式承諾方案之上，不需要可信設置。Halo2 在核心 PLONK 協議基礎上擴展了三大功能： 自定義閘（任意多項式約束）、查找論證（證明值存在於預計算表中）， 以及基於區域的電路佈局（透過晶片實現模組化電路組合）。 它是原始 Halo 方案（Bowe、Grigg、Hopwood，2019 年）的後繼者，該方案透過對 IPA 承諾使用「累積方案」 而非直接驗證，首次實現了無需可信設置的遞迴證明組合。Halo2 是驅動 Zcash Orchard 隱蔽池的證明系統。
Halo2 Domain Halo2 域
電路定義所在的求值域： $H = \{\omega^0, \omega^1, \ldots, \omega^{2^k - 1}\}$，其中 $\omega$ 是純量場中 $2^k$ 次本原單位根。參數 $k$ 的選取需確保 $2^k \geq$ 電路所需的行數（包含盲化行）。此域決定了最大電路規模並影響證明時間。在程式碼中： EvaluationDomain::new(k, j)，其中 $k$ 是以 $\log_2$ 表示的行數，$j$ 是擴展域倍增器的次數。 擴展域 $H' = \{\omega'^0, \omega'^1, \ldots, \omega'^{2^{k+j} - 1}\}$ 用於求值商多項式， 因為閘約束使其次數高於 $2^k - 1$。
Region 區域
電路表中一塊連續的矩形區域。區域內的單元格可透過相對偏移量互相引用， 使區域與位置無關。區域由佈局器分配至特定行。一個電路可包含許多區域 — 每個區域通常由一個晶片建立，以實現一項邏輯操作 （例如一次乘法、一輪雜湊運算）。核心優勢在於：晶片以相對定址定義區域， 佈局器將其解析為絕對位置，從而實現模組化電路組合。
Column Types 欄位類型
Halo2 電路表中的每個單元格都屬於以下三種欄位類型之一：
  • Advice Column (建議欄)：私密見證值，由證明者為每個證明設定。這些是證明者知道但驗證者看不到的「秘密輸入」。每個證明可以有不同的建議值。使用多項式承諾進行提交。
  • Fixed Column (固定欄)：在電路配置階段確定的常數。固定欄是驗證金鑰的一部分 — 對同一電路的每個證明都相同。選擇器是固定欄的特殊情況。
  • Instance Column (實例欄)：證明者和驗證者都已知的公開輸入。例如，原像證明中的雜湊輸出，或成員資格證明中的 Merkle 根。實例值同樣由驗證者提供。
Selector 選擇器
一種值限制為 $\{0, 1\}$ 的固定欄，用於在特定行啟用或停用閘。 當 $s_i = 1$ 時（位於第 $i$ 行），關聯的閘約束被強制執行；當 $s_i = 0$ 時， 閘多項式乘以零，從而平凡地滿足約束。Halo2 執行選擇器合併作為最佳化： 多個永遠不會同時啟用的選擇器可以打包到單一固定欄中，減少欄數從而縮小證明大小。
Custom Gate 自定義閘
對電路表中單元格的任意多項式約束。與標準 PLONK 閘 （限於 $a \cdot q_L + b \cdot q_R + a \cdot b \cdot q_M + c \cdot q_O + q_C = 0$）不同， 自定義閘可以引用不同旋轉量（行）和不同欄的單元格， 形成任意的多變量多項式表達式。一般形式： $$\sum_i s_i \cdot \prod_j \text{cell}_{ij}^{e_{ij}} = 0$$ 閘多項式的次數直接影響證明成本：更高的次數需要更大的擴展求值域。
Rotation 旋轉
在閘定義中查詢單元格時使用的行偏移量。 Rotation::cur() 指當前行（偏移 0）， Rotation::next() 指下一行（+1）， Rotation::prev() 指上一行（−1）。 也可使用任意偏移量，如 Rotation(3)。旋轉以域大小 $2^k$ 為模進行環繞， 因為約束多項式在循環群 $H$ 上求值。任何閘中使用的最大旋轉偏移量會影響盲化因子。
Layouter 佈局器
將區域分配到電路表中特定行的演算法。 SimpleFloorPlanner 是預設佈局器 — 它按照分配順序， 從上到下、逐欄地排列區域。佈局器負責將區域內的相對偏移解析為絕對行位置。 良好的佈局器可以最小化浪費的行數（從而最小化 $k$），降低證明時間。
Chip 晶片
一個可重用的電路元件，實現特定的功能。 例如 ECC chip（橢圓曲線運算）、Poseidon chip （Poseidon 雜湊）或範圍檢查晶片。一個晶片封裝三個部分： (1) 配置 — 需要哪些欄和閘， (2) 閘定義 — 多項式約束， (3) 賦值邏輯 — 如何為給定輸入填充單元格。 晶片可以組合：高層晶片可以使用低層晶片。
Proof Transcript 證明抄本
證明協議中交換的所有承諾和挑戰的有序記錄。 在非互動式（Fiat-Shamir）設定中，抄本是一個基於雜湊的物件： 每個承諾被吸收進抄本，每個挑戰從中擠出。 Halo2 使用 Blake2b 雜湊函數進行抄本雜湊。抄本確保所有挑戰都是從先前的協議訊息中 確定性地推導而來，防止證明者選擇有利的挑戰。
Challenge API 挑戰 API
Halo2 的一項功能（在後續版本中加入），允許電路使用驗證者挑戰 — 透過 Fiat-Shamir 推導的值 — 作為電路內部的建議值。 這使得需要電路操作隨機挑戰的協議成為可能，例如 Shuffle 論證和自定義查找方案。 挑戰在 configure() 期間請求，並以 Expression::Challenge 的形式在閘中可用。
Cost Model 成本模型
Halo2 中的證明大小和驗證時間取決於多個因素： (1) 建議欄的數量（每個會在證明中新增一個承諾）， (2) 固定欄的數量（驗證金鑰的一部分）， (3) 實例欄的數量， (4) 閘多項式的次數（決定擴展域大小）， (5) 查找的數量（每個會新增排列多項式承諾）， (6) 排列欄的數量（用於複製約束）。 IPA 開啟證明新增 $O(\log n)$ 個群元素。總證明大小大約為 $(\text{num\_advice} + \text{num\_lookups} \cdot 2 + \text{num\_perm\_products}) \cdot 32 + O(\log n) \cdot 64$ 位元組。
Blinding Factor 盲化因子
在求值域末尾保留給隨機盲化值的行數。 這些行填入隨機場元素，以確保零知識性（防止驗證者從多項式求值中得知見證值）。盲化因子為： $$\text{blinding\_factors} = \text{max\_query\_degree} + 1$$ 其中 max_query_degree 是所有閘和查找中使用的最大不同旋轉數。盲化行不可用於電路邏輯。
Minimum Rows 最小行數
電路必須至少有 $\text{blinding\_factors} + 1$ 行。這決定了 $k$ 的最小值：即使電路本身使用零行，我們仍需要 $2^k \geq \text{blinding\_factors} + 1$。 實際上，最小 $k$ 由以下條件決定： $2^k \geq \max(\text{circuit\_rows}, \text{blinding\_factors} + 1)$。 這同時影響求值域的大小和證明時間。

8.1 Halo2 架構概覽

Halo2 是一個 PLONKish 證明系統：它使用與 PLONK 相同的核心框架 （在結構化表上進行多項式約束，並透過排列論證強制執行複製約束），但在三個關鍵方向上進行了擴展：

  • IPA 承諾 — 以 Inner Product Argument 取代 KZG，消除了對可信設置儀式的需求。任何人都可以僅使用公開已知的參數從頭驗證證明。
  • 自定義閘 — 對不同旋轉量和欄位的單元格進行任意多項式約束，遠超 PLONK 固定的 $q_L a + q_R b + q_M ab + q_O c + q_C = 0$ 閘。
  • 查找論證 — 內建支援證明某值存在於預計算表中，對範圍檢查、位元運算和雜湊函數至關重要。
  • 基於區域的佈局 — 電路由模組化的「晶片」組成，每個晶片定義自己的區域。佈局器將這些區域解析為具體的表分配。
注意：PLONKish 不等於 PLONK

「PLONKish」指的是一系列算術化方案，它們共享 PLONK 基於表的結構和排列論證， 但在承諾方案、閘類型和額外論證方面可能有所不同。Halo2、HyperPlonk 和各種 zkEVM 系統都屬於 PLONKish。

8.2 電路 API：程式設計模型

Halo2 的電路 API 以 Circuit trait 為核心，每個電路都必須實作此 trait。 該 trait 有兩個核心方法和兩個關聯型別：

Circuit Trait

  • type Config — 一個結構體，持有電路所需的所有欄和閘的引用。
  • type FloorPlanner — 佈局策略（通常為 SimpleFloorPlanner）。
  • fn configure(meta: &mut ConstraintSystem<F>) -> Self::Config — 宣告欄位、定義閘、註冊查找。只呼叫一次以建立電路結構。
  • fn synthesize(&self, config: Self::Config, layouter: impl Layouter<F>) -> Result<(), Error> — 在區域內為單元格分配具體值。每個證明呼叫一次。
關鍵見解：結構與值的分離

configure() 定義電路長什麼樣（欄位、閘、複製約束）。 synthesize() 為特定證明填入具體值。 相同的配置可重複用於具有不同輸入的多個證明。這種分離使得驗證金鑰保持靜態。

ConstraintSystem API

在 configure() 內部，ConstraintSystem 提供了宣告電路結構的方法：

  • meta.advice_column() — 建立新的建議欄
  • meta.fixed_column() — 建立新的固定欄
  • meta.instance_column() — 建立新的實例欄
  • meta.selector() — 建立新的選擇器（最佳化的固定欄）
  • meta.create_gate(name, |meta| { ... }) — 定義自定義閘
  • meta.lookup(name, |meta| { ... }) — 註冊查找論證
  • meta.enable_equality(column) — 允許對欄進行複製約束

完整範例：乘法電路

以下範例實作了一個證明 $a \times b = c$ 的電路。它展示了欄位、選擇器、自定義閘、區域分配和複製約束。 

use halo2_proofs::{
    arithmetic::Field,
    circuit::{AssignedCell, Chip, Layouter, Region, SimpleFloorPlanner, Value},
    plonk::{Advice, Column, ConstraintSystem, Error, Fixed, Instance, Selector},
    poly::Rotation,
};

#[derive(Clone, Debug)]
struct MulConfig {
    a: Column<Advice>,
    b: Column<Advice>,
    c: Column<Advice>,
    s_mul: Selector,
    instance: Column<Instance>,
}

struct MulCircuit<F: Field> {
    a: Value<F>,
    b: Value<F>,
}

impl<F: Field> Circuit<F> for MulCircuit<F> {
    type Config = MulConfig;
    type FloorPlanner = SimpleFloorPlanner;

    fn without_witnesses(&self) -> Self {
        MulCircuit { a: Value::unknown(), b: Value::unknown() }
    }

    fn configure(meta: &mut ConstraintSystem<F>) -> Self::Config {
        // Declare columns
        let a = meta.advice_column();
        let b = meta.advice_column();
        let c = meta.advice_column();
        let s_mul = meta.selector();
        let instance = meta.instance_column();

        // Enable equality for copy constraints
        meta.enable_equality(a);
        meta.enable_equality(b);
        meta.enable_equality(c);
        meta.enable_equality(instance);

        // Define the multiplication gate:
        //   s_mul * (a * b - c) = 0
        // When s_mul = 1: enforces a * b = c
        // When s_mul = 0: no constraint (gate is "off")
        meta.create_gate("mul", |meta| {
            let a = meta.query_advice(a, Rotation::cur());
            let b = meta.query_advice(b, Rotation::cur());
            let c = meta.query_advice(c, Rotation::cur());
            let s = meta.query_selector(s_mul);
            vec![s * (a * b - c)]
        });

        MulConfig { a, b, c, s_mul, instance }
    }

    fn synthesize(
        &self,
        config: Self::Config,
        mut layouter: impl Layouter<F>,
    ) -> Result<(), Error> {
        // Assign a region for the multiplication
        let c_cell = layouter.assign_region(
            || "mul_region",
            |mut region: Region<'_, F>| {
                // Enable the selector at row 0
                config.s_mul.enable(&mut region, 0)?;

                // Assign advice values
                let a_cell = region.assign_advice(
                    || "a", config.a, 0, || self.a,
                )?;
                let b_cell = region.assign_advice(
                    || "b", config.b, 0, || self.b,
                )?;
                let c_val = self.a.and_then(|a| self.b.map(|b| a * b));
                let c_cell = region.assign_advice(
                    || "c", config.c, 0, || c_val,
                )?;

                Ok(c_cell)
            },
        )?;

        // Constrain output to equal public instance
        layouter.constrain_instance(c_cell.cell(), config.instance, 0)?;

        Ok(())
    }
}

假設證明者設定 $a = 3$，$b = 7$。電路填入：

在第 0 行，閘檢查：$1 \cdot (3 \cdot 7 - 21) = 0$。複製約束將第 0 行的 $c$ 綁定到實例欄的值 21。

8.3 自定義閘詳解

自定義閘是 Halo2 的核心表達能力。每個閘都定義為一個回傳 Vec<Expression<F>> 的閉包。向量中的每個表達式必須在閘的選擇器啟用的每一行都求值為零。

表達式類型

Expression<F> 由以下原語構建：

  • Constant(F) — 場元素字面值
  • Selector(Selector) — 查詢當前行的選擇器
  • Fixed(FixedQuery) — 以旋轉量查詢固定欄
  • Advice(AdviceQuery) — 以旋轉量查詢建議欄
  • Instance(InstanceQuery) — 以旋轉量查詢實例欄
  • Challenge(Challenge) — Fiat-Shamir 挑戰值
  • Negated(Box<Expression>)、Sum(..)、Product(..)、Scaled(..) — 算術組合子

閘的範例

布林約束

強制單元格值為 0 或 1： 

meta.create_gate("boolean", |meta| {
    let a = meta.query_advice(col_a, Rotation::cur());
    let s = meta.query_selector(s_bool);
    // s * a * (1 - a) = 0
    // Satisfied only when a = 0 or a = 1
    vec![s * a.clone() * (Expression::Constant(F::ONE) - a)]
});

閘多項式：$s \cdot a \cdot (1 - a) = 0$。這是 3 次（選擇器乘以 $a$ 的兩個因子）。

範圍檢查（4 位元）

透過位元分解證明一個值在 $[0, 15]$ 範圍內： 

// Advice columns: value, b0, b1, b2, b3
meta.create_gate("range_4bit", |meta| {
    let v = meta.query_advice(value, Rotation::cur());
    let b0 = meta.query_advice(bit0, Rotation::cur());
    let b1 = meta.query_advice(bit1, Rotation::cur());
    let b2 = meta.query_advice(bit2, Rotation::cur());
    let b3 = meta.query_advice(bit3, Rotation::cur());
    let s = meta.query_selector(s_range);
    let one = Expression::Constant(F::ONE);

    vec![
        // Decomposition: v = b0 + 2*b1 + 4*b2 + 8*b3
        s.clone() * (v - (b0.clone()
            + b1.clone() * Expression::Constant(F::from(2))
            + b2.clone() * Expression::Constant(F::from(4))
            + b3.clone() * Expression::Constant(F::from(8)))),
        // Each bit is boolean
        s.clone() * b0.clone() * (one.clone() - b0),
        s.clone() * b1.clone() * (one.clone() - b1),
        s.clone() * b2.clone() * (one.clone() - b2),
        s * b3.clone() * (one - b3),
    ]
});

條件等式

若 cond = 1，強制 $a = b$；若 cond = 0，則無約束： 

meta.create_gate("conditional_eq", |meta| {
    let a = meta.query_advice(col_a, Rotation::cur());
    let b = meta.query_advice(col_b, Rotation::cur());
    let cond = meta.query_advice(col_cond, Rotation::cur());
    let s = meta.query_selector(s_cond);
    // s * cond * (a - b) = 0
    vec![s * cond * (a - b)]
});

跨行閘（Fibonacci）

使用旋轉量強制 $a_{n+2} = a_{n+1} + a_n$： 

meta.create_gate("fibonacci", |meta| {
    let a_cur  = meta.query_advice(col_a, Rotation::cur());   // row i
    let a_next = meta.query_advice(col_a, Rotation::next());  // row i+1
    let a_next2 = meta.query_advice(col_a, Rotation(2));      // row i+2
    let s = meta.query_selector(s_fib);
    // s * (a[i+2] - a[i+1] - a[i]) = 0
    vec![s * (a_next2 - a_next - a_cur)]
});

閘的次數與成本

警告：閘的次數影響一切

閘多項式的次數決定了擴展求值域的大小。更高的次數意味著更多的求值點、 更多的 FFT 運算和更慢的證明。擴展域大小為 $2^{k+j}$，其中 $j = \lceil \log_2(d) \rceil$， $d$ 是最大閘次數。

8.4 區域、佈局器與平面規劃

區域是 Halo2 電路中模組化的基本單位。區域是行和欄的連續區塊，其中的單元格以 區域內的相對偏移定址，而非絕對行號。

為什麼需要區域？

  • 模組化：每個晶片獨立定義自己的區域。晶片作者不需要知道哪些行號可用。
  • 可組合性：多個晶片可在同一電路中使用。佈局器處理放置。
  • 可重用性：一個晶片可被多次實例化（例如 100 輪雜湊），每次建立一個新區域。

區域分配

layouter.assign_region(
    || "my_region",     // debug name
    |mut region| {
        // Row 0 within this region (absolute row determined by layouter)
        region.assign_advice(|| "x", config.a, 0, || value_x)?;
        // Row 1 within this region
        region.assign_advice(|| "y", config.a, 1, || value_y)?;
        // Enable selector at row 0 of this region
        config.s.enable(&mut region, 0)?;
        Ok(())
    },
)?;
注意：偏移量是相對的

當你寫 region.assign_advice(..., 0, ...) 時，0 是區域內的偏移量。 佈局器可能將此區域從絕對第 42 行開始放置，因此該單元格實際位於第 42 行。你永遠不需要自己指定絕對行號。

SimpleFloorPlanner

預設的 SimpleFloorPlanner 工作方式如下：

  1. 區域按照呼叫 assign_region 的順序進行分配。
  2. 每個區域被放置在下一個有足夠空閒空間的可用行。
  3. 使用不同欄位集合的區域可以在行上重疊（規劃器追蹤每行中哪些欄正在使用）。
  4. 若兩個區域使用相同的欄位，它們不能重疊，將被垂直堆疊。
  欄位：  a       b       c       fixed
         +-------+-------+-------+-------+
  第 0 行 | 區域 1         | 區域 2         |  （區域使用不同欄位，
  第 1 行 | (a, b 欄)      | (c, fixed)    |   所以可以共享行）
  第 2 行 |               |               |
         +-------+-------+-------+-------+
  第 3 行 | 區域 3 (使用 a, b, c)            |  （需要所有欄位，獨佔行）
  第 4 行 |                               |
         +-------+-------+-------+-------+
  第 5 行 |          盲化行                  |
  第 6 行 |        （隨機值）                |
  第 7 行 |                               |
         +-------+-------+-------+-------+

區域間的複製約束

區域是獨立的 — 但通常一個區域的輸出必須等於另一個區域的輸入。 這透過複製約束（排列論證的一部分）來處理： 

// In region 1: compute a * b = c, return c_cell
let c_cell = layouter.assign_region(|| "mul", |mut region| {
    // ... assign a, b, c ...
    Ok(c_cell)
})?;

// In region 2: use c_cell as input
layouter.assign_region(|| "add", |mut region| {
    // Copy c from region 1 into this region's row 0, column a
    let c_copy = c_cell.copy_advice(|| "c_in", &mut region, config.a, 0)?;
    // ... continue with c_copy ...
    Ok(())
})?;

copy_advice 方法同時分配值並建立複製約束，排列論證將強制執行此約束。

8.5 Halo2 中的 IPA 承諾

Halo2 使用 Inner Product Argument (IPA) 作為其多項式承諾方案。 這是消除可信設置需求的關鍵設計選擇。

承諾

給定多項式 $f(X) = \sum_{i=0}^{n-1} a_i X^i$，其係數向量為 $\mathbf{a} = (a_0, a_1, \ldots, a_{n-1})$，以及公開已知的生成元 $\mathbf{G} = (G_0, G_1, \ldots, G_{n-1})$，承諾是一個 Pedersen 向量承諾：

$$C = \langle \mathbf{a}, \mathbf{G} \rangle + r \cdot H = \sum_{i=0}^{n-1} a_i G_i + r H$$

其中 $r$ 是隨機盲化因子，$H$ 是用於隱藏的附加生成元。

開啟（求值證明）

為了證明 $f(z) = v$（其中 $z$ 為挑戰點），IPA 協議使用遞迴二分法。 核心思想：將長度為 $n$ 的內積聲明歸約為長度 $n/2$ 的聲明，再歸約為 $n/4$， 如此進行 $\log_2 n$ 輪。

初始設置

證明者持有 $\mathbf{a}$（係數），驗證者知道 $C$（承諾）、 $z$（求值點）和 $v = f(z)$（聲明的值）。定義 $\mathbf{b} = (1, z, z^2, \ldots, z^{n-1})$ 使得 $\langle \mathbf{a}, \mathbf{b} \rangle = v$。

遞迴二分（重複 $\log_2 n$ 次）

將 $\mathbf{a} = (\mathbf{a}_L \| \mathbf{a}_R)$、 $\mathbf{b} = (\mathbf{b}_L \| \mathbf{b}_R)$、 $\mathbf{G} = (\mathbf{G}_L \| \mathbf{G}_R)$ 分成兩半。 證明者傳送： $$L = \langle \mathbf{a}_L, \mathbf{G}_R \rangle + \langle \mathbf{a}_L, \mathbf{b}_R \rangle \cdot U$$ $$R = \langle \mathbf{a}_R, \mathbf{G}_L \rangle + \langle \mathbf{a}_R, \mathbf{b}_L \rangle \cdot U$$ 驗證者傳送挑戰 $u_j$。雙方計算： $\mathbf{a}' = \mathbf{a}_L + u_j \mathbf{a}_R$， $\mathbf{b}' = \mathbf{b}_L + u_j^{-1} \mathbf{b}_R$， $\mathbf{G}' = \mathbf{G}_L + u_j^{-1} \mathbf{G}_R$。

最終檢查

經過 $\log_2 n$ 輪後，雙方都有純量值 $a'$、$b'$ 和單一生成元 $G'$。 驗證者檢查：$C' = a' G' + a' b' U$，其中 $C'$ 是從所有 $L$ 和 $R$ 值累積的承諾。

定理：IPA 的健全性

在離散對數假設下，IPA 方案具有計算綁定性： 沒有 PPT 對手能以超過可忽略的機率，產生兩個不同的多項式 $f \neq f'$ 且具有相同的承諾 $C$。 證明大小為 $O(\log n)$ 個群元素（具體為 $2 \log_2 n$ 個元素：每輪一個 $L$ 和一個 $R$）。

8.6 證明流程

Halo2 證明者遵循一系列結構化的多項式承諾和 Fiat-Shamir 挑戰步驟。每一步都建立在前一步之上。

步驟一：提交見證多項式

對每個建議欄，將分配的值在域 $H$ 上進行插值以獲得多項式 $a_i(X)$。 在盲化行中加入隨機盲化值。承諾：$A_i = \mathsf{Com}(a_i(X))$。將所有 $A_i$ 送入抄本。

步驟二：生成挑戰

從 Fiat-Shamir 抄本中擠出挑戰：

  • $\theta$ — 用於壓縮查找表欄
  • $\beta, \gamma$ — 用於排列論證和查找論證

步驟三：查找排列

對每個查找論證，計算排列後的輸入和表欄 $A'(X)$ 和 $S'(X)$， 使得 $A'$ 是輸入表達式的排列，$S'$ 是對應的排序表。提交 $A'_i$ 和 $S'_i$。

步驟四：排列乘積

計算排列論證的大乘積多項式 $Z(X)$： $$Z(\omega^0) = 1, \quad Z(\omega^{i+1}) = Z(\omega^i) \cdot \prod_j \frac{a_j(\omega^i) + \beta \cdot \delta^j \cdot \omega^i + \gamma}{a_j(\omega^i) + \beta \cdot \sigma_j(\omega^i) + \gamma}$$ 同理，計算查找乘積多項式。提交所有乘積多項式。

步驟五：消失論證

將所有閘約束、排列約束和查找約束合併為單一多項式 $h(X)$： $$h(X) = \frac{\text{gate\_constraints}(X) + \text{perm\_constraints}(X) + \text{lookup\_constraints}(X)}{Z_H(X)}$$ 由於分子在 $H$ 上消失，$h(X)$ 是一個多項式（無餘數）。 將 $h(X)$ 分割為次數 $n$ 的區塊 $h_0, h_1, \ldots$ 並分別提交。

步驟六：多點開啟

從抄本中擠出隨機求值點 $x$。在 $x$ 處（以及需要下一行查詢時在 $x\omega$ 處） 求值所有已提交的多項式。按求值點分組開啟，並使用多點開啟方案進行批次處理。

步驟七：IPA 證明

為所有批次多項式開啟產生 Inner Product Argument 證明。 這會在證明中新增 $2 \log_2 n$ 個群元素。驗證者可透過單次 IPA 驗證檢查所有開啟。

警告：順序至關重要

Fiat-Shamir 抄本必須以與證明者產生承諾完全相同的順序吸收它們。 任何不匹配 — 即使是在 $A_1$ 之前吸收 $A_2$ — 都會產生不同的挑戰， 導致證明無法驗證。這是自定義協議實作中常見的錯誤來源。

8.7 Halo2 域配置

域參數 $k$ 是最重要的配置選擇：它決定了電路的容量並直接影響證明時間。

選擇 $k$

參數 $k$ 必須滿足：

$$2^k \geq \text{num\_circuit\_rows} + \text{blinding\_factors} + 1$$

其中 blinding_factors 取決於最大旋轉偏移量和階段數。實際上： 

// Determine minimum k for a circuit
let min_rows = circuit_rows + blinding_factors + 1;
let k = (min_rows as f64).log2().ceil() as u32;

// Example: 1000 circuit rows + 5 blinding rows + 1 = 1006
// k = ceil(log2(1006)) = 10  (gives 1024 rows)

可用行數

並非所有 $2^k$ 行都可用於電路邏輯：

$$\text{usable\_rows} = 2^k - \text{blinding\_factors} - 1$$

最後幾行保留用於盲化，另有一行保留作為「不可用」的邊界行。

擴展域

商多項式 $h(X)$ 的次數高於 $n - 1$（其中 $n = 2^k$），因為它是閘多項式的乘積除以消失多項式。 為了高效求值 $h(X)$，Halo2 使用擴展域：

$$|H'| = 2^{k+j}$$

其中 $j$ 由最大閘次數 $d$ 決定。具體而言，在一般情況下 $j = \lceil \log_2(d - 1) \rceil + 1$，確保擴展域足夠大， 能夠無混疊地表示所有中間多項式。

單位根

單位根 $\omega$ 是純量場 $\mathbb{F}_q$ 中 $2^k$ 階乘法子群的生成元。 對於 Pallas 曲線： 

// For Pallas scalar field (Fp of Vesta base field):
// The field has a large power-of-two subgroup.
// For k = 10: omega = g^(2^S / 2^10) where g is the generator
// and S is the two-adicity of the field.
//
// Pasta curves have two-adicity S = 32, meaning
// we can support k up to 32 (4 billion rows).
定義：二進位適性 (Two-Adicity)

若質數 $p$ 滿足 $p - 1 = 2^S \cdot t$（其中 $t$ 為奇數），則稱其具有二進位適性 $S$。 這意味著 $\mathbb{F}_p^*$ 包含一個 $2^S$ 階的乘法子群。 對於 Pasta 曲線，$S = 32$，因此最大域大小為 $2^{32} \approx 4 \times 10^9$ 行。

8.8 Pallas 與 Vesta 曲線

Halo2 設計為與 Pasta 曲線循環配合使用：一對名為 Pallas 和 Vesta 的橢圓曲線，專門為遞迴證明組合而設計。

什麼是曲線循環？

當兩條曲線的底場等於另一條的純量場（反之亦然）時，它們構成循環：

  • Pallas：定義在底場 $\mathbb{F}_p$ 上，純量場為 $\mathbb{F}_q$
  • Vesta：定義在底場 $\mathbb{F}_q$ 上，純量場為 $\mathbb{F}_p$
           Pallas (E_p)                     Vesta (E_q)
        底場: F_p                        底場: F_q
        純量場: F_q                      純量場: F_p
               |                                |
               |   純量場 = 底場                 |
               +------------ 循環 --------------+
               |   底場 = 純量場                 |
               |                                |

   Pallas 點的算術                    Vesta 點的算術
   座標使用 F_p                       座標使用 F_q
   純量使用 F_q                       純量使用 F_p

為什麼這對遞迴很重要

在遞迴證明組合中，Pallas 上證明的驗證電路必須執行橢圓曲線算術 （$\mathbb{F}_q$ 中的純量乘法）。若我們將此驗證器作為 Vesta 上的電路執行， 電路的原生場是 $\mathbb{F}_p$（Vesta 的純量場），而 Pallas 的純量運算在 $\mathbb{F}_q$ 中 — 即 Vesta 的底場，而非其純量場。 如果沒有循環，這將需要昂貴的非原生場算術。有了循環：

  • Pallas 證明在 Vesta 電路中驗證（Vesta 純量場 = $\mathbb{F}_p$ = Pallas 底場）
  • Vesta 證明在 Pallas 電路中驗證（Pallas 純量場 = $\mathbb{F}_q$ = Vesta 底場）
  • 每一步「活躍」曲線交替，但所有算術都是原生場算術
定理：透過累積的遞迴組合

Halo 不在電路內完整驗證 IPA 證明（這需要 $O(n)$ 的 MSM），而是使用 累積方案：電路將 IPA 實例累積到一個執行中的累加器中。 只有在最後（電路外部）才檢查累積的實例。這將電路內的驗證成本從 $O(n)$ 降低到 $O(\log n)$ 群運算。

8.9 Halo2 與 PLONK 比較

Halo2 繼承了 PLONK 的核心架構，但做出了不同的權衡取捨。 以下比較突顯了關鍵差異。

Halo2 (IPA)

  • 設置：透明 — 無需可信設置儀式。參數是確定性的。
  • 自定義閘：完整支援 — 對任何旋轉量的單元格進行任意多項式表達式。
  • 查找：內建 — 約束系統 API 中的一等查找論證。
  • 證明大小：$O(\log n)$ — IPA 為每次開啟增加對數開銷。
  • 驗證：$O(n)$ — 驗證者必須計算 MSM（多純量乘法）。
  • 遞迴：原生 — 在 Pasta 曲線循環上的累積方案。
  • 曲線：Pallas/Vesta（或搭配 KZG 後端使用 BN254）。

PLONK (KZG)

  • 設置：通用可信設置 (SRS) — 一次儀式，可重用於所有大小至 $n$ 的電路。
  • 自定義閘：原始規範中受限 — 固定閘形式 $q_L a + q_R b + q_M ab + q_O c + q_C = 0$。
  • 查找：擴展功能 — 透過 PLONKup 或類似方案添加，不在原始協議中。
  • 證明大小：$O(1)$ — KZG 開啟各為單一群元素。
  • 驗證：$O(1)$ — 常數次配對運算。
  • 遞迴：可行但更困難 — 電路內的配對驗證成本高昂。
  • 曲線：BN254、BLS12-381（需要配對友好曲線）。

使用 IPA 的情境：不需要可信設置（法規/去中心化要求）， 需要遞迴證明組合（Zcash Orchard），或驗證在鏈下進行（桌面/行動錢包）。

使用 KZG 的情境：證明必須在 Ethereum 上驗證 （位址 0x08 的 EVM 配對預編譯成本低），證明大小重要（rollup calldata 成本）， 或不需要電路內遞迴。

8.10 Halo2 中的查找

Halo2 的查找論證是約束系統的一等功能。它證明一組表達式（「輸入」）全部包含在一個表 （「表欄」）中。

定義查找

// In configure():
let input = meta.advice_column();
let table = meta.lookup_table_column();  // a special fixed column

meta.lookup("range_check", |meta| {
    let input_expr = meta.query_advice(input, Rotation::cur());
    let table_expr = meta.query_lookup_table(table);
    vec![(input_expr, table_expr)]
});

這宣告了：對於每個活躍行，input 中的值必須存在於 table 的某處。 可使用多個輸入/表的表達式對來進行多欄查找。

查找的內部運作

Halo2 中的查找論證遵循 Plookup 協議的一般結構：

  1. 證明者將輸入值和表值排序為合併的排序順序，產生排列後的欄 $A'$ 和 $S'$。
  2. 大乘積論證使用隨機挑戰 $\theta$、$\beta$、$\gamma$ 檢查輸入值的多重集合是表值多重集合的子集。
  3. 額外的約束確保 $A'$ 和 $S'$ 正確排序（$A'$ 中相鄰的不同元素必須出現在 $S'$ 中表值組的邊界處）。
注意：查找的成本

每個查找新增 2 個建議欄（用於排列後的輸入和表）、1 個乘積多項式和數個約束。 這大約相當於證明中額外的 5 個承諾。對於具有許多查找的電路 （例如 zkEVM 的字節碼、記憶體、儲存查找），查找成本可能主導整體證明時間。

8.11 晶片組合模式

實際的 Halo2 電路由可組合的晶片構建。晶片是一個自包含的模組， 配置自己的欄和閘，並提供將值分配到區域中的方法。

晶片結構

// 1. Configuration struct (holds column references)
#[derive(Clone, Debug)]
struct AddConfig {
    a: Column<Advice>,
    b: Column<Advice>,
    c: Column<Advice>,
    s_add: Selector,
}

// 2. Chip struct (holds config + any state)
struct AddChip<F: Field> {
    config: AddConfig,
    _marker: PhantomData<F>,
}

impl<F: Field> AddChip<F> {
    // 3. Configure: define columns and gates
    fn configure(
        meta: &mut ConstraintSystem<F>,
        a: Column<Advice>,
        b: Column<Advice>,
        c: Column<Advice>,
    ) -> AddConfig {
        let s_add = meta.selector();
        meta.create_gate("add", |meta| {
            let a = meta.query_advice(a, Rotation::cur());
            let b = meta.query_advice(b, Rotation::cur());
            let c = meta.query_advice(c, Rotation::cur());
            let s = meta.query_selector(s_add);
            vec![s * (a + b - c)]
        });
        AddConfig { a, b, c, s_add }
    }

    // 4. Assign: fill in values for one operation
    fn add(
        &self,
        layouter: &mut impl Layouter<F>,
        a: Value<F>,
        b: Value<F>,
    ) -> Result<AssignedCell<F, F>, Error> {
        layouter.assign_region(
            || "add",
            |mut region| {
                self.config.s_add.enable(&mut region, 0)?;
                region.assign_advice(|| "a", self.config.a, 0, || a)?;
                region.assign_advice(|| "b", self.config.b, 0, || b)?;
                let c_val = a.and_then(|a| b.map(|b| a + b));
                region.assign_advice(|| "c", self.config.c, 0, || c_val)
            },
        )
    }
}

組合晶片

高層電路透過共享欄位來組合晶片。例如，一個「算術晶片」可能結合一個 AddChip 和一個 MulChip，它們共享相同的建議欄但擁有不同的選擇器： 

fn configure(meta: &mut ConstraintSystem<F>) -> Self::Config {
    let a = meta.advice_column();
    let b = meta.advice_column();
    let c = meta.advice_column();

    // Both chips share columns a, b, c
    let add_config = AddChip::configure(meta, a, b, c);
    let mul_config = MulChip::configure(meta, a, b, c);

    ArithConfig { add_config, mul_config }
}
設計模式：欄位共享

永遠不會在同一行啟用選擇器的晶片可以共享欄位。 這減少了欄的總數，從而減少多項式承諾的數量。 SimpleFloorPlanner 確保不同晶片的區域被放置在不重疊的行中， 使選擇器衝突不可能發生。

8.12 除錯與 MockProver

Halo2 提供了 MockProver，這是一個測試工具，可以在不生成實際證明的情況下 檢查所有約束。這對電路開發至關重要。 

use halo2_proofs::dev::MockProver;

let k = 4;  // small domain for testing
let circuit = MulCircuit { a: Value::known(Fp::from(3)), b: Value::known(Fp::from(7)) };
let public_inputs = vec![Fp::from(21)];

let prover = MockProver::run(k, &circuit, vec![public_inputs]).unwrap();

// Check all constraints
prover.assert_satisfied();

// Or get detailed errors
match prover.verify() {
    Ok(()) => println!("All constraints satisfied!"),
    Err(errors) => {
        for error in errors {
            println!("Constraint failure: {:?}", error);
            // Prints: gate name, row number, which expression failed
        }
    }
}
警告：MockProver 不是安全性檢查

MockProver 驗證的是誠實證明者的賦值是否滿足所有約束。 它不檢查健全性 — 可能存在其他滿足約束但代表錯誤計算的有效賦值。 務必另外推理約束不足的電路。

電路佈局視覺化

Halo2 還透過 CircuitLayout 提供佈局視覺化工具： 

use halo2_proofs::dev::CircuitLayout;
use plotters::prelude::*;

let root = BitMapBackend::new("layout.png", (1024, 768))
    .into_drawing_area();
root.fill(&WHITE).unwrap();

CircuitLayout::default()
    .render(k, &circuit, &root)
    .unwrap();

這會生成一張圖片，顯示哪些單元格被哪些區域使用、選擇器在哪裡啟用， 以及盲化行從何處開始。對於除錯佈局效率低落的問題極為有用。

8.14 關鍵公式摘要

參考：Halo2 公式

域大小： $n = 2^k$

可用行數： $n_{\text{usable}} = 2^k - \text{blinding\_factors} - 1$

盲化因子： $\text{bf} = \max(\text{rotations across all queries}) + 1$

擴展域： $|H'| = 2^{k + j}$，其中 $j$ 取決於閘次數

IPA 承諾： $C = \langle \mathbf{a}, \mathbf{G} \rangle + r H$

IPA 證明大小： $2 \log_2 n$ 個群元素

閘約束（一般形式）： $\sum_i s_i \cdot \prod_j \text{cell}_{ij}^{e_{ij}} = 0$

排列乘積： $Z(\omega^{i+1}) = Z(\omega^i) \cdot \prod_j \frac{a_j(\omega^i) + \beta \delta^j \omega^i + \gamma}{a_j(\omega^i) + \beta \sigma_j(\omega^i) + \gamma}$

消失論證： $h(X) = \frac{\text{constraints}(X)}{Z_H(X)}$，其中 $Z_H(X) = X^n - 1$

對本研究的意義

設計選擇：為何選擇 Halo2

Halo2 是本論文的實作框架。以下從五個面向說明為何它是多項式方法的最佳選擇：

需求Halo2 的支援替代方案的不足
無可信設置 IPA 承諾 — 只需公開隨機基點 Groth16 需要電路特定 MPC 儀式；PLONK+KZG 需要 Powers-of-Tau
自定義閘 meta.create_gate() API Groth16/R1CS 僅支援 $a \cdot b = c$
查找表 meta.lookup_any() 原生支援 Groth16 無原生查找；需用約束模擬
高效能實作 Rust，原生效能 Circom/snarkjs (JS) 較慢
遞迴潛力 Pallas/Vesta 曲線循環 BN254 無曲線循環；STARK 不用曲線但證明更大
除錯工具 MockProver + 電路佈局視覺化 多數框架缺乏互動式除錯
Halo2 的具體 API 與論文電路的對應

以下是論文多項式方法電路使用的核心 Halo2 API：

  • ConstraintSystem::advice_column() — 建立 Horner 中間值列（$R$, $H_i$, $w$）
  • ConstraintSystem::fixed_column() — 建立預計算係數列（$D_1, \ldots, D_d$）
  • meta.create_gate("horner", |meta| { ... }) — 定義 Horner 乘加約束
  • meta.lookup_any("byte_check", |meta| { ... }) — ByteTable range check
  • region.assign_advice() — Prover 在區域中填入見證值
  • MockProver::run(k, &circuit, instances) — 除錯模式驗證
選擇 Halo2 的代價

Halo2 的 IPA 承諾導致驗證時間為 $O(n)$（而非 KZG 的 $O(1)$），證明大小為 $O(\log n)$（而非 Groth16 的 $O(1)$）。 這意味著：

  • 鏈上驗證成本較高：在以太坊上驗證 Halo2 證明比驗證 Groth16 證明更貴。
  • 證明較大：對 $n = 2^{17}$ 的電路，IPA 證明約 1-2 KB，Groth16 僅 128 bytes。
  • 解決方案：若需要部署到以太坊，可使用 PSE 分支的 KZG 後端，或透過遞迴壓縮後再送上鏈。
