Layer 1 — 證明系統類別：zk-SNARK

zk-SNARK 零知識簡潔非互動式知識論證
Succinct（簡潔）、Non-interactive（非互動）、ARgument of Knowledge（知識論證）。 Proof 大小為常數級或對數級，驗證時間遠小於計算本身。
選項Proof 大小驗證速度Trusted Setup安全假設
zk-SNARK ← 我們的選擇 $O(1)$ 常數級 極快（pairing） 視方案而定 DLog / AGM
zk-STARK $O(\text{polylog}(n))$ 較大 較慢（hash-based） 不需要（透明） 僅 CRHF（碰撞抗性）
選擇理由

Proof size 小、驗證速度快，適合鏈上驗證場景。STARK 的透明性優勢在本研究中不是關鍵需求。

Layer 2 — 算術化方案：PLONK

PLONK 一種算術化方案
把計算寫成 trace 表格，每行一個 gate 約束。用置換論證連接等值關係， 在 evaluation domain 上壓成多項式，透過隨機挑戰歸約成單點驗證。 支援 custom gate（自訂門），且 setup 是通用、可更新的。
選項Gate 形式靈活性SetupLookup 支援
PLONK ← 我們的選擇 任意多項式（custom gate） 高 通用、可更新 可擴展（plookup）
R1CS（Groth16） $L \cdot R = O$ 固定格式 低 每個電路重做 無原生支援
AIR（STARK 用） 轉移約束 中 透明 可擴展

PLONK 的優勢

  • Custom gate：Horner 的乘加操作可以用一個 degree-2 gate 完成，R1CS 需要拆成兩條約束
  • 通用 setup：SRS 不包含電路資訊，換函數不需要重新做 ceremony
  • Lookup 可擴展：ByteTable 的 range check 依賴 lookup argument

R1CS 的劣勢

  • 每條約束只能做一次乘法 $L \cdot R = O$
  • Horner 一步要拆成 2 條約束，degree-5 多項式需 10 條（PLONK 只要 5 個 gate）
  • Groth16 的 setup 綁定特定電路，換函數要重新做 trusted setup

Layer 3 — 軟體框架：Halo2 PSE fork

Halo2 ZK 證明框架
Zcash 團隊開發的 PLONK 實現框架。PSE（Privacy and Scaling Explorations）fork 將承諾方案從 IPA 改為 KZG， 獲得常數大小的 proof。用 Rust 撰寫，提供 Chip / Gadget / Layouter 等 API。
選項語言算術化承諾方案特色
Halo2 PSE fork ← 我們的選擇 Rust PLONK KZG Custom gate + Lookup + 跨行引用
Halo2 原版（Zcash） Rust PLONK IPA 無 trusted setup，但 proof 較大
circom + snarkjs JS / circom DSL R1CS Groth16 最小 proof，但 R1CS 限制多
arkworks Rust 多種 多種 通用庫，API 較低階
選擇理由
  1. Custom gate：設計高效的 Horner gate 和 Truncation gate
  2. Lookup argument：ByteTable 的 range check 是電路的核心組件
  3. 跨行引用（Rotation）：Horner 鏈的每步可以直接引用上一行結果，不需要額外 wiring
  4. KZG proof：常數大小，便於效能比較
  5. 生態相容：比較對象 ZKFixedPointChip 也使用 Halo2

Layer 4 — 多項式承諾方案：KZG

KZG Commitment Kate-Zaverucha-Goldberg 多項式承諾
把多項式 $f(x)$ 壓成橢圓曲線上的一個點 $C = [f(\tau)]_1$（$\tau$ 來自 trusted setup）。 之後可在任意點 $z$ 打開，證明 $f(z) = y$。Proof 為一個群元素（48 bytes on BN254）， 驗證只需一次 pairing。
選項Proof 大小驗證速度Trusted Setup假設
KZG ← 我們的選擇 $O(1)$，48 bytes 一次 pairing（極快） 需要（MPC ceremony） AGM + DLog
IPA（Inner Product Argument） $O(\log n)$ 多次標量乘法（慢） 不需要 DLog
FRI（STARK 用） $O(\log^2 n)$ Hash 運算 不需要 CRHF
Trade-off

KZG 的代價是需要 trusted setup（SRS）。SRS 由 MPC ceremony 產生——只要有一個參與者是誠實的就安全。 Zcash 和 Ethereum 都已完成大規模 ceremony，這是業界普遍接受的假設。

Layer 5 — 橢圓曲線：BN254

BN254 Barreto-Naehrig 曲線
一條 pairing-friendly 橢圓曲線。Scalar field 為 254-bit 質數 $p \approx 2^{253.97}$。電路中所有運算都是 $\bmod\; p$。
選項Field 大小Pairing生態支援
BN254 ← 我們的選擇 254 bit 支援 Ethereum 原生支援
BLS12-381 255 bit 支援 Zcash、Ethereum 2.0
為什麼 254 bit 很重要

254-bit field 為 deferred truncation 提供了 overflow headroom。 Horner 鏈中間值最大需要 $(d+1) \times pp$ bits：

  • $pp = 16, d = 5$：$(5+1) \times 16 = 96$ bits，headroom $= 254 - 96 = 158$ bits
  • $pp = 24, d = 5$：$(5+1) \times 24 = 144$ bits，headroom $= 254 - 144 = 110$ bits
  • $pp = 32, d = 7$：$(7+1) \times 32 = 256$ bits > 254 bits → overflow！不可行

硬約束：$(d+1) \times pp < 254$。這直接決定了 $(pp, d)$ 設計空間的邊界。

Layer 6 — 數值表示法：定點（Fixed-Point）

Fixed-Point Representation 定點表示法
選一個精度參數 $pp$，令 $S = 2^{pp}$。實數 $r$ 表示為整數 $R = \text{round}(r \cdot S)$。 所有運算在整數上進行，最終結果除以 $S$ 還原。
選項電路成本精度實現複雜度
定點 ← 我們的選擇 低（整數加乘 + 一次截斷） 由 $pp$ 決定 簡單
浮點（IEEE 754 風格） 極高（exponent 需 bit-level 操作） 高（動態範圍） 極複雜
純整數查表法（LUT） 表格巨大（每函數 65K+ 條目） 由表格大小決定 簡單但不可擴展
選擇理由

ZK 電路的原生運算是 $\mathbb{F}_p$ 上的整數加法和乘法。定點表示法只需要在乘法後做一次截斷（除以 $S$）， 是將實數帶入電路的最自然方式。浮點的 exponent 部分在電路中需要大量 bit 操作，成本不可接受。

Layer 7 — 函數逼近方法：Constrained Minimax

Constrained Minimax Polynomial 受限 Minimax 多項式
在所有同 degree 且滿足 $c_0 = 0$ 約束的多項式中，找到使 $L_\infty$ 誤差最小的那一個。 即 $\min_{p \in \mathcal{P}_d,\; p(0)=0} \max_{x \in [a,b]} |f(x) - p(x)|$。
選項同 degree 精度特性$c_0 = 0$
Constrained Minimax ← 我們的選擇 最優（$L_\infty$ 最小） 等振盪定理保證最優 支援
Taylor 展開 差（只在展開點附近好） 只優化一個點的誤差 天然 $c_0 = 0$（可做到）
Chebyshev 逼近 接近最佳但非最優 節點分佈近似等振盪 不保證
查表法（LUT） 精確到表格精度 每函數一張大表 不適用

Minimax vs Taylor 的精度差距

以 $pp = 16$ 為例，Minimax 在同 degree 下 ULP 比 Taylor 好 幾十到上千倍。

原因：Taylor 把所有逼近精度集中在 $x = 0$ 附近，Minimax 把誤差均勻分配到整個區間（等振盪性質）。

$c_0 = 0$ 約束的意義

強制多項式的常數項為 0，意味著 $p(0) = 0$。 好處：省掉一個加法常數項，Horner 求值的最後一步直接是 $R \cdot H_1$，截斷更乾淨。

Layer 8 — 多項式求值法：Horner

Horner's Method Horner 求值法
把 $D_d x^d + \cdots + D_1 x$ 改寫成嵌套形式 $((\cdots((D_d \cdot x + D_{d-1}) \cdot x + D_{d-2}) \cdot x + \cdots) \cdot x + D_1) \cdot x$。 只需 $d$ 次乘法和 $d$ 次加法。
選項乘法次數Gate degree可否延遲截斷
Horner ← 我們的選擇 $d$ 次（最少） 每步 degree 2 可以（序列結構，中間值可追蹤）
直接展開（$D_k \cdot x^k$ 逐項算） $O(d^2)$（需算各次冪） 高（$x^k$ 需多次乘） 困難（多條平行路徑）
Estrin 法（樹狀並行） $d$ 次 部分步驟 degree > 2 困難（中間值分支）
為什麼 Horner 完美契合 Halo2

每一步都是 $H_k = D_k + R \cdot H_{k+1}$——一個乘法 + 一個加法，剛好是 degree-2 的 custom gate。 $d$ 步 Horner = $d$ 行 trace，搭配跨行引用（Rotation），每行直接拿上一行的 $H_{k+1}$。 不需要額外 column，不需要 permutation wiring。

Layer 9 — 截斷策略：Deferred Truncation

Deferred Truncation 延遲截斷
整條 Horner 鏈中間不做截斷（不除以 $S$），讓中間值在 $\mathbb{F}_p$ 中自由膨脹。 最後只做一次 $\lfloor w_{\text{pre}} / S^d \rfloor$。前提：中間值不超過 $p$（overflow headroom 足夠）。
選項截斷次數電路成本精度前提
Deferred ← 我們的選擇 1 次 1 次 byte decomp + lookup ULP $\leq 1$（13/13 函數在 $d^*$） $(d+1) \times pp < 254$
Per-step truncation $d$ 次 $d$ 次 byte decomp + lookup 每步累積捨入誤差，全部 13/13 函數失敗 無

Deferred Truncation 的優勢

  • 電路成本降至 $1/d$：截斷是最昂貴的操作（byte decomposition + lookup），從 $d$ 次減到 1 次
  • 精度更好：中間不截斷 = 中間不捨入 = 不累積誤差
  • 實驗結果：在 $d^*$ 處，deferred 讓 13/13 函數達 ULP $\leq$ 1；per-step 讓 13/13 函數全部失敗（ULP ratio 2x–7x）

代價與限制

  • Overflow 風險：中間值膨脹到 $(d+1) \times pp$ bits，必須確保不超過 $p$（254 bits）
  • 限制 $(pp, d)$ 組合：例如 $pp = 32, d = 7$ 需要 256 bits，超過 BN254 的 254 bits
  • 依賴 BN254 的 field size：換成更小的 field 就不可行

Layer 10 — 輸入範圍驗證：ExplicitRC

ExplicitRC 顯式範圍檢查
計算 $\text{gap} = R_{\max} - R$，將 gap 拆成兩個 bytes 並查 ByteTable 驗證。 若 $R > R_{\max}$，差值在 $\mathbb{F}_p$ 中 wrap around 成巨大數，無法拆成 2 bytes，Prover 被抓到。
選項Soundness成本狀態
ExplicitRC ← 我們的選擇 完整保證 1 gate + 2 lookups 使用中
不驗證（信任 Prover） 無保證 0 不可接受
ORV（Output Range Verification） 有漏洞（7/13 函數 vulnerable） 低 已放棄
ORV 被放棄的原因

ORV 嘗試透過檢查輸出值的範圍來間接驗證輸入範圍。但 7/13 函數存在漏洞—— 惡意 Prover 可以用超出範圍的輸入，產生看起來在合理範圍內的輸出。 最嚴重的情況下誤差達 1043 ULP。ExplicitRC 的 gap decomposition 直接驗證 $R \leq R_{\max}$， 從根本上杜絕此問題。

層間依賴關係

每一層的選擇不是獨立的，它們互相配合：

依賴關係說明
Deferred Truncation → BN254 延遲截斷需要 254-bit field 的 overflow headroom
Horner → PLONK custom gate 每步乘加 = degree-2 gate，完美契合 PLONK 的 custom gate
Horner → Halo2 Rotation 跨行引用讓 Horner 鏈不需額外 wiring
ExplicitRC → ByteTable → Lookup Gap decomposition 依賴 lookup argument 做 byte range check
Constrained Minimax ($c_0 = 0$) → Horner $c_0 = 0$ 讓 Horner 最後一步是 $R \cdot H_1$，截斷更乾淨
定點 → 截斷 乘法後 scale 變 $S^2$，必須截斷回 $S$，截斷是電路主要成本
KZG → BN254 pairing KZG 的 commit 和 verify 需要 pairing-friendly 曲線
$(pp, d)$ 設計空間 → BN254 + Deferred Truncation 可行的 $(pp, d)$ 組合由 $(d+1) \times pp < 254$ 決定
設計哲學

這套技術堆疊的核心邏輯是：用多項式逼近取代查表，用代數約束取代暴力計算。 每一層選擇都在最小化電路成本的同時，維持精度（ULP $\leq$ 1）和安全性（soundness）。 最終結果：10 行、6 個 custom gates、12 個 lookups、256 條目的共享 ByteTable， 就能以零知識方式計算任意超越函數。
